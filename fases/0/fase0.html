Fase 0

Fundamentos de backend, rede e HTTP

Nível: iniciante Sem banco de dados Base para todo o curso Explicação detalhada

Antes de aprender Express, precisamos entender o que o Express esconde, organiza ou simplifica. Esta fase começa pelo que pode parecer óbvio de propósito. Palavras como “cliente”, “servidor”, “requisição”, “JSON” e “porta” aparecem o tempo inteiro no desenvolvimento web; quando seu significado fica impreciso, métodos como app.get(), response.json() e express.json() parecem funcionar por magia.

Objetivo da fase: ao final, você deverá conseguir narrar o caminho completo de uma ação no React até uma resposta da API, identificar cada parte de uma mensagem HTTP e explicar onde o Node.js e o Express entram nesse processo.

0.1 — O modelo mental geral

Uma aplicação web costuma envolver, no mínimo, dois programas executando em lugares diferentes:

usuário
  ↓ interage
frontend no navegador
  ↓ envia uma requisição HTTP
backend no servidor
  ↓ executa regras e acessa dados
backend envia uma resposta HTTP
  ↓
frontend interpreta a resposta
  ↓
tela é atualizada

Mesmo quando frontend e backend estão no mesmo computador durante o desenvolvimento, continuam sendo processos e responsabilidades diferentes. O endereço localhost apenas indica que o destino está na própria máquina.

1 — Frontend, backend, cliente e servidor

1.1 — Frontend

Frontend é a parte da aplicação com a qual o usuário normalmente interage. Em um projeto React, inclui componentes, páginas, formulários, estado visual, rotas do navegador e código que solicita dados a serviços externos. Ele costuma executar no navegador do usuário.

O frontend não deve ser considerado uma fonte confiável para proteger regras importantes. O usuário controla o navegador, pode alterar JavaScript, repetir requisições, editar o corpo enviado ou chamar a API sem abrir sua interface.

1.2 — Backend

Backend é a parte que recebe solicitações, valida dados, aplica regras de negócio, autentica usuários, conversa com bancos e serviços externos e devolve respostas. Neste curso, construiremos o backend com Node.js, Express 5 e TypeScript.

1.3 — Cliente

Cliente é o participante que inicia uma comunicação pedindo algo. Um navegador pode ser cliente, mas também um aplicativo mobile, outro backend, um programa de terminal, Postman, Insomnia ou um teste automatizado. “Cliente” é um papel na comunicação, não o nome de uma tecnologia específica.

1.4 — Servidor

Servidor pode significar duas coisas relacionadas: a máquina que hospeda um serviço ou o programa que fica disponível para receber requisições. Quando executamos app.listen(3000), iniciamos um programa servidor escutando conexões na porta 3000.

TermoPergunta que ele respondeExemplo no curso
FrontendOnde está a interface?Aplicação React
BackendOnde ficam regras, dados e integrações?API Express
ClienteQuem iniciou esta requisição?fetch() no navegador
ServidorQuem recebeu e respondeu?Processo Node.js

2 — O que é uma API

API significa Application Programming Interface, ou interface de programação de aplicações. É um contrato que permite que um software use capacidades oferecidas por outro. Uma API define quais operações estão disponíveis, quais dados devem ser enviados, quais respostas podem voltar e como erros são representados.

Nem toda API é uma API web. O navegador oferece APIs como localStorage e fetch; uma biblioteca também expõe uma API por meio de funções e classes. Neste curso, quando dissermos “API”, normalmente estaremos falando de uma API HTTP acessada pela rede.

2.1 — Endpoint

Endpoint é um ponto de acesso da API, geralmente identificado pela combinação de método HTTP e caminho. Portanto, GET /tasks e POST /tasks são endpoints diferentes, embora usem o mesmo path.

2.2 — Contrato

POST /tasks
Content-Type: application/json

{
  "title": "Estudar HTTP"
}
HTTP/1.1 201 Created
Content-Type: application/json

{
  "id": "task-123",
  "title": "Estudar HTTP",
  "completed": false
}

O contrato inclui método, URL, headers, formato do body, status e formato da resposta. TypeScript ajuda nosso código interno, mas não impede que um cliente real envie dados diferentes. Por isso, posteriormente faremos validação em tempo de execução.

3 — O que acontece quando o React executa fetch()

async function loadTasks(): Promise<void> {
  const response = await fetch("http://localhost:3000/tasks");

  if (!response.ok) {
    throw new Error(`A API respondeu com ${response.status}`);
  }

  const tasks = await response.json();
  console.log(tasks);
}

3.1 — Passo a passo

  1. O JavaScript chama fetch() e fornece uma URL e, opcionalmente, configurações.
  2. O navegador interpreta a URL e verifica protocolo, host, porta, path e políticas de segurança.
  3. Quando necessário, o nome de domínio é resolvido para um endereço de rede.
  4. O navegador estabelece ou reutiliza uma conexão com o destino.
  5. Ele monta e envia uma requisição HTTP, incluindo método, headers e possivelmente body.
  6. O processo Node.js que escuta a porta recebe os dados.
  7. Express compara método e path com as rotas registradas.
  8. Middlewares e handler executam validações, regras e operações necessárias.
  9. O backend constrói uma resposta com status, headers e, quando apropriado, body.
  10. O navegador recebe a resposta e resolve a Promise retornada por fetch().
  11. response.json() lê o body e tenta interpretá-lo como JSON.
  12. O React pode armazenar os dados no estado e renderizar novamente a interface.
Detalhe importante: fetch() normalmente não rejeita a Promise apenas porque a API respondeu 404 ou 500. Esses casos ainda são respostas HTTP válidas. Por isso, conferimos response.ok ou response.status. A Promise costuma rejeitar em falhas como erro de rede, bloqueio da requisição ou cancelamento.

3.2 — Exemplo de envio de JSON

type CreateTaskInput = {
  title: string;
};

async function createTask(input: CreateTaskInput): Promise<void> {
  const response = await fetch("http://localhost:3000/tasks", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
    },
    body: JSON.stringify(input),
  });

  if (!response.ok) {
    throw new Error(`Falha ao criar tarefa: ${response.status}`);
  }
}

4 — HTTP e HTTPS

4.1 — HTTP

HTTP é um protocolo da camada de aplicação usado para trocar mensagens entre clientes e servidores. Protocolo é um conjunto de regras compartilhadas: define como expressar uma requisição, uma resposta, métodos, headers, status e outras semânticas. HTTP não é Express; Express é uma biblioteca que nos ajuda a lidar com HTTP dentro do Node.js.

4.2 — HTTPS

HTTPS é HTTP protegido por TLS. A proteção busca oferecer confidencialidade, integridade e autenticação do servidor. Em termos práticos, terceiros no caminho não deveriam conseguir ler ou alterar facilmente o conteúdo trafegado, e o certificado ajuda o cliente a verificar com quem está falando.

HTTPS protege os dados em trânsito, mas não corrige uma aplicação insegura. Uma API em HTTPS ainda pode ter autorização incorreta, senhas fracas, SQL injection ou vazamento de informações.

AspectoHTTPHTTPS
Esquema comumhttp://https://
Porta padrão80443
Criptografia em trânsitoNão por padrãoSim, via TLS
Uso em produçãoEvitar para dados reaisObrigatório na prática

5 — Anatomia de uma URL

https://api.exemplo.com:8443/tasks/42?include=owner&page=2#details
└─┬─┘   └──────┬──────┘ └┬─┘ └───┬───┘ └─────────┬─────────┘ └──┬──┘
protocolo      domínio     porta      path              query       fragmento

5.1 — Protocolo ou esquema

https indica como o recurso deve ser acessado.

5.2 — Domínio, hostname e host

O domínio é o nome legível, como api.exemplo.com. O hostname identifica o destino na rede. Em muitos contextos práticos os termos são usados de forma próxima. O host pode incluir a porta, por exemplo api.exemplo.com:8443.

5.3 — Porta

A porta identifica qual serviço deve receber a conexão dentro de uma máquina. Uma mesma máquina pode executar vários serviços em portas diferentes. Durante o curso, é comum termos React em uma porta e Express em outra.

5.4 — Path

O path indica o caminho do recurso ou operação dentro do servidor, como /tasks/42. No Express, rotas são comparadas principalmente usando método e path.

5.5 — Query string

A query começa após ? e representa pares de chave e valor. É muito usada para filtros, busca, ordenação, paginação e opções de apresentação. Ela faz parte da URL e não deve carregar segredos. 

GET /tasks?completed=false&page=2&limit=20

5.6 — Fragmento

O fragmento começa com # e normalmente é interpretado pelo cliente. Em requisições HTTP comuns, ele não é enviado ao servidor. É incluído aqui para você não confundi-lo com query string.

6 — Requisição e resposta

6.1 — Requisição

Requisição é a mensagem enviada pelo cliente. Conceitualmente contém método, destino, headers e talvez um body. Em HTTP/1.1 textual, uma mensagem poderia se parecer com isto: 

POST /tasks HTTP/1.1
Host: localhost:3000
Content-Type: application/json
Accept: application/json

{"title":"Estudar Express"}

6.2 — Resposta

Resposta é a mensagem enviada pelo servidor após processar a requisição. Ela contém status, headers e talvez um body. 

HTTP/1.1 201 Created
Content-Type: application/json
Content-Length: 76

{"id":"task-123","title":"Estudar Express","completed":false}

6.3 — Uma requisição deve receber uma resposta

O cliente permanece esperando até receber uma resposta, a conexão falhar ou algum timeout ocorrer. Um handler que não envia resposta e não continua o pipeline deixa a requisição pendurada.

7 — Headers e body

7.1 — Headers

Headers são metadados da mensagem. Eles descrevem informações como tipo do conteúdo, formatos aceitos, autenticação, cache, origem, cookies e tamanho. Headers não são “o conteúdo principal”, mas influenciam como ele deve ser interpretado e processado.

HeaderFinalidade comum
Content-TypeInforma o formato do body enviado.
AcceptInforma quais formatos de resposta o cliente aceita.
AuthorizationTransporta credenciais de autenticação conforme um esquema.
Cache-ControlDefine instruções relacionadas a cache.
OriginInforma a origem de certas requisições do navegador.
Content-LengthIndica o tamanho do conteúdo quando aplicável.

7.2 — Body

Body é o conteúdo principal da mensagem. Pode conter JSON, texto, HTML, dados de formulário, arquivo ou bytes em outro formato. Nem toda requisição ou resposta possui body. Um status 204 No Content, por exemplo, não deve enviar conteúdo de resposta.

7.3 — Content-Type não transforma o conteúdo

Informar Content-Type: application/json apenas declara que o body deveria ser JSON. Isso não corrige automaticamente um JSON inválido. O receptor ainda precisa fazer parsing e lidar com falhas.

8 — JSON e serialização

8.1 — O que é JSON

JSON é um formato textual de troca de dados. Embora tenha sintaxe inspirada em JavaScript, JSON é texto, não um objeto JavaScript. Ele pode ser produzido e consumido por várias linguagens. 

{
  "id": 42,
  "title": "Aprender JSON",
  "completed": false,
  "tags": ["http", "backend"],
  "description": null
}

8.2 — Serialização

Serializar é converter uma estrutura em memória para um formato que possa ser armazenado ou transmitido. JSON.stringify() transforma um valor JavaScript compatível em texto JSON. 

const task = {
  id: 42,
  title: "Aprender JSON",
};

const serialized = JSON.stringify(task);
// '{"id":42,"title":"Aprender JSON"}'

8.3 — Desserialização ou parsing

const text = '{"id":42,"title":"Aprender JSON"}';
const parsed: unknown = JSON.parse(text);

Fazer parsing prova apenas que o texto possui sintaxe JSON válida. Não prova que os campos atendem ao contrato da aplicação. O valor ainda deve ser validado antes de ser confiado.

8.4 — Limitações importantes

  • JSON não possui undefined.
  • JSON não representa funções.
  • Datas normalmente viram strings e precisam de interpretação posterior.
  • BigInt não é serializado por JSON.stringify() sem tratamento.
  • Referências circulares causam erro na serialização padrão.
  • Propriedades com valor undefined podem desaparecer de objetos serializados.

9 — Métodos HTTP

O método comunica a intenção da requisição. Ele não é apenas um nome escolhido livremente; cada método possui semântica. Uma API previsível usa métodos de forma coerente.

MétodoIntenção comumExemplo
GETObter uma representação sem solicitar alteração.GET /tasks
POSTSubmeter dados, frequentemente criando algo.POST /tasks
PUTSubstituir uma representação completa ou definir o recurso.PUT /tasks/42
PATCHAplicar alteração parcial.PATCH /tasks/42
DELETESolicitar remoção.DELETE /tasks/42
HEADObter metadados equivalentes a GET, sem body da resposta.HEAD /files/manual.pdf
OPTIONSDescobrir opções de comunicação; também aparece em preflight CORS.OPTIONS /tasks

9.1 — Seguro e idempotente

Um método seguro é definido para leitura, sem solicitar alteração do estado do servidor. Idempotência significa que repetir a mesma requisição pretendida produz o mesmo efeito final que executá-la uma vez. GET é seguro e idempotente; DELETE é idempotente em sua intenção, ainda que respostas sucessivas possam ter status diferentes; POST geralmente não é idempotente.

10 — Status HTTP

O status resume o resultado da tentativa de processar a requisição. O body pode fornecer detalhes, mas o status permite que clientes, proxies e ferramentas entendam a categoria do resultado sem depender de uma mensagem escrita para humanos.

FaixaCategoriaIdeia geral
100–199InformacionalComunicação intermediária.
200–299SucessoA requisição foi processada com sucesso conforme o status.
300–399RedirecionamentoOutra ação ou localização pode ser necessária.
400–499Erro do clienteHá um problema relacionado à requisição ou permissão.
500–599Erro do servidorO servidor falhou ao cumprir uma requisição válida ou encontrou erro.

11 — Aplicações stateless

HTTP é um protocolo stateless: cada requisição deve ser compreensível por si, e o protocolo não exige que o servidor mantenha memória automática das requisições anteriores. Isso não significa que a aplicação não possua banco, sessão, cache ou estado; significa que o estado precisa ser representado e recuperado de forma explícita. 

Requisição 1: POST /login
Resposta: token ou cookie

Requisição 2: GET /profile
Cliente envia novamente o token ou cookie
Servidor recupera a identidade e processa a requisição

Essa característica facilita escalar horizontalmente: requisições sucessivas podem chegar a instâncias diferentes, desde que todas consigam acessar o estado necessário. Guardar dados críticos apenas em uma variável local do processo cria limitações e perda de dados quando o processo reinicia.

12 — O processo do Node.js

Quando executamos node dist/server.js ou tsx watch src/server.ts, o sistema operacional cria um processo. Esse processo possui memória, identificador, argumentos, variáveis de ambiente, recursos abertos e um ciclo de vida. 

console.log({
  pid: process.pid,
  platform: process.platform,
  nodeVersion: process.version,
  workingDirectory: process.cwd(),
});

12.1 — Por que o servidor continua vivo

Um script simples termina quando não há mais trabalho pendente. Um servidor continua executando porque há recursos ativos, como o socket que está escutando uma porta. O processo permanece disponível esperando novas conexões.

12.2 — Memória do processo não é banco de dados

const tasks: string[] = [];

Esse array existe apenas na memória daquela execução. Se o servidor reiniciar, os dados desaparecem. Se houver duas instâncias, cada uma terá seu próprio array. Usaremos memória em exercícios iniciais apenas para isolar conceitos de HTTP antes de introduzir persistência.

12.3 — Sinais e encerramento

Processos podem receber sinais como SIGINT e SIGTERM. Em produção, não devemos apenas interromper tudo abruptamente: aprenderemos encerramento gracioso para parar de aceitar tráfego, concluir requisições possíveis e fechar recursos.

13 — Introdução ao event loop

O JavaScript do Node.js executa, por padrão, em uma thread principal. Mesmo assim, Node consegue coordenar muitas operações de I/O sem bloquear essa thread durante toda a espera. O event loop organiza quando callbacks e continuações ficam prontos para executar. 

console.log("A");

setTimeout(() => {
  console.log("B");
}, 0);

Promise.resolve().then(() => {
  console.log("C");
});

console.log("D");

// Ordem esperada neste exemplo: A, D, C, B

setTimeout(..., 0) não significa “execute agora”. Significa que o callback poderá ser agendado depois que o código atual terminar e as regras das filas permitirem. Promises usam uma fila de microtasks, processada com prioridade própria em relação a fases como timers.

13.1 — I/O não bloqueante versus código pesado

Esperar banco, arquivo ou rede pode ser coordenado de forma assíncrona. Porém, um cálculo JavaScript longo e síncrono ocupa a thread principal e atrasa todas as outras requisições daquele processo. 

app.get("/bad-example", (request, response) => {
  const end = Date.now() + 5_000;

  while (Date.now() < end) {
    // Bloqueia a thread principal por aproximadamente 5 segundos.
  }

  response.json({ ok: true });
});
“Node é single-thread” é uma simplificação. O JavaScript da aplicação normalmente usa uma thread principal, mas o runtime, o sistema operacional e o pool de threads podem participar de certas operações. O ponto mais importante agora é: não bloqueie a thread principal com trabalho síncrono demorado.

14 — Variáveis de ambiente

Variáveis de ambiente são configurações fornecidas ao processo pelo ambiente em que ele executa. Elas permitem mudar porta, conexão de banco, URLs externas e outros valores sem editar o código-fonte. 

const rawPort = process.env.PORT;
const port = rawPort ? Number(rawPort) : 3000;

if (!Number.isInteger(port) || port < 1 || port > 65_535) {
  throw new Error("PORT deve ser um inteiro entre 1 e 65535");
}

14.1 — Tudo chega como string ou undefined

process.env.PORT não é automaticamente um número. Escrever PORT=false também não cria um booleano; cria a string "false". Configurações precisam de parsing e validação.

14.2 — Arquivo .env

PORT=3000
DATABASE_URL=postgresql://user:password@localhost:5432/app
LOG_LEVEL=info

O arquivo .env é uma conveniência de desenvolvimento para carregar variáveis. Ele não deve ser tratado como cofre. Arquivos com segredos reais não devem ser versionados. Em produção, a plataforma ou um gerenciador de segredos costuma fornecer esses valores ao processo.

14.3 — Falhar cedo

Se uma configuração obrigatória está ausente, é melhor impedir a inicialização com uma mensagem clara do que aceitar tráfego e falhar de maneira confusa durante uma requisição. Mais adiante validaremos o ambiente inteiro com Zod.

15 — Ligando todos os conceitos

// Frontend: cliente
const response = await fetch("http://localhost:3000/tasks?completed=false", {
  headers: {
    Accept: "application/json",
  },
});

if (!response.ok) {
  throw new Error(`Status inesperado: ${response.status}`);
}

const data: unknown = await response.json();
// Backend: servidor
import express from "express";

export const app = express();

app.use(express.json());

app.get("/tasks", (request, response) => {
  const completed = request.query.completed;

  return response.status(200).json({
    filters: { completed },
    items: [],
  });
});

O frontend inicia uma requisição GET; a URL contém protocolo, host, porta, path e query; headers informam o formato aceito; o processo Node recebe a conexão; Express encontra a rota; o handler lê a query e envia uma resposta 200 com JSON serializado.

16 — Edge cases: conceito, propósito e prática

16.1 — O que é um edge case

Edge case, ou caso de borda, é uma situação que ocorre nos limites, extremos ou combinações menos comuns das entradas e estados de um sistema. É o caso que não representa o caminho mais frequente, mas ainda é possível e precisa ter comportamento definido.

Exemplo simples: em um campo que aceita idade de 18 a 120, valores 18 e 120 são limites; 17, 121, texto, campo ausente e número decimal são casos que testam as bordas do contrato.

16.2 — Edge case não é necessariamente bug

O caso extremo existe independentemente de o código estar certo. Ele se torna bug quando o comportamento observado é incorreto, inseguro, inconsistente ou não definido. Pensar em edge cases é antecipar situações antes que usuários ou produção as encontrem.

16.3 — Por que edge cases existem

  • Entradas possuem limites, formatos e valores ausentes.
  • Rede e serviços externos podem falhar, atrasar ou responder parcialmente.
  • Usuários repetem ações, clicam duas vezes e usam a aplicação fora da ordem esperada.
  • Requisições concorrentes podem tentar alterar o mesmo estado.
  • Dados antigos podem não seguir o contrato atual.
  • Processos reiniciam, clientes desconectam e timeouts acontecem.
  • Atacantes procuram justamente comportamentos não considerados.

16.4 — Para que servem no desenvolvimento

Edge cases ajudam a definir validações, regras de negócio, status HTTP, tratamento de erros, testes, observabilidade e decisões de arquitetura. Uma funcionalidade não está realmente especificada apenas pelo “caminho feliz”; precisamos decidir o que acontece quando algo falta, repete, conflita, expira ou falha.

16.5 — Caminho feliz, caso de erro e caso de borda

CategoriaExemplo
Caminho felizCriar tarefa com título válido e receber 201.
Caso de erro comumCriar tarefa sem título e receber erro de validação.
Caso de bordaTítulo exatamente no limite máximo; duas criações idênticas simultâneas; cliente desconecta durante a resposta.

16.6 — Como descobrir edge cases

  1. Liste entradas e identifique mínimo, máximo, vazio, ausente, nulo e formato incorreto.
  2. Repita a mesma ação e observe se deveria ser idempotente.
  3. Mude a ordem das ações.
  4. Considere duas requisições simultâneas.
  5. Imagine dependências lentas, indisponíveis ou retornando dados inesperados.
  6. Considere permissões, ownership e tentativa de acessar dados de outra pessoa.
  7. Interrompa o processo ou desconecte o cliente em pontos diferentes.

16.7 — Os cinco edge cases iniciais da Fase 0

1. Porta inválida ou indisponível

A variável PORT pode conter texto, zero, número acima de 65535 ou uma porta já ocupada por outro processo. Devemos validar o valor e tratar erros de inicialização. Sem isso, o servidor pode nem iniciar ou falhar com uma mensagem pouco amigável.

2. Path inexistente

O cliente pode pedir /taks em vez de /tasks, usar uma versão antiga da API ou solicitar um recurso removido. O servidor deve encerrar a requisição com resposta consistente, normalmente 404, sem deixar a conexão pendurada e sem responder 200 com mensagem de erro.

3. JSON malformado

O header pode declarar JSON enquanto o body contém vírgula sobrando, aspas ausentes ou texto qualquer. O parsing falha antes de a regra de negócio rodar. A API precisa transformar essa falha em resposta 4xx controlada, em vez de encerrar o processo ou devolver detalhes internos.

4. Cliente desconecta antes da resposta

O usuário pode fechar a aba, perder a rede ou cancelar a requisição enquanto o servidor ainda trabalha. Algumas operações podem ser canceladas; outras já produziram efeito e precisam terminar com segurança. Isso afeta logs, recursos, transações e decisões sobre idempotência.

5. Header esperado não foi enviado

A API pode exigir Content-Type, autenticação ou um identificador de idempotência. Header ausente, vazio ou em formato inválido deve resultar em comportamento explícito. Nunca assuma que o frontend oficial será o único cliente.

16.8 — Edge cases e testes

Um edge case importante deve virar exemplo de contrato e, quando possível, teste automatizado. Testes não substituem o raciocínio; eles preservam a decisão para impedir regressões durante refatorações.

17 — Exercícios orientados

Exercício 1 — Anatomia completa de uma URL

Use a URL https://api.example.com:8443/users/42/posts?published=true&page=3#comments.

  1. Identifique esquema, hostname, porta, path, query e fragmento.
  2. Explique quais partes chegam ao servidor em uma requisição HTTP comum.
  3. Diga qual porta seria usada se :8443 fosse removido.
Exercício 2 — Narre o caminho do React ao Express

Escreva, com suas palavras, o que acontece desde o clique em “Salvar tarefa” até a nova tarefa aparecer na tela. Inclua serialização, método, headers, body, rota, status, parsing e nova renderização.

Exercício 3 — Escolha status e justifique

Escolha um status para cada situação e explique por que não usaria 200 em todas:

  • Tarefa criada.
  • Tarefa inexistente.
  • JSON com sintaxe inválida.
  • Usuário autenticado sem permissão.
  • E-mail já cadastrado.
  • Falha inesperada do banco.
Exercício 4 — JSON versus objeto JavaScript

Crie um objeto com uma data, uma propriedade undefined e uma função. Execute JSON.stringify(), compare o resultado e descreva o que foi perdido ou transformado.

Exercício 5 — Mapeamento de edge cases

Para um endpoint POST /appointments com início, fim e participante, liste pelo menos dez edge cases. Organize-os por entrada, tempo, repetição, concorrência, permissão e falha externa.

18 — Mini projeto da Fase 0

O projeto será uma pequena API de diagnóstico. O objetivo não é arquitetura avançada, e sim observar HTTP de forma consciente. 

GET  /health
GET  /request-info
GET  /headers
GET  /status/:code
POST /echo
  • GET /health: informa se o processo está disponível.
  • GET /request-info: devolve método, path, query e protocolo observados.
  • GET /headers: devolve uma seleção segura de headers recebidos.
  • GET /status/:code: permite observar respostas com status controlados.
  • POST /echo: recebe JSON e devolve o conteúdo, deixando claro o ciclo de serialização.
A rota de headers não deve devolver credenciais ou cookies indiscriminadamente. Até uma ferramenta didática precisa evitar expor segredos recebidos.

19 — Checklist de domínio da Fase 0

  • Consigo diferenciar frontend/backend de cliente/servidor.
  • Consigo explicar API, endpoint e contrato.
  • Consigo narrar o ciclo completo de um fetch().
  • Sei decompor uma URL.
  • Sei diferenciar headers e body.
  • Entendo JSON como texto e sei explicar serialização e parsing.
  • Consigo escolher método e status sem tratar tudo como POST e 200.
  • Entendo o significado de stateless.
  • Sei que o servidor é um processo com memória e ciclo de vida.
  • Entendo, em nível introdutório, por que código síncrono pesado bloqueia o event loop.
  • Sei que variáveis de ambiente precisam de parsing e validação.
  • Consigo identificar e justificar edge cases.

Fase 6

Controllers, services, repositories e regras de negócio

Express 5TypeScriptArquitetura em camadasRegras de negócioSistema de pedidos

Esta fase ensina a separar responsabilidades sem transformar a aplicação em uma coleção de arquivos vazios. A meta não é seguir uma arquitetura por aparência, mas criar limites que tornem o código compreensível, testável e seguro para evoluir.

Objetivo: reconhecer o papel de routes, controllers, services, repositories, schemas, mappers e DTOs; decidir onde cada regra pertence; controlar dependências; organizar módulos; e implementar um fluxo de pedidos com regras reais de estoque, pagamento, cupom e cancelamento.
requisição HTTP ↓ route + middlewares + schema ↓ controller — traduz HTTP ↓ service / use case — executa o negócio ↓ repository — lê e grava dados ↓ service decide o resultado ↓ controller traduz para HTTP ↓ resposta

1 — Modelo mental

1. A borda HTTP recebe uma intenção em formato de rota, parâmetros e body.
2. O schema garante que a entrada tem formato aceitável.
3. O controller extrai contexto HTTP e chama um caso de uso.
4. O service consulta dados e aplica regras de negócio.
5. O repository esconde detalhes de persistência.
6. O resultado volta e é traduzido para uma resposta HTTP.
Regra central: uma camada não existe para mover linhas de lugar. Ela deve possuir uma responsabilidade identificável e reduzir algum tipo de acoplamento.

2 — O problema do handler que faz tudo

app.post("/orders", async (request, response) => {
  const input = createOrderSchema.parse(request.body);
  const product = await prisma.product.findUnique({ where: { id: input.productId } });
  if (!product) return response.status(404).json({ error: "Produto não encontrado" });
  if (!product.active) return response.status(409).json({ error: "Produto inativo" });
  if (input.quantity <= 0) return response.status(422).json({ error: "Quantidade inválida" });
  if (product.stock < input.quantity) return response.status(409).json({ error: "Estoque insuficiente" });
  const order = await prisma.order.create({ data: { /* ... */ } });
  return response.status(201).json(order);
});

O código pode funcionar, mas mistura protocolo HTTP, validação, persistência, regras, cálculo e serialização. Ao crescer, cada mudança exige entender tudo ao mesmo tempo.

3 — Responsabilidade e motivo para mudar

Uma forma útil de pensar responsabilidade é perguntar: por qual motivo este código mudaria?

ParteMuda quando...
Routea URL, método ou pipeline de middlewares muda
Controllero contrato HTTP ou formato da resposta muda
Serviceuma regra ou fluxo da aplicação muda
Repositorya estratégia de persistência ou consulta muda
Schemao formato permitido na fronteira muda
Mappera representação exposta ou persistida muda

4 — Fluxo completo e direção das dependências

route → controller → service → repository ↓ ↓ HTTP regras de negócio schema protege a entrada mapper protege a saída

A seta representa conhecimento direto. O repository não deveria importar Express. O service não deveria receber Request nem devolver Response. Quanto mais para dentro, menos o código deve saber sobre transporte.

5 — O que é uma regra de negócio

Regra de negócio é uma restrição, decisão ou cálculo que existe por causa do domínio da aplicação, não por causa do Express.

Regras de formato

  • quantity deve ser número inteiro positivo.
  • couponCode tem no máximo 40 caracteres.

Normalmente ficam no schema.

Regras de negócio

  • produto inativo não pode ser comprado.
  • pedido pago não pode ser alterado.
  • cancelar restaura estoque.

Normalmente ficam no service/domínio.

Validade sintática não é validade de negócio: um UUID pode estar bem formado e ainda representar um produto inexistente ou indisponível.

6 — Route: URL, método e pipeline

ordersRouter.post(
  "/",
  authenticate,
  validate({ body: createOrderSchema }),
  orderController.create,
);

A route associa o endpoint às etapas. Ela não deve calcular total, consultar estoque ou montar queries extensas.

7 — Controller: traduz HTTP para aplicação

export class OrderController {
  constructor(private readonly createOrder: CreateOrderService) {}

  create = async (request: Request, response: Response): Promise<void> => {
    const input = response.locals.validated.body;
    const userId = response.locals.auth.userId;

    const order = await this.createOrder.execute({ userId, ...input });

    response.status(201).json(toOrderResponse(order));
  };
}

O controller conhece HTTP: status, headers, request, response e autenticação já extraída. Ele orquestra pouco e não decide regras do domínio.

Teste prático: se você precisaria copiar a regra para executar o mesmo caso de uso por uma fila ou CLI, ela provavelmente não pertence ao controller.

8 — Service / use case: executa o negócio

export class CreateOrderService {
  constructor(
    private readonly products: ProductRepository,
    private readonly orders: OrderRepository,
  ) {}

  async execute(input: CreateOrderInput): Promise<Order> {
    if (input.items.length === 0) throw new EmptyOrderError();

    const products = await this.products.findManyByIds(
      input.items.map(item => item.productId),
    );

    const order = Order.create({ customerId: input.userId, items: input.items, products });
    await this.orders.save(order);
    return order;
  }
}

O service representa uma ação da aplicação: criar pedido, pagar pedido, cancelar pedido. Nomes orientados a caso de uso costumam ser mais precisos que um enorme OrderService genérico.

9 — Repository: lê e grava dados

export interface OrderRepository {
  findById(id: string): Promise<Order | null>;
  save(order: Order): Promise<void>;
}

export class InMemoryOrderRepository implements OrderRepository {
  private readonly items = new Map<string, Order>();

  async findById(id: string): Promise<Order | null> {
    return this.items.get(id) ?? null;
  }

  async save(order: Order): Promise<void> {
    this.items.set(order.id, order);
  }
}

O repository expressa operações necessárias pelo domínio. Ele não deve virar apenas um espelho de todos os métodos do ORM.

Evite abstração cosmética: um método que apenas chama prisma.order.create(data) pode ser aceitável, mas só vale a camada se ela criar um limite útil: consultas próprias, mapeamento, substituição em testes ou isolamento do ORM.

10 — Schema: valida a fronteira

const createOrderSchema = z.strictObject({
  items: z.array(z.object({
    productId: z.uuid(),
    quantity: z.number().int().positive(),
  })).min(1),
  couponCode: z.string().trim().max(40).optional(),
});

O schema verifica forma, tipos, limites básicos e normalização. Ele não deve consultar banco para decidir toda regra do domínio.

11 — Mapper: traduz representações

export function toOrderResponse(order: Order): OrderResponse {
  return {
    id: order.id,
    status: order.status,
    totalInCents: order.totalInCents,
    items: order.items.map(item => ({
      productId: item.productId,
      quantity: item.quantity,
      unitPriceInCents: item.unitPriceInCents,
    })),
    createdAt: order.createdAt.toISOString(),
  };
}

Não exponha diretamente modelos do ORM. O mapper impede vazamento de campos internos, converte datas e mantém o contrato da API explícito.

Esses formatos podem coincidir em projetos pequenos, mas são conceitos diferentes. Separá-los quando necessário reduz acoplamento.

13 — Dependências e acoplamento

Uma dependência é algo que uma parte precisa conhecer para funcionar. Dependências inevitáveis devem apontar para direções previsíveis.

Acoplamento aceitável

  • controller conhece service.
  • service conhece contrato do repository.
  • implementação do repository conhece Prisma.

Acoplamento problemático

  • service importa Response.
  • entidade chama Prisma diretamente.
  • repository decide status HTTP.

14 — Injeção de dependências sem framework

const productRepository = new PrismaProductRepository(prisma);
const orderRepository = new PrismaOrderRepository(prisma);
const createOrderService = new CreateOrderService(productRepository, orderRepository);
const orderController = new OrderController(createOrderService);

Injeção de dependências significa fornecer colaboradores pelo lado de fora. Não exige container ou decorators. Construtores explícitos são suficientes em muitos projetos.

15 — Interfaces: quando ajudam

Interfaces ajudam quando existe uma fronteira relevante: banco real versus memória, provedor de pagamento, relógio, gerador de IDs, e-mail ou fila.

export interface Clock { now(): Date; }
export interface PaymentGateway {
  charge(input: ChargeInput): Promise<ChargeResult>;
}
Não crie interface por ritual: uma interface com uma única implementação não é automaticamente ruim, mas precisa justificar o custo com testabilidade, isolamento de tecnologia ou estabilidade do contrato.

16 — Transações e regras que atravessam repositories

Criar pedido pode reduzir estoque e gravar o pedido. Se uma operação falhar depois da outra, o sistema fica inconsistente. Esse conjunto deve ser atômico.

await unitOfWork.run(async ({ products, orders }) => {
  await products.decreaseStock(items);
  await orders.save(order);
});

A transação é uma preocupação de persistência, mas o service define quais operações precisam acontecer juntas. A implementação concreta pode usar a transação do ORM.

17 — Concorrência: estoque não é apenas um if

if (product.stock < quantity) throw new InsufficientStockError();
await products.decreaseStock(product.id, quantity);

Duas requisições podem ler o mesmo estoque antes da redução. A solução real precisa de atualização condicional, lock, controle otimista ou outra estratégia do banco.

Regra em memória não garante consistência: o service expressa a intenção, e o repository deve oferecer uma operação segura, como reserveStock(), cujo sucesso é decidido atomicamente.

18 — Regras puras e entidades

export function calculateOrderTotal(items: readonly PricedItem[]): number {
  return items.reduce((total, item) => total + item.unitPriceInCents * item.quantity, 0);
}

Regras puras não acessam rede, banco ou relógio global. São fáceis de testar. Regras que dependem de horário devem receber um relógio ou o instante atual.

É simples e didática. Em aplicações pequenas, localizar “todos os controllers” é fácil. Com muitos módulos, arquivos do mesmo recurso ficam distantes.

Organizar por feature favorece coesão: arquivos que mudam juntos ficam próximos. Não significa que tudo dentro do módulo pode se importar livremente.

21 — A pasta shared

shared deve conter elementos realmente compartilhados: configuração, banco, logger, middleware genérico e abstrações estáveis. Evite transformá-la em uma gaveta de utilitários sem dono.

A comparação ajuda, mas não é identidade. O backend é a autoridade das regras e precisa lidar com concorrência, persistência, segurança e múltiplos clientes.

23 — Quando separar arquivos e classes

  • Separe quando existe uma responsabilidade identificável.
  • Separe quando a parte pode ser testada isoladamente.
  • Separe quando a dependência tecnológica precisa ficar contida.
  • Não separe uma função de três linhas apenas para obedecer a uma árvore de pastas.
  • Comece simples e extraia quando o limite ficar claro.
Arquitetura saudável é proporcional: um CRUD pequeno pode usar funções e módulos simples. Um fluxo de pagamento merece limites mais fortes.

24 — Erros comuns

Controller gordo

Consulta banco, calcula, valida regra e serializa tudo.

Service HTTP

Recebe Request ou devolve status code.

Repository genérico

CRUD universal que não expressa operações do domínio.

Entidade anêmica por obrigação

Todo comportamento fica em services gigantes.

Arquivo vazio

Cada camada apenas chama a próxima sem acrescentar limite.

ORM vazando

Tipos e modelos do Prisma atravessam toda a aplicação.

25 — Edge cases arquiteturais e de negócio

  • Pedido sem itens.
  • Mesmo produto repetido em várias linhas.
  • Produto removido entre validação e gravação.
  • Produto ativo ao consultar e inativo ao confirmar.
  • Estoque suficiente para uma requisição, mas não para duas concorrentes.
  • Cupom expira exatamente durante a criação.
  • Cupom válido, mas incompatível com o cliente ou produtos.
  • Total calculado com preço antigo enquanto o catálogo muda.
  • Pedido pago recebe tentativa de edição ou cancelamento.
  • Cancelamento chamado duas vezes.
  • Estoque restaurado parcialmente e operação falha.
  • Gateway cobra, mas a gravação do pagamento falha.
  • Retry cria pedido ou cobrança duplicada.

Edge cases mostram por que separar responsabilidades importa: validação, regra, transação, idempotência e integração são problemas diferentes.

26 — Exemplo integrado: cancelar pedido

export class CancelOrderService {
  constructor(
    private readonly orders: OrderRepository,
    private readonly inventory: InventoryRepository,
    private readonly unitOfWork: UnitOfWork,
    private readonly clock: Clock,
  ) {}

  async execute(input: { orderId: string; actorId: string }): Promise<Order> {
    return this.unitOfWork.run(async () => {
      const order = await this.orders.findById(input.orderId);
      if (!order) throw new OrderNotFoundError();

      order.cancel({ actorId: input.actorId, cancelledAt: this.clock.now() });
      await this.inventory.restore(order.items);
      await this.orders.save(order);

      return order;
    });
  }
}
cancel = async (request: Request, response: Response): Promise<void> => {
  const { orderId } = response.locals.validated.params;
  const actorId = response.locals.auth.userId;
  const order = await this.cancelOrder.execute({ orderId, actorId });
  response.status(200).json(toOrderResponse(order));
};

O controller traduz. O service coordena. A entidade protege estado. Repositories persistem. A transação garante atomicidade.

27 — Como as camadas melhoram testes

it("bloqueia produto inativo", async () => {
  const products = new InMemoryProductRepository([
    productFactory({ id: "p1", active: false }),
  ]);
  const orders = new InMemoryOrderRepository();
  const service = new CreateOrderService(products, orders);

  await expect(service.execute({
    userId: "u1",
    items: [{ productId: "p1", quantity: 1 }],
  })).rejects.toBeInstanceOf(InactiveProductError);
});

O teste de regra não precisa subir Express nem banco real. Testes de controller verificam tradução HTTP; testes de repository verificam integração com o banco.

  4. Modele payOrder sem retornar status HTTP.

29 — Mini projeto: sistema de pedidos

Implemente uma API de pedidos em memória primeiro; depois ela poderá receber Prisma na Fase 9.

29.1 — Recursos e rotas

POST   /orders
GET    /orders/:orderId
POST   /orders/:orderId/pay
POST   /orders/:orderId/cancel
GET    /orders?status=PENDING&page=1&limit=20

29.2 — Regras obrigatórias

  • Pedido vazio é proibido.
  • Produto inativo é bloqueado.
  • Quantidade deve ser positiva.
  • Estoque precisa ser suficiente.
  • Cupom expirado não pode ser aplicado.
  • Pedido pago é imutável.
  • Cancelamento restaura estoque.
  • Cancelamento repetido não restaura estoque duas vezes.
  • Preço usado no pedido deve ser registrado como snapshot.

29.3 — Camadas mínimas

orders/
├── order.routes.ts
├── order.controller.ts
├── create-order.service.ts
├── pay-order.service.ts
├── cancel-order.service.ts

29.4 — Critérios

  • Nenhum service importa Express.
  • Nenhum repository decide status HTTP.
  • Schemas não substituem regras de negócio.
  • Controllers não consultam banco diretamente.
  • Erros do domínio são específicos.
  • Casos de uso funcionam com repositories em memória.
  • Cancelamento e restauração de estoque são tratados como uma unidade lógica.

30 — Checklist para avançar

  • Consigo explicar o motivo de existir de cada camada.
  • Sei diferenciar validação de formato e regra de negócio.
  • Meu controller apenas traduz HTTP e chama um caso de uso.
  • Meu service não depende de Express.
  • Meu repository não expõe detalhes desnecessários do ORM.
  • Sei quando uma interface ajuda e quando é ritual.
  • Entendo por que transações são necessárias em operações compostas.
  • Reconheço que concorrência exige garantia no banco.
  • Consigo organizar por camada e por módulo.
  • Sei testar regras sem iniciar servidor.
  • Consigo identificar uma abstração vazia.
  • Implementei o sistema de pedidos e seus edge cases essenciais.

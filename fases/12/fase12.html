1. O que significa colocar uma API em produção

Executar npm run dev em outra máquina não é uma estratégia de produção.

Em desenvolvimento, temos um terminal aberto, poucas requisições e liberdade para reiniciar tudo. Em produção, a aplicação atende usuários, depende de recursos remotos e precisa explicar o próprio comportamento sem que alguém esteja olhando diretamente para ela.

Desenvolvimento

  • feedback rápido;
  • logs legíveis;
  • hot reload;
  • dados descartáveis;
  • uma instância local.

Produção

  • previsibilidade;
  • segurança;
  • diagnóstico;
  • recuperação;
  • mudanças controladas.
Ideia centralUma API pronta para produção não é a que nunca falha. É a que falha de maneira visível, controlada e recuperável.

2. Observabilidade

Observabilidade é a capacidade de compreender o estado interno do sistema a partir dos sinais que ele produz. Os três sinais tradicionais são logs, métricas e traces.

Logs

Eventos individuais, com contexto e detalhes de execução.

Métricas

Valores agregados: volume, erro, latência e saturação.

Traces

Caminho de uma operação entre serviços e dependências.

Monitoramento não é exatamente observabilidade

Monitoramento responde perguntas já conhecidas, como “a CPU está alta?”. Observabilidade ajuda a investigar perguntas novas, como “por que apenas requisições de uma rota ficaram lentas depois do deploy?”.

4. Níveis de log

trace

Detalhe extremo. Quase sempre desativado em produção.

debug

Diagnóstico técnico durante investigação.

info

Evento normal e relevante.

warn

Situação inesperada, mas recuperável.

error

Uma operação falhou.

fatal

O processo não pode continuar com segurança.

Não transforme tudo em error. Se todo evento é urgente, nenhum é urgente.

5. Request ID

O Request ID identifica uma requisição e permite juntar logs produzidos por middlewares, controllers, services e repositories.

cliente ↓ x-request-id middleware ↓ controller ↓ service ↓ repository ↓ resposta
import { randomUUID } from "node:crypto";
import type { RequestHandler } from "express";

export const requestIdMiddleware: RequestHandler = (
  request,
  response,
  next,
) => {
  const incomingRequestId =
    request.header("x-request-id");

  const requestId =
    incomingRequestId ?? randomUUID();

  response.locals.requestId = requestId;
  response.setHeader("x-request-id", requestId);

  next();
};

Log HTTP

export const httpLogger: RequestHandler = (
  request,
  response,
  next,
) => {
  const startedAt = performance.now();

  response.on("finish", () => {
    const durationMs =
      performance.now() - startedAt;

    logger.info({
      event: "http.request.completed",
      requestId: response.locals.requestId,
      method: request.method,
      path: request.originalUrl,
      statusCode: response.statusCode,
      durationMs: Number(durationMs.toFixed(2)),
    });
  });

  next();
};

AsyncLocalStorage

Quando passar o contexto manualmente por todas as funções se torna inconveniente, AsyncLocalStorage pode manter o Request ID ao longo da cadeia assíncrona.

import { AsyncLocalStorage } from "node:async_hooks";

type RequestContext = {
  requestId: string;
};

export const requestContext =
  new AsyncLocalStorage<RequestContext>();

6. Métricas

Logs descrevem eventos individuais. Métricas representam valores agregados ao longo do tempo.

Counter

Só cresce: requisições totais, falhas totais.

Gauge

Sobe e desce: conexões ativas, tamanho da fila.

Histogram

Distribuição: duração das requisições.

Alta cardinalidade

Não use IDs únicos como labelsrequestId, userId, e-mail e URLs completas criam séries demais. Use rota normalizada, método e status.

7. Health check

Health checks são endpoints operacionais. Eles não devem expor credenciais, stack traces ou detalhes internos desnecessários.

GET /health/live → O processo está vivo? GET /health/ready → Banco e dependências estão prontos?

8. Liveness

Liveness responde apenas se o processo está vivo.

app.get("/health/live", (_request, response) => {
  return response.status(200).json({
    status: "alive",
  });
});
Não consulte o banco aquiSe o banco cair, reiniciar todas as instâncias da API repetidamente pode piorar o incidente.

9. Readiness

Readiness indica se a aplicação deve receber tráfego. Ela pode verificar banco, cache obrigatório, fila essencial, inicialização e estado de shutdown.

app.get("/health/ready", async (
  _request,
  response,
) => {
  if (applicationState.shuttingDown) {
    return response.status(503).json({
      status: "not_ready",
    });
  }

  try {
    await prisma.$queryRaw`SELECT 1`;

    return response.status(200).json({
      status: "ready",
    });
  } catch {
    return response.status(503).json({
      status: "not_ready",
    });
  }
});

Resultado esperado

Banco disponível

200 — a instância pode receber tráfego.

Banco indisponível

503 — a instância permanece viva, mas não pronta.

Use operações baratas e timeouts curtos. Uma verificação pesada executada frequentemente pode causar o próprio problema.

10. Startup check

Uma aplicação pode estar viva, mas ainda inicializando. Startup checks evitam que uma inicialização lenta seja confundida com falha permanente.

São úteis quando o processo precisa carregar configuração, conectar dependências, aquecer cache ou preparar modelos antes de atender tráfego.

11. Graceful shutdown

Quando a infraestrutura envia SIGTERM ou SIGINT, a aplicação deve parar de receber novas requisições, concluir o que estiver em andamento e fechar recursos.

sinal recebido ↓ readiness = false ↓ parar novas conexões ↓ concluir operações em andamento ↓ fechar banco e filas ↓ encerrar processo
const shutdownTimeoutMs = 10_000;
let isShuttingDown = false;

async function shutdown(signal: string): Promise<void> {
  if (isShuttingDown) return;

  isShuttingDown = true;
  applicationState.shuttingDown = true;

  logger.info({
    event: "server.shutdown.started",
    signal,
  });

  const timer = setTimeout(() => {
    logger.fatal({
      event: "server.shutdown.timeout",
    });

    process.exit(1);
  }, shutdownTimeoutMs);

  timer.unref();

  server.close(async (error) => {
    await prisma.$disconnect();

    clearTimeout(timer);

    logger.info({
      event: "server.shutdown.completed",
    });

    process.exit(error ? 1 : 0);
  });
}

process.on("SIGTERM", () => {
  void shutdown("SIGTERM");
});

process.on("SIGINT", () => {
  void shutdown("SIGINT");
});

server.close() para de aceitar novas conexões e aguarda o encerramento das conexões atendidas, mas filas, WebSockets, timers e outros recursos precisam de tratamento próprio.

12. Exceções e rejeições não tratadas

process.on("uncaughtException", (error) => {
  logger.fatal({
    event: "process.uncaught_exception",
    error,
  });

  void shutdown("uncaughtException");
});

process.on("unhandledRejection", (reason) => {
  logger.fatal({
    event: "process.unhandled_rejection",
    reason,
  });

  void shutdown("unhandledRejection");
});
Última linha de defesaEsses handlers não substituem o tratamento normal. Depois de uma exceção não tratada, o processo pode estar inconsistente; registre, encerre e permita que a infraestrutura o substitua.

13. Configuração por ambiente

Configuração é tudo que muda entre ambientes sem exigir alteração do código: porta, banco, nível de log, CORS, segredo, timeout e endereços de serviços.

import { z } from "zod";

const envSchema = z.object({
  NODE_ENV: z
    .enum(["development", "test", "production"])
    .default("development"),
  PORT: z.coerce.number().int().positive()
    .default(3333),
  LOG_LEVEL: z
    .enum([
      "trace", "debug", "info",
      "warn", "error", "fatal",
    ])
    .default("info"),
  DATABASE_URL: z.string().min(1),
  CORS_ORIGIN: z.string().url(),
});

const parsed = envSchema.safeParse(process.env);

if (!parsed.success) {
  console.error(
    "Invalid environment configuration",
    parsed.error.flatten().fieldErrors,
  );

  process.exit(1);
}

export const env = parsed.data;
Fail fastÉ melhor recusar a inicialização do que iniciar parcialmente com segredo ausente ou URL inválida.

O arquivo .env é conveniente localmente. Em produção, segredos devem ser injetados pela plataforma ou por um secret manager — nunca copiados para a imagem.

14. Docker

Docker empacota a aplicação e suas dependências em uma imagem reproduzível.

Dockerfile imagem container registry volume network

Dockerfile multi-stage

# syntax=docker/dockerfile:1

FROM node:24-alpine AS dependencies
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

FROM node:24-alpine AS build
WORKDIR /app
COPY --from=dependencies /app/node_modules ./node_modules
COPY package.json package-lock.json tsconfig.json ./
COPY prisma ./prisma
COPY src ./src
RUN npm run build
RUN npm prune --omit=dev

FROM node:24-alpine AS production
ENV NODE_ENV=production
WORKDIR /app

RUN addgroup -S nodeapp \
  && adduser -S nodeapp -G nodeapp

COPY --from=build --chown=nodeapp:nodeapp \
  /app/package.json ./
COPY --from=build --chown=nodeapp:nodeapp \
  /app/node_modules ./node_modules
COPY --from=build --chown=nodeapp:nodeapp \
  /app/dist ./dist
COPY --from=build --chown=nodeapp:nodeapp \
  /app/prisma ./prisma

USER nodeapp
EXPOSE 3000

CMD ["node", "dist/server.js"]

Por que vários estágios?

A imagem final não precisa carregar TypeScript, código fonte, ferramentas de build, caches e dependências de desenvolvimento.

17. Deploy

Deploy é o processo controlado de disponibilizar uma nova versão.

commit ↓ pipeline ↓ lint + typecheck + testes ↓ build da imagem ↓ registry ↓ migrations ↓ nova versão ↓ readiness ↓ tráfego

Recreate

Versão antiga para antes da nova. Simples, mas pode gerar indisponibilidade.

Rolling

Instâncias são substituídas gradualmente.

Blue-green

Dois ambientes; o tráfego muda após validação.

Canary

Uma pequena parcela recebe a nova versão primeiro.

Rollback

Retornar a imagem é apenas parte do rollback. Banco, contrato e dados escritos pela nova versão também precisam ser compatíveis.

18. Migrations em produção

Antes de executar uma migration, responda: quem executa, quando executa, quanto bloqueia, se é repetível, se há backup e se as versões antiga e nova continuam compatíveis.

Expand and contract

  1. expanda o schema de maneira compatível;
  2. publique código que aceite os dois formatos;
  3. migre os dados;
  4. remova o formato antigo depois.
Evite mudanças destrutivas imediatasRemover ou renomear uma coluna no mesmo deploy pode quebrar instâncias antigas durante um rolling deployment.

19. OpenAPI e Swagger

OpenAPI é uma especificação padronizada para descrever APIs HTTP. Swagger é um ecossistema de ferramentas que utiliza essa especificação.

OpenAPI

Contrato: endpoints, parâmetros, body, respostas, schemas e autenticação.

Swagger

Swagger UI, Editor, Codegen e outras ferramentas.

Code-first ou design-first?

Code-first reduz duplicação e fica próximo da implementação. Design-first incentiva discussão do contrato antes do código. As duas abordagens precisam de testes para evitar divergência.

21. Edge cases de produção

Banco fora do ar

Liveness permanece 200; readiness retorna 503.

SIGTERM durante requisição

A instância sai do tráfego e tenta concluir a operação.

Shutdown travado

O timeout força encerramento e gera log fatal.

Health check caro

A verificação pode causar carga e agravar o incidente.

URL em métrica

/users/1 e /users/2 geram cardinalidade; use /users/:userId.

Instância lenta

Pode continuar “saudável”; latência e saturação revelam degradação.

Migration incompatível

Versões antiga e nova podem coexistir durante o deploy.

Docs expostas

Documentação privada pode revelar endpoints administrativos.

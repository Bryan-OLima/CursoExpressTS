Fase 4

Middlewares e pipeline da requisição

Express 5TypeScriptObservabilidadeAutenticaçãoAPI administrativa

Middleware é uma função que participa do caminho de uma requisição dentro da aplicação. Ela pode observar dados, acrescentar contexto, transformar a requisição, encerrar a resposta ou entregar o controle para a próxima etapa.

Objetivo da fase: ao final, você deverá conseguir visualizar a aplicação como uma pilha ordenada, escolher o escopo correto para cada middleware, usar next() de forma consciente, compartilhar contexto com res.locals, registrar logs de conclusão e separar autenticação de autorização.
requisição ↓ request ID ↓ logger / cronômetro ↓ autenticação ↓ autorização ↓ validação ↓ controller ↓ resposta

1 — O modelo mental desta fase

1. O Express recebe uma requisição.
2. Ele percorre camadas na ordem em que foram registradas.
3. Cada camada decide se não combina, continua, responde ou encaminha um erro.
4. Dados produzidos antes podem ser usados pelas etapas seguintes.
5. Quando uma resposta termina, callbacks ligados à resposta podem registrar o resultado final.

Uma rota não é um bloco isolado. Ela faz parte da mesma pilha que contém parsers, routers, autenticação, handlers e tratamento de erros.

2 — O que é um middleware

Em Express, um middleware comum recebe request, response e next. O termo descreve uma posição no fluxo, não uma finalidade específica. Logger, autenticação, CORS, parsing de JSON e a própria função final de uma rota participam do processamento.

import type { NextFunction, Request, Response } from "express";

export function exampleMiddleware(
  request: Request,
  response: Response,
  next: NextFunction,
): void {
  console.log(request.method, request.path);
  next();
}
Mesmo que pareça óbvio: declarar a função não a executa. Ela somente entra no pipeline quando é registrada com app.use(), router.use() ou em uma rota.

3 — A assinatura e o significado dos parâmetros

ParâmetroFunçãoUso comum
requestRepresenta a entrada HTTP.Ler method, path, headers, params, query e body.
responseRepresenta a resposta em construção.Definir status, headers, body e res.locals.
nextEntrega o controle ao próximo item compatível.next() ou next(error).

Use void para middlewares síncronos que não precisam retornar uma Promise. Um return response.status(...).json(...) pode ser útil para interromper a execução da função, mas o valor retornado não vira a resposta por magia; quem envia é response.json().

4 — O que next() realmente faz

next() pede ao Express que continue procurando a próxima camada compatível depois da camada atual. Ele não reinicia a aplicação, não chama obrigatoriamente a rota seguinte no arquivo e não encerra a função JavaScript.

export function attachMessage(
  request: Request,
  response: Response,
  next: NextFunction,
): void {
  response.locals.message = "Criado no middleware";
  next();

  // Este código ainda pode executar depois que as etapas seguintes terminarem.
  console.log("A pilha retornou para attachMessage");
}

4.1 — next() não equivale a return

next();
console.log("Isto ainda executa");

Quando nada deve acontecer depois, escreva return next() ou chame next() e encerre a função explicitamente. O objetivo é legibilidade e evitar fluxo acidental.

5 — Toda etapa precisa continuar ou encerrar

Continuar

next();

Outra camada ficará responsável por responder.

Encerrar

response.status(401).json({
  error: "UNAUTHENTICATED",
});

A própria etapa conclui a resposta.

export function brokenMiddleware(
  request: Request,
  response: Response,
  next: NextFunction,
): void {
  console.log("Recebi a requisição");
  // Não respondeu e não chamou next().
}
Resultado: a conexão pode permanecer aguardando porque o pipeline não foi continuado nem encerrado. Esse é um dos bugs fundamentais desta fase.

6 — Ordem de registro

A pilha do Express é ordenada. Em geral, uma camada registrada depois não consegue preparar algo necessário para uma camada que executou antes.

app.use(requestId);
app.use(requestLogger);
app.use(authenticate);
app.use("/admin", requireRole("ADMIN"), adminRouter);
app.use(notFoundHandler);
app.use(errorHandler);

Nesse desenho, o logger já enxerga o request ID. A autenticação ocorre antes da autorização. O 404 fica depois das rotas. O middleware de erro fica no fim.

6.1 — Ordem dentro de uma rota

router.delete(
  "/users/:userId",
  authenticate,
  requireRole("ADMIN"),
  deleteUserController,
);

Trocar autenticação e autorização pode fazer a autorização procurar um usuário que ainda não foi identificado.

7 — Middleware global

Registrado no app sem path restritivo, participa de todas as requisições que chegam depois dele.

app.use(express.json());
app.use(requestId);
app.use(requestLogger);

É apropriado para responsabilidades realmente transversais: parsing, request ID, logs, headers gerais e configuração de segurança. “Global” não significa “deve conter todas as regras”.

Custo global: uma operação cara adicionada globalmente afeta inclusive /health, rotas públicas e endpoints que não precisam dela.

8 — Middleware por router

Um Router possui sua própria pilha. Isso permite aplicar uma política a um conjunto coerente de endpoints.

import { Router } from "express";

export const adminRouter = Router();

adminRouter.use(authenticate);
adminRouter.use(requireRole("ADMIN"));

adminRouter.get("/audit-logs", listAuditLogsController);
adminRouter.delete("/users/:userId", deleteUserController);

app.use("/admin", adminRouter);

Esse desenho reduz repetição e deixa explícito que o módulo administrativo compartilha uma fronteira de acesso.

9 — Middleware por rota

Use quando a regra pertence a uma operação específica, não ao router inteiro.

router.patch(
  "/projects/:projectId",
  authenticate,
  requireProjectMembership,
  updateProjectController,
);

A verificação de participação no projeto depende do recurso do path; aplicá-la globalmente seria artificial.

10 — Vários handlers e subpilhas

Uma rota pode receber várias funções. O Express executa a sequência enquanto cada uma chama next().

router.post(
  "/reports",
  authenticate,
  requireRole("ADMIN"),
  attachReportContext,
  createReportController,
);

Também é possível passar arrays, mas evite abstrações que escondam demais a ordem. Para quem lê a rota, a cadeia de responsabilidades deve permanecer compreensível.

11 — Montagem, path e URLs

Ao montar um router, o prefixo do app.use() combina com o path interno.

app.use("/api/v1/admin", adminRouter);

// Dentro do router:
adminRouter.get("/users", listUsersController);
// Endpoint final: GET /api/v1/admin/users
PropriedadeIdeia
request.originalUrlURL original recebida, incluindo o prefixo de montagem.
request.baseUrlParte do path em que o router foi montado.
request.pathPath relativo percebido no contexto atual.

Em logs centrais, originalUrl costuma ser mais informativo que somente path.

12 — Middlewares built-in e de terceiros

Nem todo middleware precisa ser escrito por você. express.json(), express.urlencoded() e express.static() são middlewares fornecidos pelo Express. Bibliotecas como CORS, compressão e loggers também podem fornecer middlewares.

app.use(express.json({ limit: "1mb" }));
Terceiros ainda fazem parte da sua aplicação: revise finalidade, configuração, manutenção e impacto. Instalar um middleware não elimina a necessidade de entender em que posição ele executa.

13 — Compartilhando dados com res.locals

response.locals é um objeto ligado à resposta atual. Ele serve para guardar dados produzidos por uma etapa e consumidos pelas seguintes, sem criar estado global compartilhado entre requisições.

type AuthenticatedUser = {
  id: string;
  role: "ADMIN" | "MEMBER";
};

export function authenticate(
  request: Request,
  response: Response,
  next: NextFunction,
): void {
  const user: AuthenticatedUser = {
    id: "user-42",
    role: "ADMIN",
  };

  response.locals.user = user;
  next();
}

13.1 — Tipando locals

type RequestLocals = {
  requestId: string;
  user?: AuthenticatedUser;
};

type AppResponse = Response;

Para uma aplicação maior, você pode padronizar aliases ou fazer module augmentation. Nesta fase, prefira tipos explícitos e fáceis de rastrear.

Evite: guardar o usuário atual em uma variável global. Requisições concorrentes poderiam sobrescrever o estado umas das outras.

14 — Request ID

Um request ID identifica uma execução HTTP específica. Ele permite correlacionar as linhas de log produzidas por diferentes camadas da mesma requisição.

import { randomUUID } from "node:crypto";
import type { NextFunction, Request, Response } from "express";

export function requestId(
  request: Request,
  response: Response,
  next: NextFunction,
): void {
  const id = randomUUID();

  response.locals.requestId = id;
  response.setHeader("x-request-id", id);

  next();
}

14.1 — Aceitar ID vindo do cliente?

Pode ser útil para propagação entre serviços, mas não confie cegamente em qualquer tamanho ou formato. Uma estratégia segura é validar um header recebido ou gerar um valor interno e guardar separadamente IDs externos de correlação.

15 — Logging de requisição

Um bom log de acesso registra o desfecho, não apenas a chegada. Método, URL, status, duração e request ID formam uma base útil.

export function requestLogger(
  request: Request,
  response: Response,
  next: NextFunction,
): void {
  const startedAt = performance.now();

  response.on("finish", () => {
    const durationMs = performance.now() - startedAt;

    console.log({
      requestId: response.locals.requestId,
      method: request.method,
      url: request.originalUrl,
      statusCode: response.statusCode,
      durationMs: Number(durationMs.toFixed(2)),
    });
  });

  next();
}

Em produção, um logger estruturado costuma ser preferível a strings montadas manualmente, pois facilita busca, filtros e ingestão por ferramentas.

15.1 — O que não registrar

  • Senhas, tokens completos, cookies de sessão e chaves.
  • Body inteiro por padrão.
  • Dados pessoais sem necessidade e política definida.
  • Headers indiscriminadamente.

16 — Medindo tempo de resposta

Date.now() funciona para medidas simples. performance.now() é monotônico e apropriado para medir duração dentro do processo.

const startedAt = performance.now();
// ...
const durationMs = performance.now() - startedAt;

A duração observada pelo servidor não é exatamente a latência percebida pelo cliente: rede, proxy, TLS e download do corpo também contam do lado do cliente.

17 — Eventos finish e close

finish

Indica que a resposta foi entregue ao sistema subjacente para envio. É o evento comum para registrar status e duração final.

close

Pode ocorrer quando a conexão é encerrada antes da conclusão normal. É útil para observar cancelamentos ou desconexões.

let finished = false;

response.on("finish", () => {
  finished = true;
});

response.on("close", () => {
  if (!finished) {
    console.warn("Resposta encerrada antes de finish");
  }
});

Evite registrar o mesmo acesso duas vezes. Defina claramente qual evento gera o log principal e qual gera somente um sinal complementar.

18 — Contexto por requisição

Request ID, usuário autenticado, tenant e permissões são exemplos de contexto. res.locals funciona bem quando o dado é consumido no pipeline HTTP. Em sistemas maiores, AsyncLocalStorage pode propagar contexto sem passar o objeto manualmente, mas adiciona complexidade e não é necessário para aprender o modelo central.

Regra prática: comece com dependências explícitas. Adote contexto implícito somente quando o problema real justificar.

19 — Autenticação: quem é você?

Autenticação transforma uma credencial em uma identidade verificada. O middleware normalmente lê um header/cookie, valida a credencial e anexa o principal autenticado ao contexto.

type Role = "ADMIN" | "MEMBER";

type AuthUser = {
  id: string;
  role: Role;
};

export function authenticate(
  request: Request,
  response: Response,
  next: NextFunction,
): void {
  const authorization = request.header("authorization");

  if (!authorization?.startsWith("Bearer ")) {
    response.status(401).json({
      error: "UNAUTHENTICATED",
      message: "Credencial ausente.",
    });
    return;
  }

  // Simulação didática. Tokens reais serão estudados depois.
  response.locals.user = {
    id: "user-42",
    role: authorization === "Bearer admin-token" ? "ADMIN" : "MEMBER",
  };

  next();
}
Importante: este token fixo é somente uma simulação para estudar o pipeline. Não é uma estratégia de autenticação para produção.

20 — Autorização: você pode fazer isto?

Autorização ocorre depois da autenticação. Ela compara identidade, papel, permissão e relação com o recurso.

export function requireAdmin(
  request: Request,
  response: Response,
  next: NextFunction,
): void {
  const user = response.locals.user;

  if (!user) {
    response.status(401).json({ error: "UNAUTHENTICATED" });
    return;
  }

  if (user.role !== "ADMIN") {
    response.status(403).json({ error: "FORBIDDEN" });
    return;
  }

  next();
}

Papéis são uma forma simples de autorização. Regras reais podem considerar propriedade do recurso, organização, estado, ação e permissões granulares.

21 — Middleware factory

Uma factory recebe configuração e devolve um middleware. É ideal para políticas parametrizadas.

type Role = "ADMIN" | "MEMBER";

export function requireRole(...allowedRoles: Role[]) {
  return function roleMiddleware(
    request: Request,
    response: Response,
    next: NextFunction,
  ): void {
    const user = response.locals.user;

    if (!user) {
      response.status(401).json({ error: "UNAUTHENTICATED" });
      return;
    }

    if (!allowedRoles.includes(user.role)) {
      response.status(403).json({ error: "FORBIDDEN" });
      return;
    }

    next();
  };
}
router.get(
  "/audit-logs",
  authenticate,
  requireRole("ADMIN"),
  listAuditLogsController,
);

O nome deve revelar que requireRole(...) cria uma função para uso posterior.

22 — 401 versus 403

StatusSignificado práticoExemplo
401A requisição não possui autenticação válida.Token ausente, inválido ou expirado.
403A identidade foi reconhecida, mas não possui acesso.MEMBER tentando excluir outro usuário.

Não use 403 para esconder todo problema de autenticação por hábito. Também existem cenários em que retornar 404 evita revelar a existência de um recurso; essa decisão pertence ao modelo de ameaça e precisa ser consistente.

23 — Middlewares assíncronos no Express 5

No Express 5, uma função async que rejeita uma Promise ou lança erro encaminha a falha ao pipeline de erros.

export async function loadUser(
  request: Request,
  response: Response,
  next: NextFunction,
): Promise<void> {
  const user = await userRepository.findById(request.params.userId);

  if (!user) {
    response.status(404).json({ error: "USER_NOT_FOUND" });
    return;
  }

  response.locals.userResource = user;
  next();
}

Isso reduz wrappers repetitivos para erros assíncronos. Ainda é necessário decidir quais falhas devem virar resposta local e quais devem seguir para o error handler.

24 — next(error) e o pipeline de erros

next(error) pula os middlewares comuns restantes e procura um middleware de erro, identificado por quatro parâmetros.

import type { ErrorRequestHandler } from "express";

export const errorHandler: ErrorRequestHandler = (
  error,
  request,
  response,
  next,
) => {
  console.error({
    requestId: response.locals.requestId,
    error,
  });

  if (response.headersSent) {
    next(error);
    return;
  }

  response.status(500).json({
    error: "INTERNAL_SERVER_ERROR",
    requestId: response.locals.requestId,
  });
};

O tratamento detalhado de erros terá uma fase própria. Aqui importa entender que a cadeia de erros é separada e também depende da ordem de registro.

25 — Erros comuns e anti-patterns

25.1 — Não responder nem continuar

console.log("executou");
// faltou next() ou response...

25.2 — Chamar next() depois de responder

response.status(401).json({ error: "UNAUTHENTICATED" });
next(); // outra etapa pode tentar responder de novo

25.3 — Executar lógica depois de next() sem intenção

next();
response.setHeader("x-late-header", "value");

25.4 — Ordem invertida

app.use("/admin", requireRole("ADMIN"));
app.use(authenticate);

25.5 — Middleware “faz tudo”

Uma função que autentica, consulta recursos, valida body, executa regra de negócio e responde perde reutilização e torna o fluxo difícil de testar.

25.6 — Mutação sem contrato

Adicionar propriedades arbitrárias em request ou response.locals sem tipo e convenção faz etapas posteriores dependerem de conhecimento oculto.

26 — Edge cases em middlewares

Edge cases são situações nos limites do fluxo normal. Eles existem porque clientes interrompem conexões, headers podem faltar, middlewares podem responder cedo, credenciais expiram e múltiplas etapas podem disputar o controle.

26.1 — Continuação e resposta

  • Middleware chama next() duas vezes.
  • Responde e depois chama next().
  • Chama next(), continua executando e tenta alterar headers tarde demais.
  • Uma condição não contemplada deixa a requisição pendurada.

26.2 — Identidade e autorização

  • Header Authorization ausente, vazio, malformado ou repetido.
  • Token válido para usuário removido, bloqueado ou com papel alterado.
  • Usuário autenticado tenta acessar recurso de outro tenant.
  • ADMIN tenta remover a própria última conta administrativa.

26.3 — Logging e observabilidade

  • Request ID recebido é enorme ou contém caracteres inesperados.
  • Resposta fecha antes do evento finish.
  • Logger lança erro e interfere no endpoint.
  • Logs expõem token ou dado pessoal.
  • Stream longo distorce a interpretação da duração.

26.4 — Concorrência

  • Duas requisições simultâneas não podem compartilhar usuário ou request ID.
  • Variáveis globais mutáveis produzem vazamento entre requisições.
  • Listeners precisam ser adicionados uma vez por resposta, sem crescimento acidental.
Como usar edge cases: transforme cada item em teste mental ou automatizado. O objetivo não é criar paranoia, mas tornar explícito o que o sistema fará quando o caminho feliz não acontecer.

28 — Exemplo integrado

import express from "express";
import { randomUUID } from "node:crypto";
import type { NextFunction, Request, Response } from "express";

const app = express();

type Role = "ADMIN" | "MEMBER";
type User = { id: string; role: Role };
type Locals = { requestId: string; user?: User };
type AppResponse = Response<unknown, Locals>;

app.use(express.json());

app.use((request, response: AppResponse, next) => {
  const requestId = randomUUID();
  response.locals.requestId = requestId;
  response.setHeader("x-request-id", requestId);
  next();
});

app.use((request, response: AppResponse, next) => {
  const startedAt = performance.now();

  response.on("finish", () => {
    console.log({
      requestId: response.locals.requestId,
      method: request.method,
      url: request.originalUrl,
      status: response.statusCode,
      durationMs: performance.now() - startedAt,
    });
  });

  next();
});

function authenticate(
  request: Request,
  response: AppResponse,
  next: NextFunction,
): void {
  const token = request.header("authorization");

  if (!token) {
    response.status(401).json({ error: "UNAUTHENTICATED" });
    return;
  }

  response.locals.user = {
    id: token === "admin-token" ? "admin-1" : "member-1",
    role: token === "admin-token" ? "ADMIN" : "MEMBER",
  };

  next();
}

function requireRole(...roles: Role[]) {
  return (
    request: Request,
    response: AppResponse,
    next: NextFunction,
  ): void => {
    const user = response.locals.user;

    if (!user) {
      response.status(401).json({ error: "UNAUTHENTICATED" });
      return;
    }

    if (!roles.includes(user.role)) {
      response.status(403).json({ error: "FORBIDDEN" });
      return;
    }

    next();
  };
}

app.get(
  "/admin/dashboard",
  authenticate,
  requireRole("ADMIN"),
  (request, response: AppResponse) => {
    response.status(200).json({
      requestId: response.locals.requestId,
      user: response.locals.user,
      metrics: { activeUsers: 128 },
    });
  },
);
Limite didático: a autenticação é simulada. A finalidade do exemplo é visualizar ordem, escopo, locals e decisões de interrupção.

29 — Exercícios orientados

Exercício 1 — Trace a ordem

Receba uma aplicação com seis app.use() e duas rotas. Antes de executar, escreva quais funções rodam para GET /public, GET /admin e uma rota inexistente.

Exercício 2 — Corrija o pipeline

Encontre todos os caminhos de uma função condicional e marque se cada caminho responde, chama next() ou fica pendurado.

Exercício 3 — Request ID

Crie um middleware que gere UUID, grave em res.locals, devolva em x-request-id e faça o logger utilizá-lo.

Exercício 4 — Matriz de acesso

Implemente MEMBER e ADMIN. MEMBER lê o próprio perfil; ADMIN lista usuários e bloqueia contas. Decida quais regras pertencem ao router e quais pertencem à rota.

Exercício 5 — Eventos de resposta

Registre finish e close sem duplicar o log principal. Simule uma conexão encerrada.

Dicas progressivas: primeiro desenhe o pipeline em texto; depois identifique o dado produzido por cada etapa; por último escreva o código. Se a rota precisa “saber” como o token foi validado, as responsabilidades ainda estão misturadas.

30.5 — Regras de negócio

  • Usuário bloqueado não acessa endpoints privados.
  • MEMBER só altera o próprio perfil.
  • Somente ADMIN pode bloquear contas.

  • Toda alteração administrativa cria evento de auditoria com request ID e actor ID.
  • Logs nunca guardam token completo.

30.6 — Casos de teste mental

  1. Acessar /me sem credencial.
  2. MEMBER acessar /admin/users.
  3. ADMIN bloquear a própria conta.
  4. Token representa usuário que acabou de ser bloqueado.
  5. Controller lança erro assíncrono.
  6. Rota inexistente ainda recebe request ID e log.
  7. Logger registra status 204 sem body.
  8. Cliente encerra a conexão durante a resposta.

30.7 — Sequência recomendada

  1. Implemente request ID e confirme o header.
  2. Implemente logger e confirme status/duração no finish.
  3. Crie autenticação simulada e tipagem de locals.
  4. Implemente a factory requireRole.
  5. Separe router público, privado e administrativo.
  6. Crie auditoria usando identidade e request ID.
  7. Teste todos os caminhos que interrompem ou continuam.
  8. Adicione 404 e error handler no final.

31 — Checklist de domínio

  • Consigo explicar middleware como etapa do pipeline, não apenas como “função que roda antes”.
  • Sei que a ordem de registro altera o comportamento.
  • Consigo identificar todos os caminhos que respondem, continuam ou encaminham erro.
  • Sei que next() não encerra a função JavaScript.
  • Sei escolher middleware global, por router ou por rota.
  • Entendo a composição de vários handlers.
  • Sei usar res.locals para contexto específico da requisição.
  • Não uso variável global para armazenar usuário atual.
  • Consigo gerar e propagar request ID.
  • Registro logs no fim da resposta e evito dados sensíveis.
  • Conheço a diferença prática entre finish e close.
  • Separo autenticação de autorização.
  • Sei explicar 401 e 403.
  • Consigo criar uma middleware factory.
  • Entendo encaminhamento de rejeições assíncronas no Express 5.
  • Sei por que o error handler fica no final.

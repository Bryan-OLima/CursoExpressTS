Fase 5

Validação de dados com Zod

Zod 4Express 5TypeScriptRuntime validationAPI de eventos

Todo dado vindo de fora da aplicação deve ser tratado como não confiável: body, params, query, headers, cookies, arquivos, variáveis de ambiente e respostas de serviços externos. O TypeScript ajuda enquanto escrevemos o programa, mas não verifica o JSON que atravessa a rede. Zod ocupa essa fronteira.

Objetivo: transformar valores externos desconhecidos em dados validados e tipados; escolher entre parsing, coerção e transformação; compor schemas reutilizáveis; validar PATCH; produzir erros úteis; e integrar tudo ao pipeline do Express sem poluir controllers.
entrada externa (unknown) ↓ schema Zod ↓ parsing + validação ↓ coerção / normalização controlada ↓ dado confiável e tipado ↓ controller / service

1 — Modelo mental

1. A rede entrega valores em runtime, não tipos TypeScript.
2. O schema descreve e executa verificações reais.
3. O parsing pode falhar e precisa virar uma resposta previsível.
4. Quando funciona, usamos o valor retornado pelo Zod.
5. O controller recebe entrada normalizada e concentra-se no caso de uso.
Regra central: Request<..., CreateEventBody> não prova que o cliente enviou esse formato. É apenas uma promessa feita ao compilador.

2 — O que conta como dado externo

Entrada HTTP

  • request.body
  • request.params
  • request.query
  • headers, cookies e arquivos

Outras fronteiras

  • process.env
  • webhooks e filas
  • respostas de APIs
  • dados legados do banco

“Externo” também inclui frontends com bug, integrações desatualizadas e registros antigos. Não é apenas uma questão de ataque.

3 — TypeScript não valida em runtime

type CreateEventBody = { title: string; startsAt: string };
const body = request.body as CreateEventBody;
body.title.trim(); // pode quebrar se title for 42

Depois da compilação, o tipo desaparece. as não converte nem inspeciona. Mesmo que seu React valide o formulário, o backend precisa validar novamente: qualquer cliente pode chamar a API.

4 — O que é um schema

import * as z from "zod";

const createEventSchema = z.object({
  title: z.string().trim().min(3).max(120),
  startsAt: z.iso.datetime({ offset: true }),
  capacity: z.number().int().positive(),
});

O schema é uma especificação executável, um validador de runtime e uma fonte de inferência de tipos. O curso usa Zod 4.

5 — parse() e safeParse()

parse

const event = createEventSchema.parse(input);

Retorna o dado ou lança ZodError.

safeParse

const result = createEventSchema.safeParse(input);
if (!result.success) {
  console.log(result.error.issues);
} else {
  console.log(result.data);
}

Retorna uma união discriminada.

Use result.data: ele contém o valor validado e possivelmente transformado. Continuar usando o input bruto desperdiça o trabalho do schema.

6 — Schemas primitivos

z.string(); z.number(); z.boolean(); z.bigint();
z.date(); z.undefined(); z.null(); z.unknown();

z.number() não aceita "10". Converter texto em número é uma decisão explícita de coerção.

7 — Strings e formatos

const attendeeSchema = z.object({
  name: z.string().trim().min(2).max(80),
  email: z.email(),
  website: z.url().optional(),
  id: z.uuid(),
  eventDate: z.iso.date(),
});

z.string() aceita string vazia. Quando texto vazio não é válido, use .trim().min(1) ou um limite de domínio. Formato válido não significa recurso existente.

8 — Números e limites

const capacity = z.number().int().min(1).max(10_000);
const price = z.number().nonnegative().finite();

Considere inteiros, decimais, NaN, infinito e intervalos. Para dinheiro, geralmente prefira centavos inteiros ou um tipo decimal.

9 — Objetos e chaves extras

const createUserSchema = z.strictObject({
  name: z.string().trim().min(2),
  email: z.email(),
});
// { name, email, role: "ADMIN" } falha.

Em entradas sensíveis, rejeitar campos extras ajuda a detectar erros e tentativas de enviar propriedades controladas pelo servidor.

10 — Arrays

const tagsSchema = z
  .array(z.string().trim().min(1).max(30))
  .min(1)
  .max(10);

Valide os itens e a coleção: tamanho, duplicidade e ordem podem ser regras relevantes.

11 — Ausente, undefined e null

Schema Aceita
z.string() string
.optional() string ou ausência
.nullable() string ou null
.nullish() string, null ou ausência

Em PATCH, ausência pode significar “não altere”; null pode significar “limpe”.

12 — Enums e unions discriminadas

const visibility = z.enum(["PUBLIC", "PRIVATE", "UNLISTED"]);

const destination = z.discriminatedUnion("type", [
  z.object({ type: z.literal("ONLINE"), meetingUrl: z.url() }),
  z.object({ type: z.literal("PHYSICAL"), address: z.string().min(5) }),
]);

Uma propriedade discriminadora evita objetos ambíguos com muitos campos opcionais.

13 — Validando body

const result = createEventSchema.safeParse(request.body);
if (!result.success) {
  return response.status(422).json({
    error: { code: "INVALID_BODY", issues: result.error.issues },
  });
}
const body = result.data;

express.json() faz parsing do JSON; Zod valida o valor resultante. São responsabilidades diferentes.

14 — Validando params

const eventParamsSchema = z.object({ eventId: z.uuid() });
const params = eventParamsSchema.parse(request.params);

Params chegam como strings. UUID válido ainda pode apontar para um evento inexistente; isso é outra verificação.

15 — Query strings

const listEventsQuerySchema = z.object({
  search: z.string().trim().min(1).optional(),
  status: z.enum(["DRAFT", "PUBLISHED", "CANCELLED"]).optional(),
  page: z.coerce.number().int().min(1).default(1),
  limit: z.coerce.number().int().min(1).max(100).default(20),
});

Query é texto por natureza. Parâmetros repetidos podem virar arrays; teste conscientemente.

16 — Coerção

z.coerce.number().parse("42"); // 42
z.coerce.boolean().parse("false"); // true!
Armadilha: a coerção booleana usa a verdade do JavaScript. Qualquer string não vazia é true. Para query, aceite explicitamente "true" e "false". 
const boolQuery = z.enum(["true", "false"])
  .transform(value => value === "true");

17 — preprocess()

const optionalWebsite = z.preprocess(
  input => input === "" ? undefined : input,
  z.url().optional(),
);

A função recebe unknown antes de o schema interno validar. Útil, mas pode esconder contratos ruins se usado em excesso.

18 — Transformações

const slugSchema = z.string().trim().min(3).transform(value =>
  value.toLowerCase().replace(/[^a-z0-9]+/g, "-")
);

type SlugInput = z.input<typeof slugSchema>;
type SlugOutput = z.output<typeof slugSchema>;

Transformação pode alterar valor e tipo. Sanitização não substitui escaping contextual, queries parametrizadas ou autorização.

19 — z.infer

type CreateEventInput = z.infer<typeof createEventSchema>;

O schema vira fonte do tipo quando estrutura e validação representam o mesmo contrato. Um DTO HTTP não é necessariamente sua entidade de domínio.

20 — Composição

const baseEvent = z.object({
  id: z.uuid(), title: z.string().trim().min(3), createdAt: z.iso.datetime(),
});
const createEvent = baseEvent.omit({ id: true, createdAt: true });
const summary = baseEvent.pick({ id: true, title: true });
const withCapacity = createEvent.extend({ capacity: z.number().int().positive() });

Reutilize conceitos estáveis, não apenas para reduzir linhas. Derivação mecânica de schemas do banco pode vazar campos privados.

21 — PATCH e partial()

const updateEventSchema = z.object({
  title: z.string().trim().min(3),
  description: z.string().trim().max(2000).nullable(),
  startsAt: z.iso.datetime({ offset: true }),
}).partial().refine(
  body => Object.keys(body).length > 0,
  { error: "Envie ao menos um campo." },
);

Não permita campos como ownerId e createdAt. Um schema parcial puro costuma aceitar {}.

22 — refine()

const period = z.object({
  startsAt: z.iso.datetime({ offset: true }),
  endsAt: z.iso.datetime({ offset: true }),
}).refine(
  data => new Date(data.endsAt).getTime() > new Date(data.startsAt).getTime(),
  { path: ["endsAt"], error: "O término deve ocorrer depois do início." },
);

Use path para associar a issue ao campo mais útil para o cliente.

23 — superRefine()

schema.superRefine((data, ctx) => {
  if (data.minAge > data.maxAge) {
    ctx.addIssue({ code: "custom", path: ["maxAge"], message: "Intervalo inválido." });
  }
  if (data.waitlistCapacity > data.capacity * 2) {
    ctx.addIssue({ code: "custom", path: ["waitlistCapacity"], message: "Limite excedido." });
  }
});

É útil para relatar vários problemas na mesma execução.

24 — Validação assíncrona

const uniqueSlug = z.string().refine(
  async slug => !(await repository.existsBySlug(slug)),
  { error: "Slug já usado." },
);
const result = await uniqueSlug.safeParseAsync(input);

Use parseAsync ou safeParseAsync. Uma consulta prévia não garante unicidade contra concorrência; a restrição do banco continua necessária.

25 — Middleware reutilizável

import type { NextFunction, Request, Response } from "express";
import type { ZodType } from "zod";

export function validateBody(schema: ZodType) {
  return async (req: Request, res: Response, next: NextFunction) => {
    const result = await schema.safeParseAsync(req.body);
    if (!result.success) {
      res.status(422).json({
        error: {
          code: "VALIDATION_ERROR",
          issues: result.error.issues.map(issue => ({
            path: issue.path, code: issue.code, message: issue.message,
          })),
        },
      });
      return;
    }
    res.locals.validatedBody = result.data;
    next();
  };
}

26 — Onde guardar o dado validado

  • Substituir request.body: simples, mas muta o objeto.
  • Usar res.locals: separa bruto e validado.
  • Criar helper tipado de rota: melhor ergonomia, mais abstração. 
type Locals = { validatedBody: CreateEventInput };
response: Response<unknown, Locals>
const input = response.locals.validatedBody;

27 — ZodError e contrato público

for (const issue of result.error.issues) {
  console.log(issue.code, issue.path, issue.message);
}

Não exponha stack trace, input completo ou detalhes internos. Crie um contrato estável: 

{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Os dados enviados são inválidos.",
    "issues": [{ "path": ["title"], "code": "too_small", "message": "..." }]
  }
}

28 — 400 ou 422?

Ambos aparecem no mercado. Uma convenção coerente é: JSON malformado → 400; conteúdo compreendido, mas inválido → 422. Documente a escolha e nunca use 500 para erro do cliente.

29 — Erros comuns

Confiar no Request tipado

O compilador não inspeciona tráfego.

Usar o input bruto

Ignora normalizações.

Coagir tudo

Aceita entradas ambíguas.

Schema universal

Criação, PATCH e resposta diferem.

Banco em todo refine

Mistura validação e I/O.

Expor issues sem adaptar

Acopla a API à biblioteca.

30 — Edge cases

30.1 — Body ausente

Sem parser, content-type ou body, a entrada pode ser undefined.

30.2 — JSON malformado

Falha em express.json(), antes do Zod; trate como 400.

30.3 — Chaves extras

Escolha conscientemente entre rejeitar, remover ou preservar.

30.4 — Espaços

" " passa em string simples. Use trim antes de min.

30.5 — Datas e fuso

ISO válido não resolve horário de verão nem regra “no futuro”.

30.6 — Booleano na query

Boolean("false") é true.

30.7 — Número vazio

Teste string vazia, espaços, NaN, infinito e notação exponencial.

30.8 — Query repetida

?status=A&status=B pode virar array.

30.9 — PATCH vazio

{} pode passar em partial.

30.10 — optional versus null

Ausência e null podem ter semânticas diferentes.

30.11 — Concorrência

Refine assíncrono não substitui constraint unique.

30.12 — Payload grande

Configure limite no parser; Zod não é limitador de upload.

32 — Exemplo integrado

const createEventSchema = z.strictObject({
  title: z.string().trim().min(3).max(120),
  description: z.string().trim().max(2000).nullable().default(null),
  startsAt: z.iso.datetime({ offset: true }),
  endsAt: z.iso.datetime({ offset: true }),
  visibility: z.enum(["PUBLIC", "PRIVATE"]).default("PRIVATE"),
  capacity: z.number().int().min(1).max(10_000),
  tags: z.array(z.string().trim().min(1).max(30)).max(10).default([]),
}).refine(
  ({ startsAt, endsAt }) => new Date(endsAt).getTime() > new Date(startsAt).getTime(),
  { path: ["endsAt"], error: "O término deve ocorrer depois do início." },
);

app.post("/events", (request, response) => {
  const result = createEventSchema.safeParse(request.body);
  if (!result.success) {
    return response.status(422).json({
      error: { code: "VALIDATION_ERROR", issues: result.error.issues },
    });
  }
  const event = {
    id: crypto.randomUUID(),
    ...result.data,
    status: "DRAFT",
    createdAt: new Date().toISOString(),
  };
  return response.status(201).location(`/events/${event.id}`).json({ data: event });
});

O exemplo garante forma, limites, defaults e ordem das datas. Ainda não garante autorização, conflito de agenda, persistência nem que o evento está no futuro.

33 — Exercícios

  1. Schema de params com UUID e casos inválidos.
  2. Query com busca, status, page e limit entre 1 e 100.
  3. Parser explícito de booleanos, sem z.coerce.boolean().
  4. PATCH com campos permitidos e body não vazio.
  5. Localização como união online/presencial.
  6. Contrato próprio de issues com field, code e message.
  7. Casos de teste para null, espaços, chaves extras, query repetida e datas fora de ordem.

34 — Mini projeto: cadastro de eventos

POST   /events
GET    /events
GET    /events/:eventId
PATCH  /events/:eventId
POST   /events/:eventId/publish
POST   /events/:eventId/cancel

Validação

  • Título de 3 a 120 caracteres.
  • Descrição opcional ou nula até 2.000.
  • Datas ISO com offset e término posterior.
  • Capacidade inteira de 1 a 10.000.
  • Visibilidade controlada e até 10 tags sem vazias.
  • Params UUID, paginação e PATCH seguro.

Regras de negócio

  • Evento publicado não pode começar no passado.
  • Evento cancelado não pode ser publicado.
  • Alteração de horário de evento publicado deve ser tratada conscientemente.
  • Validação estrutural e regra de negócio devem permanecer separadas.

35 — Checklist

  • Explico por que TypeScript não valida JSON.
  • Diferencio parse e safeParse.
  • Uso o valor retornado pelo schema.
  • Diferencio optional, nullable e nullish.
  • Valido body, params e query.
  • Entendo riscos de coerção.
  • Modelo unions discriminadas.
  • Uso z.infer sem confundir DTO e entidade.
  • Crio PATCH parcial seguro.
  • Uso refine e superRefine.
  • Lido com validação assíncrona.
  • Formato erros em contrato próprio.
  • Separo validação estrutural e regra de negócio.

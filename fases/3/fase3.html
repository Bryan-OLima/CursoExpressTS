FASE 3 ● DESENHO DE APIs REST

Modele recursos, não comandos.

Nesta fase, você aprenderá a transformar regras de negócio em uma API HTTP coerente: URLs que representam recursos, métodos com semântica correta, status informativos e contratos que clientes conseguem consumir sem adivinhação.

00 · VISÃO DA FASE

O que você deverá saber ao terminar

REST não é decorar que GET busca e POST cria. O objetivo é desenvolver um modelo mental que permita justificar cada decisão pública da API.

  • Identificar recursos e desenhar URIs estáveis.
  • Escolher o método HTTP de acordo com a intenção e a semântica.
  • Distinguir parâmetros de rota, query string e corpo.
  • Projetar filtros, busca, ordenação e paginação.
  • Usar status HTTP de maneira consistente.
  • Modelar erros e regras de negócio previsíveis.
  • Reconhecer edge cases antes que apareçam em produção.
Ideia central: uma boa API reduz o número de decisões que cada consumidor precisa tomar. Ela é previsível porque problemas semelhantes recebem soluções semelhantes.
01 · FUNDAMENTO

O que REST realmente é

REST significa Representational State Transfer. É um estilo arquitetural descrito por Roy Fielding, não uma biblioteca, protocolo ou formato de arquivo. Uma API pode usar HTTP e JSON sem seguir bem as ideias REST.

Na prática deste curso, trabalharemos com APIs HTTP orientadas a recursos. O cliente manipula representações desses recursos por meio de métodos HTTP. A URL identifica o alvo; o método expressa a intenção.

Recurso

Algo relevante para o domínio: livro, autor, empréstimo, usuário, tarefa.

Representação

Uma forma de descrever o estado do recurso, normalmente JSON em nossa API.

Estado da aplicação

Informações mantidas pelo servidor: quais livros existem, quais estão emprestados.

Transferência

O cliente envia ou recebe representações para consultar ou alterar esse estado.

Importante: “RESTful” não significa seguir uma tabela de nomes mecanicamente. As restrições clássicas incluem cliente-servidor, ausência de estado de sessão no servidor, cache, interface uniforme, sistema em camadas e código sob demanda opcional. Nesta fase, o foco é a interface HTTP uniforme.

03 · DESIGN

Como nomear rotas

Prefira substantivos que representem o domínio. A ação já é comunicada pelo método HTTP. Rotas orientadas a comandos duplicam informação e criam convenções próprias que o consumidor precisa memorizar.

EvitePrefiraPor quê
POST /create-taskPOST /tasksPOST já comunica criação na coleção.
GET /get-all-tasksGET /tasksGET já comunica leitura.
POST /delete-taskDELETE /tasks/:taskIdO método e o identificador deixam a intenção explícita.
GET /book-by-id/:idGET /books/:bookIdA coleção e o segmento identificador já explicam a consulta.
  • Use substantivos, normalmente no plural: /books, /authors.
  • Mantenha um padrão de nomes: kebab-case ou outro, sem alternar.
  • Evite detalhes internos como /database-books.
  • Evite extensões de formato como /books.json.
  • Não coloque informações sensíveis na URL, pois URLs aparecem em logs e históricos.
Nem toda ação vira CRUD com elegância. Operações como cancelar, publicar ou renovar podem ser modeladas como mudança de estado ou como criação de um sub-recurso. O domínio deve orientar a escolha.
04 · SEMÂNTICA HTTP

GET, POST, PUT, PATCH e DELETE

MétodoIntenção típicaSeguro?Idempotente?
GETLer uma coleção ou recurso, sem solicitar alteração de estado.SimSim
POSTCriar recurso na coleção ou executar processamento definido pelo recurso.NãoNão garantido
PUTCriar ou substituir a representação no URI conhecido.NãoSim
PATCHAplicar alteração parcial.NãoNão garantido
DELETERemover a associação ou recurso identificado.NãoSim
import express from "express";

const app = express();
app.use(express.json());

app.get("/books", listBooks);
app.post("/books", createBook);
app.get("/books/:bookId", getBook);
app.put("/books/:bookId", replaceBook);
app.patch("/books/:bookId", updateBook);
app.delete("/books/:bookId", deleteBook);
GET não deve causar efeitos de negócio. Não use GET /loans/:id/return para devolver um livro. Crawlers, pré-carregamento e retries podem disparar GET sem intenção humana de alterar dados.
05 · CONFIABILIDADE

Idempotência e métodos seguros

Uma operação é idempotente quando repetir a mesma requisição produz o mesmo efeito pretendido no estado do servidor que executá-la uma vez. Isso não exige respostas byte a byte idênticas; timestamps, logs e códigos podem variar.

// Executar uma ou cinco vezes mantém o mesmo estado final:
PUT /books/42
{ "title": "Clean Code", "isbn": "9780132350884", "authorId": "7" }

// A primeira pode remover; as seguintes apenas constatam que já não existe:
DELETE /books/42

POST normalmente não é idempotente. Repetir uma criação após timeout pode criar dois empréstimos. Em operações críticas, é possível aceitar uma chave de idempotência e guardar o resultado da primeira execução.

POST /loans
Idempotency-Key: 9b27d179-9f8a-4a31-a22e-17df1d96b3b1

{
  "bookId": "book-42",
  "memberId": "member-9"
}
Seguro não significa protegido. Em HTTP, método seguro é aquele cuja semântica solicitada é somente leitura, como GET. Autenticação e segurança contra ataques são outro assunto.
06 · ATUALIZAÇÃO

PUT versus PATCH

PUT · substituição

O cliente envia uma representação completa. Campos ausentes podem ser removidos ou voltar ao padrão, conforme o contrato.

PATCH · alteração parcial

O cliente envia apenas as mudanças. Campos ausentes permanecem como estavam.

// Estado atual
{
  "id": "book-1",
  "title": "O Hobbit",
  "isbn": "9780007525492",
  "authorId": "author-1",
  "pages": 310
}

// PUT: representação completa esperada
PUT /books/book-1
{
  "title": "O Hobbit — Edição Ilustrada",
  "isbn": "9780007525492",
  "authorId": "author-1",
  "pages": 320
}

// PATCH: somente a mudança
PATCH /books/book-1
{ "pages": 320 }
PATCH vazio: decida o contrato. Você pode tratá-lo como requisição válida sem mudança, mas em APIs de negócio geralmente é mais útil retornar 400 ou 422 informando que ao menos um campo atualizável é necessário.

08 · CONSULTAS

Filtros, busca e ordenação

Uma URI de coleção deve aceitar critérios opcionais de consulta sem criar uma rota nova para cada combinação.

GET /books?authorId=author-7&available=true
GET /books?search=clarice
GET /books?sort=title&order=asc
GET /books?publishedFrom=2000&publishedTo=2020
RecursoQueryInterpretação
Filtro exatoavailable=trueRestringe por um valor conhecido.
Busca textualsearch=senhorProcura termo em campos definidos no contrato.
Ordenaçãosort=title&order=ascDefine campo e direção.
IntervalopublishedFrom=2000Define limite inferior ou superior.
Defina uma ordenação padrão. Sem ordem estável, paginação pode repetir ou omitir itens quando o banco devolve linhas em ordem não garantida. Use um desempate determinístico, como title ASC, id ASC.
09 · ESCALA

Paginação por página e por cursor

Offset / página

GET /books?page=2&limit=20

{
  "data": [/* 20 livros */],
  "pagination": {
    "page": 2,
    "limit": 20,
    "totalItems": 87,
    "totalPages": 5,
    "hasNextPage": true,
    "hasPreviousPage": true
  }
}

É simples para interfaces com páginas numeradas, mas offsets altos podem custar caro e alterações entre requisições podem deslocar resultados.

Cursor

GET /books?limit=20&after=eyJpZCI6ImJvb2stMjAifQ

{
  "data": [/* 20 livros */],
  "pageInfo": {
    "nextCursor": "eyJpZCI6ImJvb2stNDAifQ",
    "hasNextPage": true
  }
}

É melhor para feeds e grandes conjuntos em mudança, mas não permite saltar naturalmente para uma página arbitrária.

Validações mínimas: página inteira maior ou igual a 1; limite inteiro positivo; limite máximo; cursor bem-formado; campos de ordenação presentes em uma allowlist.
10 · RESPOSTAS

Status HTTP que você precisa dominar

StatusUse quandoExemplo na biblioteca
200 OKA operação foi concluída e há representação na resposta.Livro encontrado ou atualizado.
201 CreatedUm novo recurso foi criado.Novo autor, livro ou empréstimo.
204 No ContentSucesso sem corpo de resposta.Livro removido ou empréstimo devolvido sem representação.
400 Bad RequestA requisição é malformada ou parâmetros básicos são inválidos.page=abc ou JSON inválido.
401 UnauthorizedCredenciais ausentes ou inválidas.Token não enviado ou expirado.
403 ForbiddenIdentidade conhecida, mas sem permissão.Membro tenta cadastrar livros, ação exclusiva de bibliotecário.
404 Not FoundO recurso alvo não existe ou não deve ser revelado.bookId inexistente.
409 ConflictConflito com o estado atual ou restrição de unicidade.ISBN já cadastrado; livro já emprestado.

Não transforme toda falha em 500. ISBN duplicado é uma situação conhecida do domínio, não uma pane inesperada. O status deve ajudar o cliente a decidir o que fazer.
11 · CONTRATO

Erros previsíveis e legíveis por máquina

Respostas de erro devem manter a mesma estrutura em todos os endpoints. Um texto solto obriga o cliente a interpretar mensagens humanas, que podem mudar.

HTTP/1.1 409 Conflict
Content-Type: application/problem+json

{
  "type": "https://api.example.com/problems/isbn-already-exists",
  "title": "ISBN já cadastrado",
  "status": 409,
  "detail": "Já existe um livro com o ISBN 9780132350884.",
  "instance": "/books",
  "code": "ISBN_ALREADY_EXISTS",
  "fields": {
    "isbn": ["O ISBN precisa ser único."]
  }
}

A RFC 9457 define o formato “Problem Details for HTTP APIs”. Campos de extensão, como code e fields, podem complementar o formato quando documentados.

  • Um código estável para decisões do frontend.
  • Uma mensagem humana sem vazar detalhes internos.
  • Erros por campo para formulários.
  • Um identificador de correlação para suporte e logs, quando necessário.
  • Nunca retornar stack trace em produção.
12 · RELAÇÕES

Rotas aninhadas

Rotas aninhadas expressam pertencimento ou escopo. Use-as quando a relação é importante para identificar ou consultar o recurso.

GET  /authors/:authorId/books
POST /authors/:authorId/books
GET  /members/:memberId/loans
GET  /books/:bookId/loans

Evite profundidade excessiva:

// difícil de ler, manter e reutilizar
GET /libraries/:libraryId/sections/:sectionId/shelves/:shelfId/books/:bookId/loans

// recurso diretamente identificável
GET /loans/:loanId
Regra prática: um ou dois níveis de aninhamento costumam ser suficientes. Se o sub-recurso possui identidade global, ofereça também uma URI direta.
13 · EVOLUÇÃO

Versionamento sem quebrar consumidores

Versionamento deve ser consequência de mudanças incompatíveis, não um ritual automático. Adicionar um campo opcional geralmente é compatível; remover ou renomear campos pode quebrar clientes.

Versão na URI

/v1/books. Explícita, simples de rotear e observar. É comum em APIs públicas.

Versão por media type

Accept: application/vnd.library.v1+json. Mantém a URI, mas aumenta a complexidade operacional.

GET /v1/books
GET /v2/books
  • Documente o que conta como mudança incompatível.
  • Comunique depreciação e prazo de migração.
  • Evite manter versões eternamente.
  • Não crie v2 apenas para corrigir um bug interno sem alterar contrato.
14 · CONSISTÊNCIA

Contratos previsíveis

Consistência vale mais que preferências isoladas. Escolha um padrão e aplique-o em toda a API.

Envelope de coleção

{ data, pagination } facilita metadados e evolução.

Datas

Use um formato consistente, normalmente ISO 8601 em UTC.

Nulos e ausentes

Defina quando um campo aparece como null e quando é omitido.

Identificadores

Mantenha tipo e nome estáveis: bookId, UUID/string, sem alternar.

type BookResponse = {
  id: string;
  title: string;
  isbn: string;
  authorId: string;
  available: boolean;
  createdAt: string;
  updatedAt: string;
};

type PaginatedResponse<T> = {
  data: T[];
  pagination: {
    page: number;
    limit: number;
    totalItems: number;
    totalPages: number;
  };
};
Não exponha diretamente o modelo do banco. A resposta HTTP é um contrato público. Colunas internas, nomes legados e relações do ORM não devem determinar automaticamente a representação da API.

16 · DOMÍNIO

Regras de negócio e respostas

RegraVerificaçãoResposta provável
Livro indisponível não pode ser emprestado.Antes de criar empréstimo, consultar disponibilidade de forma atômica.409 BOOK_UNAVAILABLE
Empréstimo devolvido não pode ser devolvido novamente.Estado atual precisa ser ACTIVE.409 LOAN_ALREADY_RETURNED
Autor com livros não pode ser removido sem estratégia.Verificar dependências ou política de cascata.409 AUTHOR_HAS_BOOKS
ISBN não pode se repetir.Validação de aplicação + índice único no banco.409 ISBN_ALREADY_EXISTS
Página e limite precisam ser válidos.Inteiros em intervalos definidos.400 INVALID_PAGINATION
Concorrência: verificar “available = true” e depois inserir o empréstimo em passos independentes não basta. Duas requisições podem passar na verificação. A regra precisa de transação, atualização condicional, lock ou restrição equivalente no banco.

Modelagem da devolução

// Opção 1: transição de estado parcial
PATCH /loans/:loanId
{ "status": "RETURNED" }

// Opção 2: criação de um evento/sub-recurso de devolução
POST /loans/:loanId/returns
{ "returnedAt": "2026-07-10T20:15:00.000Z" }

A segunda opção pode representar melhor o domínio quando a devolução possui dados próprios, auditoria ou processamento específico.

17 · ROBUSTEZ

Edge cases: onde APIs “certas” quebram

Edge case é uma situação nos limites das hipóteses normais: repetição, concorrência, valores extremos, estados intermediários ou combinações raras. Eles existem porque rede, usuários e dados reais não obedecem ao fluxo feliz.

Dois usuários tentam pegar o último exemplar
Ambos consultam disponibilidade antes de qualquer gravação. Sem controle transacional, dois empréstimos podem ser criados. O teste deve executar requisições concorrentes e verificar que apenas uma vence.
Cliente recebe timeout depois que o POST foi concluído
O cliente não sabe se o empréstimo foi criado e repete a requisição. Sem chave de idempotência ou outra estratégia, pode ocorrer duplicidade.
Livro é removido enquanto há empréstimo ativo
A política precisa ser explícita: bloquear, fazer soft delete, preservar snapshot ou permitir com histórico. A restrição não deve depender apenas da interface.
Query contém valores duplicados
?sort=title&sort=isbn pode chegar como array ou ter comportamento dependente do parser. O contrato deve definir se múltiplos valores são aceitos.
Limite excessivamente grande
limit=1000000 pode consumir memória e sobrecarregar banco. Defina teto e aplique-o na validação.
Ordenação por campo não permitido
Nunca interpolar o valor arbitrariamente em SQL. Use allowlist que traduza nomes públicos para colunas conhecidas.
DELETE repetido
A operação continua idempotente mesmo se a segunda resposta for 404. Idempotência trata do efeito no estado, não da obrigação de retornar o mesmo status.
PATCH com null
Ausência de campo e valor null podem ter significados diferentes. Por exemplo, omitir publishedAt mantém o valor; enviar null pode removê-lo.
Data válida no formato, impossível no domínio
Uma data pode ser ISO válida, mas futura quando o domínio exige passado. Validação estrutural e validação de negócio são camadas distintas.
Recurso existe, mas usuário não pode saber
Em alguns cenários, retornar 404 em vez de 403 reduz vazamento de informação. Essa política deve ser consistente e documentada internamente.
Para que servem: discutir edge cases cedo transforma incidentes imprevisíveis em decisões testáveis. Cada caso importante deve resultar em regra, restrição, teste ou observabilidade.
18 · PRÁTICA

Exercícios orientados

Reescreva rotas orientadas a comandos

Transforme /create-author, /list-available-books, /return-loan e /remove-book em endpoints orientados a recursos. Justifique cada método.

Defina contratos completos

Escreva request, response, status de sucesso e erros de POST /books e POST /loans.

Implemente consultas de coleção

Adicione busca por título, filtro por autor e disponibilidade, ordenação permitida e paginação validada.

Implemente PUT e PATCH

Crie testes demonstrando que PUT trata representação completa e PATCH preserva campos omitidos.

Modele erros consistentes

Use um único formato para validação, recurso inexistente, conflito e erro inesperado.

Teste idempotência

Repita PUT e DELETE e demonstre o estado final. Depois repita POST e identifique o risco.

Teste concorrência de empréstimo

Dispare duas tentativas para o mesmo livro e garanta que apenas uma seja criada.

Documente decisões

Crie um pequeno guia da API explicando nomes, envelopes, datas, nulos, paginação e política de versionamento.

Critério de conclusão: você deve conseguir explicar “por quê” para cada URI, método, status e estrutura de resposta, e não apenas fazer o endpoint funcionar.
19 · AUTOAVALIAÇÃO

Checklist de domínio da Fase 3

  • Consigo diferenciar recurso, URI, rota e endpoint.
  • Consigo explicar por que verbos na URL geralmente são redundantes.
  • Sei quais métodos são seguros e quais são idempotentes por definição.
  • Sei justificar PUT ou PATCH em um caso concreto.
  • Sei escolher entre route param, query e body.
  • Sei projetar paginação estável e validar seus limites.
  • Não confundo 401 com 403 nem 400 com 409.
  • Sei usar 201, 204 e Location corretamente.
  • Consigo modelar erro sem expor stack trace.
  • Entendo quando uma rota aninhada ajuda e quando vira excesso.
  • Consigo apontar mudanças compatíveis e incompatíveis.
  • Reconheço que regras críticas precisam de garantias no banco.
  • Consigo listar edge cases do mini projeto antes de implementá-lo.

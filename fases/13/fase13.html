Fase 13

Recursos avançados de backend

Express 5TypeScriptStreamsFilasTempo realResiliência

Esta fase reúne recursos que deixam de parecer “avançados” quando vistos como combinações de fundamentos já estudados: HTTP, camadas, banco, erros, testes, segurança e produção. O objetivo não é decorar bibliotecas, mas compreender modelos de execução, contratos e falhas.

Objetivo: implementar uploads e downloads seguros; transferir dados com streams; usar cache e negociação HTTP; receber webhooks sem duplicar efeitos; desacoplar tarefas com filas; enviar e-mails de modo confiável; aplicar timeout, retry e circuit breaker; e escolher entre SSE e WebSocket.
requisição ou evento ↓ validação + autenticação + limites ↓ trabalho síncrono mínimo ↓ persistência / fila / stream ↓ resposta previsível ↓ processamento, entrega ou atualização em tempo real

1 — Modelo mental

Entrada: arquivo, requisição, evento externo ou conexão persistente.
Contrato: formato, tamanho, identidade, ordem e semântica.
Efeito: persistir, publicar, transferir, notificar ou agendar.
Falha: timeout, duplicidade, desconexão, indisponibilidade ou dado inválido.
Recuperação: retry controlado, idempotência, fila de erro, compensação ou retomada.
Princípio central: “assíncrono” não significa “sem falhas”. Significa que a resposta HTTP e a conclusão do trabalho deixam de ocorrer necessariamente no mesmo momento.

2 — Trabalho síncrono e assíncrono

Síncrono

O cliente espera o trabalho terminar antes da resposta.

  • Validação curta.
  • Consulta necessária para responder.
  • Operação rápida e previsível.

Assíncrono

A API confirma aceitação e o trabalho continua em outro fluxo.

  • Envio de e-mail.
  • Processamento de imagem.
  • Geração de relatório.
  • Notificações em lote.
POST /reports
→ 202 Accepted
{
  "data": {
    "jobId": "job_123",
    "status": "QUEUED"
  }
}

3 — Custos, limites e backpressure

Recursos avançados frequentemente movem muitos bytes ou consomem serviços lentos. O backend precisa limitar tamanho, duração, concorrência e quantidade de tentativas. Sem limites, uma funcionalidade correta em desenvolvimento pode esgotar memória, CPU, conexões ou orçamento em produção.

Backpressure: ocorre quando quem produz dados é mais rápido do que quem consome. Streams e filas existem, entre outros motivos, para controlar essa diferença.

5 — Uploads

Uploads normalmente chegam como multipart/form-data. Diferente de JSON, esse formato transporta campos e arquivos. Em Express, uma biblioteca de multipart interpreta o corpo e expõe metadados ou streams.

app.post(
  "/avatars",
  upload.single("avatar"),
  async (request, response) => {
    if (!request.file) {
      return response.status(422).json({
        error: { code: "FILE_REQUIRED", message: "Envie um avatar." },
      });
    }

    const avatar = await avatarService.save({
      userId: response.locals.user.id,
      file: request.file,
    });

    return response.status(201).json({ data: avatar });
  },
);

Guardar o arquivo no disco local pode funcionar em desenvolvimento, mas costuma falhar em ambientes com múltiplas instâncias ou filesystem efêmero. Em produção, é comum usar armazenamento de objetos.

6 — Validação de arquivos

  • Limite de tamanho antes de consumir memória excessiva.
  • Quantidade máxima de arquivos.
  • Extensões permitidas apenas como sinal auxiliar.
  • Content-Type não deve ser aceito cegamente.
  • Inspeção dos bytes iniciais quando necessário.
  • Nome gerado pelo servidor, sem confiar no nome original.
  • Diretório e permissões controlados.
  • Varredura ou processamento isolado para tipos de maior risco.
Não confie em image/png enviado pelo cliente: metadados podem mentir. Validação precisa considerar conteúdo, destino e risco.

7 — Streams

Uma stream permite processar dados em partes, sem carregar tudo na memória. Em Node.js, arquivos, requisições e respostas podem ser streams. Isso é essencial para arquivos grandes.

import { createReadStream } from "node:fs";
import { pipeline } from "node:stream/promises";

app.get("/exports/:fileName", async (request, response) => {
  response.type("text/csv");
  response.attachment(request.params.fileName);

  await pipeline(
    createReadStream(`/safe/exports/${request.params.fileName}`),
    response,
  );
});

pipeline() coordena propagação de erros e encerramento das streams. Ainda é necessário validar o caminho para impedir path traversal.

8 — Downloads

Downloads devem definir tipo, nome, autorização e política de cache. O backend também precisa decidir se o conteúdo é público, privado, temporário ou gerado sob demanda.

response.setHeader(
  "Content-Disposition",
  'attachment; filename="relatorio.csv"',
);
response.type("text/csv");
Nunca forme um caminho de arquivo concatenando diretamente um parâmetro da URL. ../../ pode escapar do diretório esperado.

9 — Cache HTTP

Cache HTTP reduz transferência e processamento, mas exige entender frescor e revalidação. O servidor pode informar por quanto tempo uma resposta pode ser reutilizada e se caches compartilhados podem armazená-la.

response.setHeader(
  "Cache-Control",
  "private, max-age=60, must-revalidate",
);

10 — ETag e requisições condicionais

ETag identifica uma versão da representação. O cliente pode enviar If-None-Match; se nada mudou, o servidor responde 304 Not Modified sem reenviar o corpo.

const etag = `"user-${user.id}-${user.updatedAt.getTime()}"`;
response.setHeader("ETag", etag);

if (request.headers["if-none-match"] === etag) {
  return response.status(304).end();
}

return response.json({ data: user });

Express já possui suporte a ETag em respostas, mas criar um valor manual ajuda a entender a semântica. ETag também pode participar de controle otimista com If-Match.

11 — Compressão

Compressão reduz bytes transferidos, especialmente em JSON, HTML, CSS e texto. Ela consome CPU e não ajuda igualmente em formatos já comprimidos, como JPEG, PNG, ZIP e muitos vídeos.

Regra prática: comprima respostas textuais acima de um limite razoável; não tente comprimir tudo indiscriminadamente.

12 — Content negotiation

Negociação de conteúdo permite escolher uma representação com base em headers como Accept, Accept-Language e Accept-Encoding.

app.get("/reports/:reportId", async (request, response) => {
  const format = request.accepts(["json", "text/csv"]);

  if (!format) {
    return response.status(406).json({
      error: { code: "NOT_ACCEPTABLE", message: "Formato não suportado." },
    });
  }

  const report = await reportService.get(request.params.reportId);
  return format === "json"
    ? response.json({ data: report })
    : response.type("text/csv").send(toCsv(report));
});

13 — Webhooks

Webhook é uma requisição HTTP enviada por outro sistema quando um evento acontece. O receptor precisa assumir que a entrega pode atrasar, repetir, chegar fora de ordem ou nunca chegar.

app.post("/webhooks/payments", rawBodyMiddleware, async (request, response) => {
  webhookVerifier.verify({
    rawBody: request.body,
    signature: request.header("x-provider-signature"),
  });

  await webhookService.accept(request.body);
  return response.status(204).end();
});

Alguns provedores calculam a assinatura sobre o corpo bruto. Fazer parsing e reserialização antes da verificação pode alterar os bytes e invalidar a assinatura.

14 — Assinatura, autenticidade e replay

  • Use segredo separado para webhooks.
  • Compare assinaturas de modo seguro.
  • Inclua timestamp na verificação quando o provedor oferecer.
  • Rejeite eventos muito antigos quando apropriado.
  • Registre o identificador externo do evento.
  • Não use apenas IP de origem como autenticação principal.

15 — Idempotency keys

Uma idempotency key permite repetir uma requisição sem repetir seu efeito. É útil em pagamentos, criação de pedidos e outras operações sensíveis a duplicidade.

const key = request.header("idempotency-key");
if (!key) throw new ValidationError("Idempotency-Key é obrigatório.");

const previous = await idempotencyRepository.find(key);
if (previous) {
  return response.status(previous.statusCode).json(previous.body);
}

const result = await orderService.create(input);
await idempotencyRepository.save(key, {
  statusCode: 201,
  body: { data: result },
});
return response.status(201).json({ data: result });

A chave deve ser associada ao usuário, endpoint e fingerprint do payload. Reutilizar a mesma chave com outro corpo deve gerar conflito.

16 — Timeouts

Timeout limita quanto tempo uma operação pode ocupar recursos. Ele deve existir em chamadas externas, conexões, filas e trabalhos internos potencialmente lentos.

const response = await fetch(url, {
  signal: AbortSignal.timeout(5_000),
});
Timeout não desfaz automaticamente o efeito remoto. Uma requisição pode expirar no cliente e ainda ser concluída pelo serviço externo. Idempotência continua necessária.

17 — Retry

Retry é adequado para falhas transitórias, não para qualquer erro. Deve ter limite, atraso crescente, jitter e consciência de idempotência.

async function retry<T>(
  operation: () => Promise<T>,
  attempts = 3,
): Promise<T> {
  let lastError: unknown;

  for (let attempt = 1; attempt <= attempts; attempt += 1) {
    try {
      return await operation();
    } catch (error) {
      lastError = error;
      if (attempt === attempts) break;
      await new Promise(resolve => setTimeout(resolve, 200 * 2 ** (attempt - 1)));
    }
  }

  throw lastError;
}

Na prática, filtre erros retryable e adicione jitter para evitar que várias instâncias repitam ao mesmo tempo.

18 — Circuit breaker em nível conceitual

Closed

Chamadas passam normalmente. Falhas são contadas.

Open

Chamadas falham rapidamente para proteger o sistema.

Half-open

Algumas chamadas de teste verificam se o serviço se recuperou.

Fallback

Resposta degradada, cache ou enfileiramento, quando fizer sentido.

Circuit breaker não substitui timeout. Ele reage a um padrão de falhas; timeout limita cada tentativa.

19 — Filas

Uma fila desacopla produção e consumo de trabalho. A API publica uma mensagem; um worker processa depois. Isso melhora latência e absorve picos, mas introduz entrega repetida, atraso e consistência eventual.

await queue.add("send-welcome-email", {
  userId: user.id,
  email: user.email,
});

return response.status(201).json({ data: user });
Considere entrega “pelo menos uma vez”: o mesmo job pode ser processado novamente. O consumidor precisa ser idempotente.

20 — Workers, tentativas e dead-letter

Worker é o processo que consome jobs. Ele precisa registrar início, sucesso, falha, duração e número de tentativas. Mensagens que sempre falham não podem bloquear a fila indefinidamente.

  • Retry limitado com backoff.
  • Dead-letter queue ou estado FAILED.
  • Timeout do job.
  • Concorrência configurável.
  • Idempotência por job ou efeito.
  • Observabilidade e ferramenta de inspeção.

21 — Tarefas agendadas

Tarefas agendadas executam em horários ou intervalos. Exemplos: expirar reservas, consolidar métricas e emitir lembretes. Em múltiplas instâncias, todas podem tentar executar o mesmo cron.

Problema clássico: três réplicas da API executam o mesmo cron e enviam três e-mails. Use scheduler centralizado, lock distribuído, fila com unicidade ou desenho idempotente.

22 — E-mails

Envio de e-mail é integração externa e não deve, em geral, segurar a resposta principal. Modele template, destinatário, idioma, deduplicação, status e falhas.

await emailQueue.add("password-reset", {
  userId,
  resetTokenId,
  locale: "pt-BR",
});

Evite colocar tokens sensíveis completos em logs ou payloads duráveis sem necessidade. Links de recuperação devem expirar e ser de uso único.

23 — Consistência entre banco e fila

Salvar no banco e publicar na fila são dois efeitos separados. Se o banco confirmar e a publicação falhar, o estado pode ficar inconsistente. Um padrão importante é o transactional outbox.

await prisma.$transaction(async tx => {
  const order = await tx.order.create({ data: orderData });
  await tx.outboxEvent.create({
    data: {
      type: "ORDER_CREATED",
      aggregateId: order.id,
      payload: JSON.stringify({ orderId: order.id }),
    },
  });
});

Outro processo publica eventos pendentes e marca o outbox como entregue. Assim, estado e intenção de publicar são persistidos atomicamente.

24 — Server-Sent Events

SSE mantém uma resposta HTTP aberta para o servidor enviar eventos ao cliente. É unidirecional, simples e adequado a progresso, notificações e atualizações de status.

app.get("/jobs/:jobId/events", async (request, response) => {
  response.setHeader("Content-Type", "text/event-stream");
  response.setHeader("Cache-Control", "no-cache");
  response.setHeader("Connection", "keep-alive");
  response.flushHeaders();

  const send = (event: string, data: unknown) => {
    response.write(`event: ${event}\n`);
    response.write(`data: ${JSON.stringify(data)}\n\n`);
  };

  send("connected", { jobId: request.params.jobId });

  const unsubscribe = jobEvents.subscribe(request.params.jobId, send);
  request.on("close", unsubscribe);
});

O cliente pode reconectar automaticamente. Para recuperação, considere IDs de evento e Last-Event-ID.

25 — WebSocket

WebSocket estabelece canal bidirecional persistente. É adequado quando cliente e servidor precisam trocar mensagens frequentes em baixa latência, como chat e colaboração em tempo real.

  • Autenticação no handshake ou primeira mensagem.
  • Autorização por sala ou recurso.
  • Heartbeat para detectar conexões mortas.
  • Limite de tamanho e frequência de mensagens.
  • Validação de todo payload.
  • Reconexão e deduplicação no cliente.

Escolha a opção mais simples que atende o comportamento. “Tempo real” não obriga WebSocket.

27 — Escala e estado de conexão

Com múltiplas instâncias, um cliente pode estar conectado a uma instância enquanto o evento nasce em outra. É necessário um barramento compartilhado, pub/sub ou infraestrutura gerenciada. Também pode ser necessário sticky session, dependendo da biblioteca e do transporte.

Persistência continua importante: WebSocket entrega atualização, mas histórico e estado durável pertencem ao banco. Conexão em tempo real não deve virar a única fonte da verdade.

28 — Erros comuns

  1. Ler arquivos enormes inteiros na memória.
  2. Confiar apenas em extensão ou MIME declarado.
  3. Responder ao webhook antes de persistir sua aceitação sem estratégia.
  4. Executar retry infinito.
  5. Repetir POST não idempotente automaticamente.
  6. Enviar e-mail dentro da transação principal.
  7. Tratar fila como garantia de “exatamente uma vez”.
  8. Executar cron em cada réplica sem coordenação.
  9. Usar WebSocket para algo resolvido por polling simples.
  10. Não remover listeners quando a conexão fecha.
  11. Guardar estado importante apenas na memória do processo.
  12. Ocultar falhas assíncronas sem observabilidade.

29 — Edge cases

Upload interrompido: remova arquivo parcial e reverta metadados incompletos.
Mesmo webhook duas vezes: use identificador externo e constraint unique.
Webhook fora de ordem: compare versão, timestamp ou consulte o estado atual no provedor.
Timeout com efeito concluído: consulte por idempotency key antes de repetir.
Job executado duas vezes: efeito deve ser idempotente.
Mensagem venenosa: limite tentativas e mova para análise.
Worker cai após o efeito, antes do ACK: o job pode reaparecer.
Cron no horário de verão: defina timezone e semântica esperada.
SSE reconecta após perder eventos: use ID e retomada quando necessário.
WebSocket desconecta durante envio: confirme persistência e sequência, não apenas emissão.
Cache de resposta privada: configure private ou no-store.
ETag com representação diferente: inclua versão da representação e variáveis relevantes.

31 — Exemplo integrado: relatório assíncrono

// Controller: aceita o pedido rapidamente
app.post("/reports", async (request, response) => {
  const input = createReportSchema.parse(request.body);
  const job = await reportService.request({
    userId: response.locals.user.id,
    ...input,
  });

  await reportQueue.add("generate-report", { reportId: job.id });

  return response
    .status(202)
    .location(`/reports/${job.id}`)
    .json({ data: job });
});

// Worker: executa o trabalho pesado
reportWorker.process(async ({ data }) => {
  const report = await reportRepository.findById(data.reportId);
  if (!report || report.status === "READY") return;

  await reportRepository.markProcessing(report.id);

  try {
    const stream = reportGenerator.createCsvStream(report.filters);
    const file = await storage.uploadStream(stream);
    await reportRepository.markReady(report.id, file);
    await reportEvents.publish(report.id, { status: "READY" });
  } catch (error) {
    await reportRepository.markFailed(report.id);
    throw error;
  }
});

O exemplo combina 202, fila, worker idempotente, stream, armazenamento, estado persistido e atualização em tempo real.

34 — Mini projeto: central de processamento de mídia

Construa uma API em que usuários enviam imagens, acompanham o processamento e baixam versões transformadas.

34.1 — Fluxo

  1. POST /media recebe upload e cria registro UPLOADED.
  2. A API publica um job de processamento.
  3. O worker valida conteúdo, gera thumbnail e versão otimizada.
  4. O progresso é persistido e publicado via SSE.
  5. GET /media/:id informa estado.
  6. GET /media/:id/download entrega arquivo autorizado por stream.
  7. Um webhook opcional avisa outro sistema quando o item fica pronto.

34.2 — Regras

  • Usuário só acessa seus arquivos.
  • Arquivo possui limite de tamanho e dimensões.
  • Mesmo job pode rodar duas vezes sem duplicar versões.
  • Falha após número máximo de tentativas marca FAILED.
  • Download usa nome seguro e tipo correto.
  • Webhook de saída possui assinatura, timeout e retry limitado.
  • Eventos SSE não vazam mídia de outro usuário.
  • Metadados e intenção de processamento são persistidos atomicamente.

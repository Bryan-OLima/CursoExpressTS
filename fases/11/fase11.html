Fase 11

Segurança de APIs

Express 5TypeScriptDefesa em profundidadeOWASPAuditoria prática

Segurança não é um recurso isolado que “ligamos” no fim do projeto. Ela aparece no desenho das rotas, na validação, na autenticação, no banco, nos logs, no deploy e até nas permissões do processo que executa a aplicação.

Objetivo: reconhecer ameaças comuns, reduzir a superfície de ataque, configurar proteções HTTP, controlar abuso, evitar exposição de dados, tratar uploads com desconfiança e auditar uma API deliberadamente vulnerável.
cliente não confiável ↓ proxy / TLS / limites de conexão ↓ CORS + headers defensivos ↓ rate limit + autenticação ↓ parsing com limite + validação ↓ autorização + regras de negócio ↓ queries parametrizadas + menor privilégio ↓ resposta mínima + logs redigidos

1 — Modelo mental: segurança é gestão de risco

Não existe API “100% segura”. Existe uma API cujo risco foi identificado, reduzido, monitorado e aceito de forma consciente. Segurança envolve responder quatro perguntas:

O que protegemos?

Dados pessoais, credenciais, dinheiro, disponibilidade, reputação, infraestrutura e ações privilegiadas.

De quem?

Usuário curioso, bot automatizado, invasor externo, dependência comprometida ou operador interno com acesso excessivo.

Com qual impacto?

Exposição, alteração, destruição, indisponibilidade, fraude ou uso indevido de recursos.

Erro de iniciante: pensar que “uso JWT e Helmet” significa que a API está segura. Esses mecanismos cobrem apenas partes específicas do problema.

2 — Ameaça, vulnerabilidade, exploração e impacto

TermoSignificadoExemplo
AmeaçaAlgo capaz de causar dano.Um bot tentando milhares de senhas.
VulnerabilidadeFraqueza que permite o dano.Login sem limite de tentativas.

Essa separação evita soluções superficiais. Antes de instalar uma biblioteca, descreva o abuso que deseja impedir e o que acontecerá se o controle falhar.

3 — Defesa em profundidade

Defesa em profundidade significa não depender de um único controle. Se a validação falhar, a query parametrizada ainda dificulta injeção. Se uma credencial vazar, o menor privilégio limita o estrago. Se uma senha for descoberta, rate limit, rotação de sessão e alertas reduzem o impacto.

Preventivo

Evita ou dificulta: validação, autorização, limites, TLS, isolamento e permissões mínimas.

Detectivo

Revela comportamento anormal: logs estruturados, métricas, alertas, auditoria e trilha de eventos.

Corretivo

Reduz o dano após o incidente: revogação, rotação de secrets, restauração, bloqueio e resposta a incidentes.

Recuperação

Permite retornar ao estado confiável: backups testados, migrações reversíveis e runbooks.

4 — Validação é necessária, mas não suficiente

Zod pode provar que role é uma string permitida, mas não prova que o usuário atual pode alterar essa role. Pode validar que amount é positivo, mas não que a conta possui saldo. Pode validar um nome de arquivo, mas não provar que o conteúdo é inofensivo.

const updateProfileSchema = z.object({
  displayName: z.string().trim().min(2).max(80),
});

// Validação estrutural: o valor tem formato aceitável.
// Autorização: quem está fazendo a alteração?
// Regra de negócio: esta alteração é permitida agora?
// Persistência: apenas campos explicitamente autorizados serão gravados.
Regra: dados válidos ainda podem ser maliciosos, indevidos para aquele usuário ou perigosos naquele contexto.

5 — Superfície de ataque

Superfície de ataque é o conjunto de pontos pelos quais alguém pode interagir com o sistema. Cada rota pública, header aceito, parser, upload, webhook, painel administrativo, porta exposta e dependência adiciona possibilidades de abuso.

  • Remova rotas, métodos e campos que não são necessários.
  • Não exponha documentação administrativa publicamente sem necessidade.
  • Não habilite parsers globais para formatos que a API não utiliza.
  • Evite retornar propriedades internas “porque já estão no objeto”.
  • Desabilite ou proteja endpoints de debug, métricas e health detalhado.

6 — CORS

CORS é uma política aplicada por navegadores que controla quais origens podem ler respostas de outra origem. Ele não é autenticação, não bloqueia chamadas de servidores, scripts fora do navegador, cURL ou ferramentas como Postman.

import cors from "cors";

const allowedOrigins = new Set([
  "https://app.example.com",
  "https://admin.example.com",
]);

app.use(cors({
  origin(origin, callback) {
    // Requisições sem Origin podem vir de ferramentas, mobile ou servidor.
    if (!origin) return callback(null, true);

    if (allowedOrigins.has(origin)) {
      return callback(null, true);
    }

    return callback(new Error("Origin não permitida"));
  },
  credentials: true,
  methods: ["GET", "POST", "PATCH", "DELETE"],
  allowedHeaders: ["Content-Type", "Authorization", "X-Request-Id"],
}));

6.1 — Origem não é URL completa

Uma origem combina esquema, host e porta. Paths não fazem parte dela. https://app.example.com/a e https://app.example.com/b possuem a mesma origem.

6.2 — Credenciais e wildcard

Quando cookies ou credenciais cross-origin são usados, a origem deve ser tratada explicitamente. Uma configuração permissiva copiada sem entender pode expor respostas para frontends não autorizados.

CORS não protege sua API contra acesso: ele controla a leitura da resposta pelo JavaScript do navegador. A API continua precisando de autenticação e autorização.

7 — Helmet e headers defensivos

Helmet configura diversos headers HTTP defensivos. Em uma API JSON, alguns têm impacto maior que outros; ainda assim, a configuração cria uma base consistente e reduz comportamentos inseguros do navegador.

import helmet from "helmet";

app.use(helmet({
  // Ajuste políticas conscientemente caso a mesma aplicação sirva HTML.
  contentSecurityPolicy: false,
}));

app.disable("x-powered-by");

Não desative políticas apenas para “sumir com um erro”. Descubra se a aplicação serve HTML, documentação, arquivos ou somente JSON e configure conforme o contexto.

Sobre X-Powered-By: ocultá-lo reduz informação trivial sobre a stack, mas não substitui atualização, correções e controles reais.

8 — HTTPS

HTTPS protege a comunicação em trânsito contra leitura e alteração por intermediários. Tokens, cookies, senhas e dados pessoais não devem atravessar redes em HTTP simples.

  • Em produção, normalmente o TLS termina no proxy reverso, load balancer ou plataforma.
  • A comunicação interna também pode exigir TLS conforme o ambiente e o risco.
  • Cookies de autenticação em produção devem usar Secure.
  • Redirecione HTTP para HTTPS no perímetro, não apenas dentro da aplicação.
HTTPS não torna conteúdo confiável: ele protege o canal. Um cliente autenticado ainda pode enviar payload malicioso por uma conexão criptografada.

9 — Proxy reverso

Um proxy reverso recebe conexões externas e encaminha requisições para a aplicação. Pode terminar TLS, aplicar limites, comprimir respostas, distribuir carga e esconder a porta interna do processo Node.js.

internet → proxy reverso / load balancer → aplicação Express → banco

O proxy adiciona uma fronteira. Headers como X-Forwarded-For e X-Forwarded-Proto passam a influenciar IP e protocolo percebidos pela aplicação. Por isso, confiar neles indiscriminadamente é perigoso.

10 — trust proxy

O Express usa trust proxy para decidir se deve confiar nas informações enviadas pelo proxy. Isso afeta request.ip, request.ips, request.protocol e cookies seguros.

// Exemplo comum quando existe exatamente um proxy confiável à frente.
app.set("trust proxy", 1);

O valor correto depende da topologia. Usar true sem entender a cadeia pode fazer a aplicação confiar em headers forjados pelo cliente, afetando rate limiting, auditoria e decisões de segurança.

Não copie a configuração do provedor às cegas: documente quantos proxies confiáveis existem e teste qual IP a aplicação realmente observa.

11 — Rate limiting

Rate limiting restringe quantas requisições uma identidade pode fazer em uma janela. A identidade pode ser IP, usuário, token, chave de API ou uma combinação. O objetivo é reduzir abuso, custo e indisponibilidade — não “garantir” que ataques sejam impossíveis.

import { rateLimit } from "express-rate-limit";

const apiLimiter = rateLimit({
  windowMs: 60_000,
  limit: 120,
  standardHeaders: "draft-8",
  legacyHeaders: false,
});

app.use("/api", apiLimiter);

11.1 — Limite global versus limite por operação

OperaçãoEstratégia
Leitura pública barataLimite moderado e cache quando possível.
LoginLimite mais rígido por IP e identificador normalizado.
Recuperação de senhaLimite rígido, resposta genérica e cooldown.
Relatório caroLimite por usuário, fila e controle de concorrência.

11.2 — Ambiente distribuído

Armazenamento em memória não é suficiente quando existem várias instâncias, porque cada processo conta separadamente. Uma store compartilhada ou controle no gateway mantém a política consistente.

12 — Proteção contra brute force

Brute force tenta muitas combinações para descobrir um segredo. Credential stuffing reutiliza credenciais vazadas de outros serviços. Proteções precisam considerar tanto IP quanto a conta alvo.

const loginLimiter = rateLimit({
  windowMs: 15 * 60_000,
  limit: 10,
  keyGenerator(request) {
    const email = String(request.body?.email ?? "")
      .trim()
      .toLowerCase();

    return `${request.ip}:${email}`;
  },
  skipSuccessfulRequests: true,
});
  • Não revele se o e-mail existe com mensagens diferentes.
  • Use hash de senha adequado, MFA quando aplicável e sessões revogáveis.
  • Considere atraso progressivo e alertas de comportamento anormal.
  • Evite bloqueio permanente fácil de explorar contra vítimas.
Trade-off: uma regra agressiva por IP pode bloquear vários usuários atrás de NAT corporativo. Uma regra apenas por conta permite ataque distribuído. Combine sinais.

13 — Limite de payload

Parsers precisam de limites. Sem eles, um cliente pode consumir memória e CPU enviando bodies enormes. Configure apenas os formatos usados pela aplicação.

app.use(express.json({
  limit: "100kb",
  strict: true,
}));

app.use(express.urlencoded({
  extended: false,
  limit: "50kb",
}));

O limite ideal depende da rota. Uma API de notas talvez aceite poucos quilobytes; upload de arquivos deve usar fluxo e limites próprios, não um JSON gigantesco em Base64.

14 — Timeouts, concorrência e consumo de recursos

Disponibilidade também é segurança. Requisições lentas, queries caras e integrações sem timeout podem ocupar conexões até o serviço colapsar.

  • Defina timeout para chamadas HTTP externas e banco quando suportado.
  • Limite paginação, profundidade de filtros e tamanho de resultados.
  • Use filas para tarefas demoradas e controle concorrência.
  • Não faça processamento pesado de arquivo no event loop sem estratégia.
  • Considere cancelamento quando o cliente desconectar.

15 — Mass assignment

Mass assignment ocorre quando propriedades recebidas do cliente são copiadas diretamente para uma entidade ou update do banco. O atacante envia campos que não deveriam ser alteráveis.

// Vulnerável:
await prisma.user.update({
  where: { id: userId },
  data: request.body,
});

// Explícito:
const input = updateProfileSchema.parse(request.body);

await prisma.user.update({
  where: { id: userId },
  data: {
    displayName: input.displayName,
    bio: input.bio,
  },
});

Mesmo que o TypeScript mostre um DTO sem role, o objeto real em runtime pode possuir propriedades extras. Use schemas estritos quando adequado e mapeie campos de escrita explicitamente.

Nunca: data: { ...request.body } em operações sensíveis.

16 — Injeção

Injeção acontece quando dado não confiável é interpretado como comando, query ou código. SQL injection é a forma clássica, mas existem injeções em shell, templates, LDAP, NoSQL e logs.

16.1 — SQL

// Evite concatenar entrada:
const unsafe = `SELECT * FROM users WHERE email = '${email}'`;

// Prefira APIs parametrizadas / ORM:
const user = await prisma.user.findUnique({
  where: { email },
});

16.2 — Shell

// Perigoso quando filename vem do cliente:
// exec(`convert ${filename} output.png`);

// Prefira biblioteca nativa; se processo externo for inevitável,
// use argumentos separados, allowlist e ambiente isolado.

16.3 — Ordenação dinâmica

const allowedSorts = {
  createdAt: "createdAt",
  title: "title",
} as const;

const sortField = allowedSorts[input.sortBy] ?? "createdAt";
Validação por allowlist: quando a entrada escolhe comportamento estrutural — coluna, operação, formato — aceite somente opções conhecidas.

17 — Vazamento de dados

Uma API pode cumprir a ação e ainda vazar informações na resposta. Entidades internas quase nunca devem ser serializadas diretamente.

type UserResponse = {
  id: string;
  name: string;
  avatarUrl: string | null;
};

function toUserResponse(user: User): UserResponse {
  return {
    id: user.id,
    name: user.name,
    avatarUrl: user.avatarUrl,
  };
}

Campos como passwordHash, token de recuperação, flags internas, dados de auditoria, e-mails privados e identificadores de provedores devem ser excluídos por desenho, não “lembrados” em cada controller.

17.1 — Erros também vazam

Stack trace, SQL, paths do servidor, nomes de tabela e detalhes de dependências ajudam diagnóstico interno, mas não devem ser enviados ao cliente em produção.

18 — Logs sensíveis

Logs são dados persistidos. Tudo que vai para eles pode ser copiado para agregadores, backups e sistemas de terceiros. Não registre senhas, tokens, cookies, números completos de documentos ou bodies inteiros por padrão.

function redactHeaders(headers: Record<string, unknown>) {
  const copy = { ...headers };

  if ("authorization" in copy) copy.authorization = "[REDACTED]";
  if ("cookie" in copy) copy.cookie = "[REDACTED]";

  return copy;
}
  • Use request ID para correlação sem depender de dados pessoais.
  • Defina retenção e acesso aos logs.
  • Separe log operacional de trilha de auditoria.
  • Sanitize caracteres de controle em campos livres quando necessário.

19 — Secrets

Secrets incluem senhas de banco, chaves de assinatura, tokens de provedores e credenciais de serviço. Eles não devem ser versionados, embutidos no bundle ou enviados ao frontend.

const envSchema = z.object({
  DATABASE_URL: z.string().min(1),
  JWT_SECRET: z.string().min(32),
  NODE_ENV: z.enum(["development", "test", "production"]),
});

export const env = envSchema.parse(process.env);
  • Use gerenciador de secrets da plataforma quando disponível.
  • Rotacione credenciais periodicamente e imediatamente após suspeita de vazamento.
  • Use secrets distintos por ambiente.
  • Não imprima o objeto completo de variáveis de ambiente.
  • Planeje rotação sem indisponibilidade quando a chave assina tokens.

20 — Uploads

Upload combina várias ameaças: esgotamento de disco e memória, conteúdo executável, nomes maliciosos, arquivos poliglotas, metadados sensíveis e processamento vulnerável.

import multer from "multer";

const upload = multer({
  storage: multer.memoryStorage(),
  limits: {
    fileSize: 5 * 1024 * 1024,
    files: 1,
    fields: 5,
  },
});

app.post(
  "/avatars",
  authenticate,
  upload.single("avatar"),
  uploadAvatarController,
);

Memória não é automaticamente mais segura: arquivos grandes ou concorrência podem esgotar o processo. Armazenamento temporário em disco também exige limpeza, permissões e isolamento.

21 — Validação de arquivos

Extensão e Content-Type declarados pelo cliente não provam o conteúdo. Valide tamanho, assinatura conhecida quando possível, dimensões, quantidade e fluxo de processamento.

  • Gere um nome interno aleatório; não use o nome original como path.
  • Evite servir uploads no mesmo domínio com execução permissiva.
  • Armazene fora da raiz pública quando não precisarem ser públicos.
  • Reencode imagens em vez de confiar no arquivo original quando adequado.
  • Faça varredura antimalware para cenários que exigem isso.
  • Remova metadados que possam conter localização ou informação pessoal.
Path traversal: um nome como ../../config.env não pode decidir onde o servidor escreverá.

22 — Princípio do menor privilégio

Cada componente deve possuir apenas as permissões necessárias e pelo menor tempo possível.

Banco

A aplicação não precisa ser superusuária. Separe credenciais de migration e runtime.

Sistema operacional

O processo Node não deve executar como root nem escrever em diretórios desnecessários.

Cloud

A identidade da aplicação deve acessar apenas buckets, filas e secrets usados.

Aplicação

Roles e ownership devem restringir ações e dados, não apenas esconder botões no frontend.

23 — Dependências e cadeia de suprimentos

Dependências executam dentro do seu processo e recebem acesso aos mesmos dados e permissões. Reduza quantidade, mantenha lockfile, revise atualizações e remova pacotes sem uso.

  • Prefira pacotes amplamente mantidos e com escopo claro.
  • Revise scripts de instalação e mudanças relevantes.
  • Use auditoria automática como sinal, não como substituto de análise.
  • Atualize com testes; versões antigas acumulam vulnerabilidades e incompatibilidades.
  • Não instale um pacote para algo trivial que a plataforma já oferece.

24 — Baseline seguro para Express

import express from "express";
import cors from "cors";
import helmet from "helmet";
import { rateLimit } from "express-rate-limit";

export const app = express();

app.set("trust proxy", 1);
app.disable("x-powered-by");

app.use(helmet());
app.use(cors({
  origin: ["https://app.example.com"],
  credentials: true,
}));

app.use(rateLimit({
  windowMs: 60_000,
  limit: 120,
  standardHeaders: "draft-8",
  legacyHeaders: false,
}));

app.use(express.json({
  limit: "100kb",
  strict: true,
}));

app.get("/health", (_request, response) => {
  return response.status(200).json({ status: "ok" });
});
Baseline não é receita universal: trust proxy, CORS, limites, CSP e cookies dependem da topologia e dos clientes reais.

25 — Como auditar uma rota

Use um roteiro repetível em vez de procurar apenas “código estranho”.

  1. Quem consegue chamar a rota?
  2. Qual identidade foi autenticada e onde ela é armazenada?
  3. Há autorização por role e ownership?
  4. Todos os dados externos passam por parsing e validação?
  5. Campos gravados são explicitamente selecionados?
  6. Queries são parametrizadas?
  7. A resposta contém somente dados públicos necessários?
  8. O custo da operação é limitado?
  9. Logs evitam dados sensíveis?
  10. Erros retornam contrato previsível sem detalhes internos?

25.1 — Exemplo vulnerável

app.patch("/users/:id", async (request, response) => {
  const user = await prisma.user.update({
    where: { id: request.params.id },
    data: request.body,
  });

  console.log("updated user", user);

  return response.json(user);
});

25.2 — Problemas

  • Sem autenticação e autorização.
  • Mass assignment.
  • Sem validação de params e body.
  • Possível vazamento de hash e campos internos na resposta e no log.
  • Sem contrato de erro previsível.

25.3 — Direção corrigida

app.patch(
  "/me/profile",
  authenticate,
  validate(updateProfileSchema),
  updateProfileController,
);

// O service usa authenticatedUser.id.
// O repository recebe apenas displayName e bio.
// O mapper retorna somente o perfil público.

27 — Edge cases de segurança

Edge cases são situações menos óbvias que quebram uma suposição do fluxo principal. Em segurança, atacantes procuram exatamente essas bordas.

Identidade e rede

  • Vários usuários compartilham o mesmo IP.
  • IPv6 muda a estratégia de agrupamento.
  • Proxy adiciona múltiplos valores encaminhados.
  • Token válido pertence a usuário desativado.
  • Conta muda de role enquanto sessão continua ativa.

Dados e parsing

  • JSON duplicado ou profundamente aninhado.
  • Unicode visualmente semelhante em e-mail ou nome.
  • Payload comprimido pequeno que expande muito.
  • Campo extra é ignorado pelo tipo, mas persiste em runtime.
  • Número válido causa custo extremo numa query.

Uploads

  • Arquivo vazio com MIME permitido.
  • Imagem com dimensões gigantes e poucos bytes.
  • Nome reservado ou com separadores de path.
  • Arquivo válido no início e malicioso no restante.
  • Upload interrompido deixa arquivo temporário.

Disponibilidade

  • Muitas requisições lentas abaixo do limite.
  • Relatório válido consome toda a CPU.
  • Store do rate limiter fica indisponível.
  • Cliente desconecta durante operação cara.
  • Logs excessivos saturam disco ou provedor.

28 — Testes de segurança

Testes não provam ausência de vulnerabilidades, mas impedem regressões nos controles conhecidos.

describe("PATCH /me/profile", () => {
  it("rejects an unauthenticated request", async () => {});
  it("ignores or rejects forbidden fields such as role", async () => {});
  it("does not expose passwordHash in the response", async () => {});
  it("rejects a body above the configured limit", async () => {});
});

describe("POST /sessions", () => {
  it("applies a limit after repeated failed attempts", async () => {});
  it("returns the same public message for unknown email and wrong password", async () => {});
});

28.1 — O que testar além do happy path

  • Ausência, expiração e revogação de credenciais.
  • Usuário autenticado tentando recurso de outro usuário.
  • Campos extras, tipos inesperados e payload grande.
  • Headers de segurança esperados.
  • Respostas sem stack trace e sem campos internos.
  • Rate limit sob múltiplas identidades.

29 — Exercícios orientados

Exercício 1 — Classifique os controles

Classifique CORS, rate limiting, logs, backup e revogação como preventivo, detectivo, corretivo ou recuperação. Alguns podem ocupar mais de uma categoria.

Pergunte: esse controle impede, revela, reduz dano ou restaura?

Exercício 2 — Audite um update

Receba uma rota que usa data: request.body. Liste todas as propriedades que um atacante tentaria enviar e proponha DTO, schema, autorização e mapper.

Exercício 3 — Desenhe uma política de login

Defina limite por IP, por conta, janela, resposta pública, monitoramento e comportamento quando a store do limiter falhar.

Exercício 4 — Upload de avatar

Defina formatos aceitos, tamanho máximo, dimensões, nome interno, armazenamento, limpeza temporária e resposta.

Exercício 5 — CORS de três ambientes

Modele origens de development, preview e production sem usar reflexão irrestrita da origem recebida.

30 — Mini projeto: auditoria de uma API vulnerável

Você receberá uma API de documentos deliberadamente vulnerável. O objetivo não é adicionar bibliotecas aleatórias, mas produzir um relatório de risco, corrigir os fluxos e provar as correções com testes.

30.1 — Funcionalidades

POST   /sessions
POST   /password-recovery
GET    /documents
POST   /documents
PATCH  /documents/:documentId
DELETE /documents/:documentId
POST   /avatars
GET    /admin/audit

30.2 — Vulnerabilidades inseridas

  • CORS refletindo qualquer origem com credenciais.
  • Login sem limite e mensagens que enumeram usuários.
  • Body sem limite.
  • Update com mass assignment.
  • Falta de ownership.
  • Resposta retornando entidade completa.
  • Logs com token e senha.
  • Secret hardcoded.
  • trust proxy configurado incorretamente.
  • Upload baseado apenas em extensão e nome original.
  • Credencial de banco com privilégios excessivos.

30.3 — Entregáveis

  1. Inventário de ativos e superfície de ataque.
  2. Tabela com vulnerabilidade, exploração, impacto, severidade e correção.
  3. Configuração de baseline HTTP.
  4. Refatoração de rotas, schemas, services, repositories e mappers.
  5. Testes de regressão para cada falha corrigida.
  6. Documento curto com riscos residuais e decisões de trade-off.

30.4 — Critério de sucesso

A API não precisa apenas “funcionar”. Você deve conseguir explicar qual ameaça cada controle reduz, onde ele pode falhar e como detectar abuso ou regressão.

31 — Checklist para avançar

  • Consigo explicar por que CORS não é autenticação.
  • Sei o papel de Helmet sem tratá-lo como solução completa.
  • Configuro limite de body e entendo o motivo.
  • Sei diferenciar rate limit global, por rota, IP, conta e usuário.
  • Entendo brute force e credential stuffing.
  • Não uso request.body diretamente em updates.
  • Sei reconhecer injeção e usar allowlists e parametrização.
  • Uso DTOs de resposta para impedir vazamento acidental.
  • Sei quais dados não devem aparecer em logs.
  • Valido secrets no startup e planejo rotação.
  • Entendo proxy reverso e configuro trust proxy conforme a topologia.
  • Trato uploads como conteúdo não confiável.
  • Aplico menor privilégio no banco, sistema e cloud.
  • Consigo auditar uma rota de ponta a ponta.

Fase 10

Autenticação e autorização

Express 5TypeScriptPrismaCookiesJWTNotas privadas

Autenticação responde “quem está fazendo esta requisição?”. Autorização responde “essa identidade pode executar esta ação sobre este recurso?”. As duas aparecem juntas com frequência, mas resolvem problemas diferentes.

Objetivo da fase: construir um sistema de login que trate senha, sessão, tokens, cookies, logout, recuperação de conta e autorização como partes de um mesmo modelo de segurança — não como trechos isolados copiados de um tutorial.
credencial apresentada ↓ identidade verificada ↓ sessão ou tokens emitidos ↓ requisição autenticada ↓ permissão + propriedade verificadas ↓ ação permitida ou negada

1 — O modelo mental desta fase

1. O usuário prova uma identidade com uma credencial, como senha ou provedor externo.
2. A aplicação não repete o login em cada requisição; ela cria continuidade por sessão ou token.
3. O servidor valida a credencial de continuidade e produz um contexto autenticado.
4. A regra de autorização avalia papel, permissão, propriedade e estado do recurso.
5. Logout, expiração, troca de senha e suspeita de comprometimento precisam encerrar ou limitar essa continuidade.

O ponto mais importante é perceber que estar logado não significa poder fazer qualquer coisa. Também não significa que o servidor deva confiar em um userId enviado no body.

2 — Identidade, autenticação e autorização

Identificação

O usuário afirma quem é: informa um e-mail, username ou ID.

Autenticação

Ele apresenta uma prova: senha, passkey, código temporário ou identidade federada.

Autorização

A aplicação decide se a identidade autenticada pode executar determinada ação.

Auditoria

O sistema registra quem fez o quê, quando e sobre qual recurso.

401 e 403: em geral, 401 Unauthorized significa que não há autenticação válida; 403 Forbidden significa que a identidade é conhecida, mas não possui autorização. O nome histórico do 401 é confuso.

3 — Fluxo completo de autenticação

  1. Cadastro: validar entrada, normalizar e-mail, gerar hash da senha e criar usuário.
  2. Login: localizar a conta, comparar a senha de forma segura e aplicar controles contra abuso.
  3. Emissão: criar uma sessão opaca ou um par de tokens.
  4. Persistência no cliente: configurar cookies e política de transporte.
  5. Autenticação das rotas: extrair credencial, validar, carregar contexto mínimo e rejeitar credenciais expiradas ou revogadas.
  6. Autorização: verificar papel, permissão e ownership.
  7. Renovação: prolongar a sessão de maneira controlada.
  8. Encerramento: invalidar a sessão ou família de refresh tokens e limpar cookies.

4 — Senhas não são armazenadas

O banco não deve guardar a senha original nem uma versão reversivelmente criptografada. Deve guardar o resultado de uma função de derivação de chave para senhas. No login, a aplicação executa a função de verificação usando o hash armazenado.

const user = await usersRepository.findByEmail(email);

if (!user) {
  throw new InvalidCredentialsError();
}

const passwordMatches = await passwordHasher.verify(
  user.passwordHash,
  password,
);

if (!passwordMatches) {
  throw new InvalidCredentialsError();
}

Use a mesma resposta para e-mail inexistente e senha incorreta. Caso contrário, a API ajuda um atacante a descobrir quais contas existem.

5 — Hash, salt e pepper

Hash de senha

Resultado unidirecional produzido por um algoritmo deliberadamente caro. A verificação compara a senha candidata com o hash armazenado.

Salt

Valor aleatório único por senha. Evita hashes iguais para senhas iguais e dificulta tabelas pré-computadas. Bibliotecas modernas normalmente geram e embutem o salt no hash.

Pepper

Segredo adicional mantido fora do banco, por exemplo em secret manager. É uma defesa extra, não substitui salt nem algoritmo adequado.

Work factor

Configuração de custo. Deve tornar tentativas em massa caras sem causar indisponibilidade no servidor.

Não invente criptografia. Não use SHA-256 puro, MD5 ou uma sequência manual de hashes para senhas. Use uma biblioteca reconhecida e mantenha parâmetros versionáveis.

6 — Argon2id, scrypt e bcrypt

Para um projeto novo, Argon2id costuma ser a primeira opção; scrypt é uma alternativa sólida. bcrypt ainda é amplamente usado, mas possui limitações, inclusive de tamanho de entrada. O curso pode encapsular a implementação para não acoplar regras de negócio a uma biblioteca.

export interface PasswordHasher {
  hash(plainText: string): Promise<string>;
  verify(hash: string, plainText: string): Promise<boolean>;
}

export class CreateUserService {
  constructor(
    private readonly users: UsersRepository,
    private readonly hasher: PasswordHasher,
  ) {}
}

Esse contrato melhora testes, permite migração futura e evita espalhar chamadas de biblioteca por controllers e services.

7 — Cadastro seguro

  • Valide formato, tamanho e política de senha na fronteira.
  • Normalize e-mail de maneira consciente, sem alterar identificadores arbitrariamente.
  • Garanta unicidade também no banco.
  • Não confie apenas em uma consulta “já existe?”; duas requisições podem competir.
  • Gere o hash antes de persistir.
  • Nunca retorne passwordHash.
  • Considere verificação de e-mail antes de liberar ações sensíveis.
const email = input.email.trim().toLowerCase();

const passwordHash = await hasher.hash(input.password);

try {
  const user = await users.create({ email, passwordHash });
  return toPublicUser(user);
} catch (error) {
  if (isUniqueConstraintViolation(error)) {
    throw new EmailAlreadyInUseError();
  }
  throw error;
}

8 — Login seguro

O login é uma superfície de ataque. Além de verificar credenciais, precisa limitar tentativas, evitar enumeração, registrar eventos relevantes e controlar contas desabilitadas ou não verificadas.

const GENERIC_MESSAGE = "E-mail ou senha inválidos.";

if (!user || !(await hasher.verify(user.passwordHash, password))) {
  throw new InvalidCredentialsError(GENERIC_MESSAGE);
}

if (user.status !== "ACTIVE") {
  throw new InvalidCredentialsError(GENERIC_MESSAGE);
}
Tempo de resposta: retornar imediatamente para usuário inexistente e executar um hash caro apenas para usuários existentes pode criar diferença mensurável. Sistemas mais sensíveis usam um hash fictício para reduzir esse sinal.

9 — Sessões

Em autenticação baseada em sessão, o navegador recebe um identificador aleatório. O estado real fica no servidor, geralmente em Redis ou banco. O cookie contém somente o identificador opaco.

type Session = {
  id: string;
  userId: string;
  expiresAt: Date;
  revokedAt: Date | null;
};

const session = await sessions.create({
  userId: user.id,
  expiresAt: addDays(new Date(), 7),
});

response.cookie("__Host-session", session.id, cookieOptions);

Vantagens: revogação simples, estado centralizado e cookie pequeno. Custos: armazenamento e consulta compartilhados quando a aplicação escala horizontalmente.

10 — Cookies

Cookie é um mecanismo HTTP do navegador. O servidor envia Set-Cookie; o navegador armazena o valor e o envia em requisições futuras que correspondam a domínio, path e política de SameSite.

response.cookie("__Host-refresh", refreshToken, {
  httpOnly: true,
  secure: env.NODE_ENV === "production",
  sameSite: "lax",
  path: "/",
  maxAge: 1000 * 60 * 60 * 24 * 30,
});

Cookie não é sinônimo de sessão. Um cookie pode transportar ID de sessão, refresh token ou preferência visual. JWT também não é sinônimo de “armazenar no localStorage”.

11 — JWT

JWT é um formato compacto de claims assinado. A assinatura permite detectar alteração; ela não torna o payload secreto. Qualquer pessoa que possua o token normalmente consegue decodificar o payload.

type AccessTokenClaims = {
  sub: string;       // subject: ID do usuário
  sid: string;       // sessão ou família
  role: "USER" | "ADMIN";
  iat: number;
  exp: number;
  iss: string;
  aud: string;
};
  • Valide algoritmo permitido, assinatura, exp, iss e aud.
  • Não coloque senha, documentos ou dados sensíveis no payload.
  • Mantenha claims pequenas e estáveis.
  • Não use JWT só porque a aplicação possui React.

12 — Sessão ou JWT?

Regra prática: comece pela opção mais simples que satisfaça a arquitetura. “Stateless” não é uma virtude automática, e refresh token seguro quase sempre exige persistência.

13 — Access token e refresh token

O access token autoriza chamadas à API e deve ter vida curta. O refresh token possui vida maior e serve apenas para obter um novo par. Ele é uma credencial de alto valor.

POST /sessions → access token curto → refresh token longo GET /notes Authorization: Bearer <access> POST /sessions/refresh cookie refresh → novo access → novo refresh

Não use refresh token como access token. Restrinja seu uso ao endpoint de renovação e armazene no servidor somente uma representação segura, como hash do token.

14 — Rotação e detecção de reutilização

Na rotação, cada refresh consome o token atual e emite outro. O token anterior deixa de ser válido. Os tokens pertencem a uma família para que reutilização de um token antigo possa indicar vazamento.

await prisma.$transaction(async (tx) => {
  const stored = await tx.refreshToken.findUnique({
    where: { tokenHash },
  });

  if (!stored || stored.revokedAt || stored.expiresAt <= new Date()) {
    throw new InvalidRefreshTokenError();
  }

  await tx.refreshToken.update({
    where: { id: stored.id },
    data: { revokedAt: new Date(), replacedById: nextTokenId },
  });

  await tx.refreshToken.create({ data: nextTokenData });
});

A operação precisa ser atômica. Duas renovações simultâneas exigem uma política clara: uma vence e a outra é rejeitada, ou existe uma pequena janela de tolerância cuidadosamente projetada.

15 — Logout e revogação

Limpar o cookie no navegador não basta quando existe credencial válida no servidor ou um refresh token roubado. Logout deve revogar a sessão ou token atual. “Sair de todos os dispositivos” revoga toda a família ou incrementa uma versão de sessão da conta.

await sessions.revokeById(auth.sessionId);

response.clearCookie("__Host-refresh", {
  httpOnly: true,
  secure: true,
  sameSite: "lax",
  path: "/",
});

return response.status(204).end();

Os atributos usados em clearCookie precisam corresponder aos atributos relevantes usados na criação.

16 — HttpOnly, Secure e SameSite

HttpOnly

Impede leitura pelo JavaScript da página. Reduz roubo direto por XSS, embora não impeça um script malicioso de disparar requisições.

Secure

Faz o navegador enviar o cookie apenas por HTTPS, com exceções de desenvolvimento para localhost.

SameSite

Controla envio em contexto cross-site. Strict, Lax e None representam compromissos diferentes.

Prefixo __Host-

Exige Secure, Path=/ e ausência de Domain, restringindo o cookie ao host.

SameSite=None exige Secure. Defina SameSite explicitamente; não dependa de defaults de navegador.

17 — CSRF

CSRF explora credenciais enviadas automaticamente pelo navegador, especialmente cookies. Um site malicioso tenta induzir o navegador autenticado a executar uma ação em outro site.

  • Use SameSite adequado como uma camada.
  • Para cenários cross-site, use token anti-CSRF ou padrão equivalente.
  • Valide Origin ou Referer em ações sensíveis quando apropriado.
  • Não aceite mudança de estado por GET.
  • Reautentique ou exija confirmação para ações de alto impacto.
const origin = request.get("origin");

if (!origin || !allowedOrigins.has(origin)) {
  throw new ForbiddenError("Origem não permitida.");
}

HttpOnly não resolve CSRF. CORS também não deve ser tratado como mecanismo primário de autorização.

18 — CORS e credenciais

CORS é uma política aplicada por navegadores que controla leitura de respostas cross-origin. Para enviar cookies em uma requisição cross-origin, o frontend usa credenciais e o servidor autoriza uma origem específica.

app.use(cors({
  origin: "https://app.example.com",
  credentials: true,
}));
await fetch("https://api.example.com/sessions/refresh", {
  method: "POST",
  credentials: "include",
});
Não combine credenciais com origem curinga. Quando há cookies, responda apenas para origens explicitamente confiáveis e mantenha a política coerente com CSRF.

19 — Middleware de autenticação

O middleware transforma uma credencial externa em contexto confiável para as próximas camadas. Ele não deve assumir que decodificar um JWT equivale a verificar assinatura e claims.

export const authenticate: RequestHandler = async (
  request,
  response,
  next,
) => {
  const token = extractBearerToken(request);

  if (!token) {
    throw new UnauthorizedError();
  }

  const claims = await accessTokens.verify(token);

  response.locals.auth = {
    userId: claims.sub,
    sessionId: claims.sid,
    role: claims.role,
  };

  next();
};

20 — Contexto autenticado

Use um objeto pequeno e explícito. Não espalhe o payload inteiro do token pela aplicação. Quando a regra precisa do usuário atual, carregue os dados necessários do repositório.

type AuthContext = {
  userId: string;
  sessionId: string;
  role: "USER" | "ADMIN";
};

declare global {
  namespace Express {
    interface Locals {
      auth: AuthContext;
    }
  }
}

Outra opção é tipar Response<Body, AuthLocals> localmente. Evite afirmações globais que tornem auth aparentemente obrigatório em rotas públicas.

21 — RBAC

Role-Based Access Control associa permissões a papéis, como USER, MODERATOR e ADMIN. É útil para capacidades amplas, mas não substitui ownership.

export function requireRole(
  ...allowed: AuthContext["role"][]
): RequestHandler {
  return (_request, response, next) => {
    if (!allowed.includes(response.locals.auth.role)) {
      throw new ForbiddenError();
    }
    next();
  };
}

Evite if (role === "ADMIN") espalhado em dezenas de services. Centralize políticas ou expresse permissões do domínio.

22 — Ownership

Ownership verifica se o recurso pertence ao usuário autenticado ou está legitimamente compartilhado com ele.

const note = await notesRepository.findById(noteId);

if (!note) {
  throw new NotFoundError("Note");
}

if (note.userId !== auth.userId) {
  throw new ForbiddenError();
}

Em muitos sistemas, buscar diretamente por id + userId é melhor: reduz risco de esquecer a checagem e pode responder 404 tanto para inexistente quanto para recurso alheio.

const note = await notesRepository.findOwnedById({
  noteId,
  userId: auth.userId,
});

if (!note) {
  throw new NotFoundError("Note");
}

23 — Políticas de autorização

Autorização pode considerar mais do que papel e dono: status da conta, compartilhamento, organização, plano contratado, horário, estado do recurso e ação solicitada.

function canUpdateNote(
  auth: AuthContext,
  note: Note,
): boolean {
  return auth.role === "ADMIN" || note.userId === auth.userId;
}

Para regras importantes, prefira funções nomeadas ou serviços de política testáveis. O controller autentica a entrada HTTP; a regra de domínio decide o acesso.

24 — Recuperação de senha

  1. Cliente solicita recuperação com e-mail.
  2. API responde genericamente, exista ou não a conta.
  3. Servidor gera token aleatório de uso único.
  4. Banco guarda hash do token, expiração e estado de consumo.
  5. E-mail contém o token original em URL HTTPS.
  6. Na redefinição, o token é verificado e consumido atomicamente.
  7. A senha recebe novo hash e sessões existentes podem ser revogadas.
type PasswordResetToken = {
  id: string;
  userId: string;
  tokenHash: string;
  expiresAt: Date;
  usedAt: Date | null;
};
Não envie senha nova por e-mail. Não use token previsível. Não armazene o token original. Não permita reutilização.

25 — Verificação de e-mail

O fluxo é semelhante ao de recuperação, mas confirma posse do endereço. O token deve expirar, ser de uso único e apontar para uma finalidade específica.

  • Não reutilize token de verificação como token de recuperação.
  • Ao trocar o e-mail, marque o novo endereço como não verificado.
  • Limite reenvios para evitar spam.
  • Considere invalidar tokens antigos quando um novo for emitido.
  • Não confunda e-mail verificado com identidade juridicamente comprovada.

26 — Rate limiting

Rotas de login, refresh, recuperação, verificação e cadastro precisam de limites próprios. Um único limite global pode ser insuficiente ou prejudicar usuários legítimos.

Por IP

Útil contra rajadas, mas NAT e redes corporativas agrupam usuários.

Por conta

Ajuda contra ataques direcionados, mas pode permitir bloqueio malicioso de uma vítima.

Combinado

IP, identificador normalizado, dispositivo, janela e sinais de risco.

Distribuído

Em múltiplas instâncias, use armazenamento compartilhado e operações atômicas.

28 — Modelagem com Prisma

model User {
  id             String         @id @default(uuid())
  email          String         @unique
  passwordHash   String
  role           Role           @default(USER)
  emailVerifiedAt DateTime?
  sessions       Session[]
  notes          Note[]
  createdAt      DateTime       @default(now())
  updatedAt      DateTime       @updatedAt
}

model Session {
  id        String    @id
  userId    String
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  expiresAt DateTime
  revokedAt DateTime?
  createdAt DateTime  @default(now())

  @@index([userId])
  @@index([expiresAt])
}

model Note {
  id        String   @id @default(uuid())
  userId    String
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  title     String
  content   String
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@index([userId, updatedAt])
}

enum Role {
  USER
  ADMIN
}

Para refresh token rotativo, modele família, hash, expiração, revogação e relacionamento de substituição. Índices devem refletir consultas reais e limpeza de expirados.

29 — Testes de autenticação e autorização

  • Cadastro cria hash e nunca retorna hash.
  • E-mail duplicado é rejeitado mesmo sob concorrência.
  • Login aceita credencial correta e rejeita incorreta com mensagem genérica.
  • Token expirado, assinatura inválida, issuer errado e audience errada são rejeitados.
  • Refresh rotaciona e invalida o anterior.
  • Reutilização revoga a família conforme política.
  • Logout invalida sessão e limpa cookie.
  • Usuário não lê, edita nem exclui nota de outro usuário.
  • Admin só recebe exceções explicitamente previstas.
  • Reset token é de uso único e expira.
it("should not allow a user to read another user's note", async () => {
  const owner = await createUser();
  const stranger = await createUser();
  const note = await createNote({ userId: owner.id });

  const response = await request(app)
    .get(`/notes/${note.id}`)
    .set("Authorization", bearerFor(stranger));

  expect(response.status).toBe(404);
});

30 — Erros comuns

  • Guardar senha em texto ou usar hash rápido.
  • Confiar no userId recebido no body.
  • Tratar JWT como criptografia.
  • Access token com duração excessiva.
  • Refresh token eterno, sem rotação ou revogação.
  • Guardar refresh token puro no banco.
  • Colocar token de longa duração em localStorage sem avaliar XSS.
  • Acreditar que HttpOnly elimina CSRF.
  • Usar CORS como autorização.
  • Retornar “e-mail não cadastrado” no login ou reset.
  • Permitir que role seja definido no cadastro público.
  • Fazer logout apenas no frontend.

31 — Edge cases

31.1 — Concorrência no refresh

Duas abas renovam ao mesmo tempo. O primeiro token é consumido e o segundo parece reutilização. Defina sincronização no cliente ou uma janela controlada no servidor.

31.2 — Troca de senha

Decida se revoga todas as sessões, exceto a atual, ou todas sem exceção. Para conta comprometida, revogação ampla é mais segura.

31.3 — Mudança de papel

Se o papel estiver em um JWT de longa duração, a permissão antiga permanece até expirar. Tokens curtos ou consulta de estado reduzem a janela.

31.4 — Usuário excluído ou suspenso

Um token criptograficamente válido pode apontar para conta que já não deve operar. Avalie estado da sessão/conta nas ações sensíveis.

31.5 — Clock skew

Relógios podem divergir. Use pequena tolerância, NTP e tempos em UTC; não transforme tolerância em validade excessiva.

31.6 — Reenvio de e-mail

Vários tokens ativos criam confusão. Invalide anteriores ou aceite apenas o mais recente.

31.7 — Cookie em desenvolvimento

Secure, domínios, portas e SameSite podem se comportar diferente em localhost. Mantenha configuração explícita por ambiente sem enfraquecer produção.

31.8 — Conta criada por OAuth sem senha

passwordHash pode ser nulo. O fluxo para definir senha exige prova adicional e não pode assumir que toda conta possui credencial local.

31.9 — E-mail com caixa e Unicode

Normalização ingênua pode fundir identidades distintas. Defina política e preserve o valor de apresentação quando necessário.

31.10 — Recurso compartilhado

Ownership simples deixa de bastar. Modele membros, permissões e remoção de acesso com integridade transacional.

notesRouter.post(

  const note = await createNoteService.execute({

Observe: o userId não vem do body. Ele nasce da credencial verificada. A propriedade da nota é determinada pelo contexto autenticado.
export async function updateNote(input: {
  noteId: string;
  userId: string;
  title?: string;
  content?: string;
}) {
  const note = await notesRepository.findOwnedById({
    noteId: input.noteId,
    userId: input.userId,
  });

  if (!note) {
    throw new NotFoundError("Note");
  }

  return notesRepository.update(note.id, {
    title: input.title,
    content: input.content,
  });
}

35 — Mini projeto: API de notas privadas

35.1 — Funcionalidades

  • Cadastro com e-mail e senha.
  • Verificação de e-mail.
  • Login com access token curto.
  • Refresh token rotativo em cookie HttpOnly.
  • Logout da sessão atual e de todas as sessões.
  • Recuperação e redefinição de senha.
  • CRUD de notas privadas.
  • Rota administrativa limitada para listar usuários, sem expor credenciais.

35.3 — Regras obrigatórias

  • Senha nunca é persistida ou registrada em logs.
  • Refresh token é rotacionado e armazenado como hash.
  • Usuário só acessa as próprias notas.
  • IDs recebidos do cliente não definem ownership.
  • Tokens de e-mail e reset são de uso único e expiram.
  • Troca de senha revoga sessões segundo política documentada.
  • Login e recuperação possuem rate limiting.
  • Mensagens não permitem enumerar contas.
  • Cookies possuem configuração segura por ambiente.
  • Testes cobrem sucesso, expiração, revogação, concorrência e acesso cruzado.

36 — Checklist para avançar

  • Sei diferenciar identificação, autenticação, autorização e auditoria.
  • Consigo explicar hash, salt, pepper e work factor.
  • Não uso hash rápido para senhas.
  • Sei comparar sessão opaca e JWT sem declarar um vencedor universal.
  • Entendo que JWT assinado não esconde o payload.
  • Sei por que access token deve ser curto e refresh token precisa de proteção adicional.
  • Consigo explicar rotação, família e reuse detection.
  • Entendo logout local, logout no servidor e logout global.
  • Sei o papel de HttpOnly, Secure e SameSite.
  • Não confundo CORS com CSRF ou autorização.
  • Sei implementar RBAC e ownership separadamente.
  • Não confio em userId enviado pelo cliente.
  • Consigo projetar reset e verificação com tokens aleatórios, hash, expiração e uso único.
  • Sei testar acesso cruzado e concorrência no refresh.

Testes com Vitest e Supertest

Testar não é apenas verificar se uma função “retorna algo”. Nesta fase, você aprenderá a testar comportamentos, regras de negócio, respostas HTTP e casos extremos sem acoplar a suíte a detalhes frágeis.

1

1. Visão geral

Testes são uma forma executável de descrever comportamentos esperados. Um bom teste não prova que o sistema nunca terá defeitos; ele aumenta nossa confiança para alterar o código e detecta regressões importantes cedo.

Ideia principal

Teste bom descreve uma regra observável. “Chama método X” normalmente é detalhe interno; “rejeita horário sobreposto” é comportamento.

Confiança

Permite refatorar com uma rede de segurança.

Documentação

Mostra exemplos executáveis do contrato.

Design

Expõe acoplamentos e dependências escondidas.

2

2. Tipos de teste

A classificação depende do tamanho da parte exercitada, das dependências reais usadas e da fronteira observada.

Unitário

Escopo pequeno, dependências controladas, feedback muito rápido.

Integração

Partes reais colaborando, mais confiança e custo maior.

HTTP

Contrato externo observado como um cliente observaria.

2

2.1 Teste unitário

Exercita uma unidade pequena — normalmente uma função, classe ou service — com dependências controladas. Deve ser rápido, determinístico e explicar uma regra de negócio.

import { describe, expect, it } from "vitest";
import { calculateTotal } from "./calculate-total";

describe("calculateTotal", () => {
  it("applies the percentage discount", () => {
    const total = calculateTotal({ subtotal: 200, discount: 0.1 });

    expect(total).toBe(180);
  });
});

Um teste unitário não precisa obrigatoriamente testar uma única função, mas sua fronteira deve ser pequena e clara.

2

2.2 Teste de integração

Verifica a colaboração entre partes reais: service + repository, rota + middleware + controller, ou aplicação + banco isolado.

  • Confirma que contratos entre camadas combinam.
  • Detecta configuração incorreta, serialização e queries inválidas.
  • É mais lento; por isso deve ser seletivo e isolado.
2

2.3 Teste HTTP

Envia uma requisição contra a aplicação Express e observa o contrato HTTP: status, headers e body. Com Supertest, não é necessário abrir uma porta real para cada teste.

const response = await request(app)
  .post("/tasks")
  .send({ title: "Estudar testes" })
  .expect(201);

expect(response.body).toEqual({
  id: expect.any(String),
  title: "Estudar testes",
  completed: false,
});

3. Arrange, Act, Assert

AAA separa preparação, ação e verificação. A estrutura reduz ruído e ajuda a localizar a causa de uma falha.

it("should reject overlapping appointments", async () => {
  // Arrange
  const repository = new InMemoryAppointmentsRepository();
  const service = new CreateAppointmentService(repository);
  await repository.create({ startsAt: new Date("2026-08-10T10:00:00Z") });

  // Act
  const action = service.execute({
    startsAt: new Date("2026-08-10T10:30:00Z"),
  });

  // Assert
  await expect(action).rejects.toMatchObject({
    code: "APPOINTMENT_CONFLICT",
  });
});

Perguntas para montar o teste

  1. Quais dados precisam existir antes da ação?
  2. Qual é a única ação principal?
  3. Qual resultado prova que a regra foi respeitada?
  4. No erro, o que não pode ter acontecido?

4. Configurando Vitest

Vitest fornece runner, assertions, hooks, spies, mocks e cobertura. Em backend TypeScript, o ambiente normalmente é Node.

npm install -D vitest supertest @types/supertest
// vitest.config.ts
import { defineConfig } from "vitest/config";

export default defineConfig({
  test: {
    environment: "node",
    include: ["src/**/*.test.ts"],
    clearMocks: true,
  },
});
{
  "scripts": {
    "test": "vitest",
    "test:run": "vitest run",
    "test:coverage": "vitest run --coverage"
  }
}

5. Testabilidade e desenho

Código testável costuma ter dependências explícitas, regras separadas de transporte HTTP e poucos efeitos globais escondidos.

Mais testável

  • app separado de server
  • dependências recebidas no construtor
  • relógio e IDs substituíveis
  • regras em services

Menos testável

  • listen ao importar módulo
  • singleton global mutável
  • Date.now espalhado
  • acesso direto à rede dentro da regra
// app.ts: pode ser importado pelo teste
export const app = express();

// server.ts: abre a porta somente em produção/desenvolvimento
import { app } from "./app";
app.listen(3333);
6

6. Mocks, stubs, spies e fakes

Test doubles substituem colaboradores. O nome importa menos que a intenção, mas conhecer as diferenças evita testes frágeis.

Stub: devolve respostas pré-definidas.

Spy: registra chamadas para inspeção.

Mock: substituição configurável com expectativas.

Fake: implementação simplificada e funcional.

6

6.1 Fake repository

Um fake possui uma implementação simplificada, porém funcional. Um repositório em memória permite exercitar regras sem acoplar o teste ao banco.

type Task = { id: string; title: string; completed: boolean };

export class InMemoryTasksRepository {
  public items: Task[] = [];

  async create(task: Task): Promise<Task> {
    this.items.push(task);
    return task;
  }

  async findByTitle(title: string): Promise<Task | null> {
    return this.items.find((task) => task.title === title) ?? null;
  }
}

O fake deve respeitar o contrato importante do repositório real. Se o banco é case-insensitive e o fake não é, os testes podem mentir.

6

6.2 Quando usar mocks — e quando evitar

Mocks são úteis em fronteiras caras, lentas ou externas. Evite mockar cada detalhe interno, pois isso acopla o teste à implementação.

Use com intenção

  • gateway de pagamento
  • e-mail/SMS
  • API de terceiro
  • tempo, UUID e fila

Evite por padrão

  • mockar o próprio objeto testado
  • mockar DTO simples
  • verificar sequência interna irrelevante
  • recriar o framework inteiro
const mailer = {
  send: vi.fn().mockResolvedValue(undefined),
};

await service.execute(input);

expect(mailer.send).toHaveBeenCalledWith(
  expect.objectContaining({ to: input.email }),
);

7. Testando regras de negócio

Services são bons alvos para testes unitários porque concentram decisões como conflito, permissões e transições de estado.

describe("CreateTaskService", () => {
  it("rejects duplicated titles", async () => {
    const repository = new InMemoryTasksRepository();
    const service = new CreateTaskService(repository);

    await service.execute({ title: "Estudar Vitest" });

    await expect(
      service.execute({ title: "Estudar Vitest" }),
    ).rejects.toMatchObject({ code: "TASK_CONFLICT" });

    expect(repository.items).toHaveLength(1);
  });
});
Assertion negativa útil

Além de verificar o erro, confirme que nenhum segundo registro foi persistido.

8. Testando Express com Supertest

Supertest cria requisições HTTP contra o objeto app. Isso exercita middlewares, validação, rotas, controllers e o tratamento de erros.

import request from "supertest";
import { describe, expect, it } from "vitest";
import { app } from "../app";

describe("POST /tasks", () => {
  it("creates a task", async () => {
    const response = await request(app)
      .post("/tasks")
      .send({ title: "Testar a API" });

    expect(response.status).toBe(201);
    expect(response.headers["content-type"]).toContain("application/json");
    expect(response.body).toMatchObject({
      title: "Testar a API",
      completed: false,
    });
  });
});

Prefira assertions sobre o contrato público. Evite snapshots gigantes de respostas dinâmicas.

9. Banco e estado isolados

Cada teste precisa começar de um estado conhecido. Compartilhar dados entre testes cria dependência de ordem e falhas intermitentes.

  1. Use banco exclusivo para testes.
  2. Aplique migrations antes da suíte.
  3. Limpe dados entre testes ou use transações revertidas.
  4. Nunca aponte testes destrutivos para produção.
  5. Gere dados dentro do próprio teste ou por builders explícitos.
Proteção recomendada

Falhe imediatamente se NODE_ENV !== "test" ou se a URL não identificar claramente o banco de teste.

9

9.1 Hooks e ciclo de vida

beforeAll, beforeEach, afterEach e afterAll organizam recursos. Use o menor escopo possível e não esconda preparação essencial demais.

beforeAll(async () => {
  await database.connect();
  await database.migrate();
});

beforeEach(async () => {
  await database.clear();
});

afterAll(async () => {
  await database.disconnect();
});

Se o setup contém a história principal do cenário, deixe-o no próprio teste. Hooks excessivos tornam o teste difícil de ler.

10. O que testar

Priorize comportamentos que importam: caminho feliz, validação, conflito, inexistência, autorização e limites.

Sucesso201 e recurso persistido
Validação422 e formato previsível
Conflito409 e nenhuma gravação extra
Inexistente404 sem detalhes internos
Autorização401/403 conforme contexto
Limitesvazio, máximo, datas e paginação
10

10.1 Edge cases

Edge cases vivem nos limites: texto vazio, datas iguais, paginação extrema, duplicidade, concorrência e formatos inesperados.

  • Título com apenas espaços.
  • Mesmo título com caixa diferente.
  • IDs bem formados, porém inexistentes.
  • page = 0, negativa ou maior que a última.
  • Duas requisições concorrentes criando o mesmo recurso.
  • Datas exatamente no limite de uma janela.
  • Body ausente, JSON inválido ou content-type incorreto.

Nem todo edge case merece um teste isolado. Escolha os que podem quebrar uma regra, contrato ou segurança.

11

11. Testando erros sem vazar detalhes

O teste deve confirmar o contrato público do erro, não a stack trace nem a mensagem interna de uma biblioteca.

const response = await request(app)
  .get("/tasks/not-found")
  .expect(404);

expect(response.body).toEqual({
  error: {
    code: "TASK_NOT_FOUND",
    message: "Tarefa não encontrada.",
  },
});

expect(response.body.error).not.toHaveProperty("stack");
12

12. Cobertura sem obsessão

Cobertura indica quais linhas ou ramos foram executados. Ela não mede qualidade das assertions nem relevância dos cenários.

O que ela ajuda a revelar

  • ramos nunca exercitados
  • tratamento de erro esquecido
  • arquivos sem testes

O que ela não garante

  • assertions corretas
  • cenários relevantes
  • ausência de regressões
“100% executado” não significa “100% verificado”.
13

13. Anti‑patterns

Testes podem passar e ainda assim oferecer pouca confiança. Reconhecer sinais de fragilidade é parte da habilidade.

  • Teste sem assertion: só confirma que não lançou erro.
  • Teste de implementação: quebra ao refatorar sem mudança de comportamento.
  • Dados compartilhados: a ordem altera o resultado.
  • Sleep arbitrário: gera lentidão e flakiness.
  • Mock excessivo: todos os colaboradores são inventados e a integração real nunca é exercitada.
  • Um teste faz tudo: falha difícil de diagnosticar.

14. Organização dos arquivos

Uma organização previsível facilita descobrir testes, compartilhar builders e separar testes unitários de integração.

src/
├── app.ts
├── server.ts
├── modules/
│   └── tasks/
│       ├── create-task.service.ts
│       ├── create-task.service.test.ts
│       ├── task.routes.ts
│       └── task.repository.ts
└── tests/
    ├── factories/
    ├── repositories/
    ├── setup.ts
    └── tasks.http.test.ts

Colocar unitários perto da implementação facilita manutenção. Testes HTTP e infraestrutura podem viver em uma pasta central.

15. Mini projeto: API de tarefas

A API será testada em sucesso, validação, conflito e recurso inexistente, cobrindo service e contrato HTTP.

Contrato

POST /tasks, GET /tasks/:taskId e PATCH /tasks/:taskId.

Regras

  • title é obrigatório e possui tamanho mínimo.
  • não pode haver duas tarefas ativas com o mesmo título normalizado.
  • recurso inexistente retorna 404.
  • conflito retorna 409.
  • erros seguem o formato padronizado da Fase 7.

Suíte mínima

  1. service cria tarefa.
  2. service rejeita duplicidade.
  3. POST retorna 201.
  4. POST inválido retorna 422.
  5. POST duplicado retorna 409.
  6. GET inexistente retorna 404.
describe("Tasks HTTP", () => {
  it("returns 422 when title is missing", async () => {
    const response = await request(app)
      .post("/tasks")
      .send({})
      .expect(422);

    expect(response.body.error.code).toBe("VALIDATION_ERROR");
  });

  it("returns 409 for a duplicated active task", async () => {
    await request(app).post("/tasks").send({ title: "Ler" }).expect(201);

    const response = await request(app)
      .post("/tasks")
      .send({ title: "  ler  " })
      .expect(409);

    expect(response.body.error.code).toBe("TASK_CONFLICT");
  });
});

16. Exercícios orientados

Os exercícios começam pela intenção e só depois pedem implementação. A pergunta central é: qual evidência prova o comportamento?

Exercício 1 — intenção

Escreva o nome dos testes antes do código. Cada nome deve terminar uma frase do tipo: “deve…”

Exercício 2 — matriz

Para cada endpoint, liste sucesso, validação, autorização, inexistência e conflito. Marque o que realmente se aplica.

Exercício 3 — fake fiel

Implemente um repository em memória que normalize títulos da mesma forma que o banco real.

Exercício 4 — teste resistente

Refatore o service sem alterar comportamento. O teste deveria continuar passando sem mudanças?

Exercício 5 — falha intencional

Introduza um bug e confirme que pelo menos um teste relevante falha pelo motivo correto.

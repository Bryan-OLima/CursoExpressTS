Fase 7

Tratamento de erros no Express 5

O objetivo não é “impedir que erros aconteçam”. É construir um caminho controlado para transformar falhas conhecidas em respostas úteis, registrar falhas inesperadas e preservar os detalhes internos que nunca deveriam chegar ao cliente.

1. O modelo mental

Uma exceção nasce em algum ponto da aplicação. Ela deve subir até uma fronteira capaz de decidir três coisas: o que responder, o que registrar e se o processo continua saudável.

Route→Controller→Service→Repository→erro→Error handler→resposta + log
Separação importante: a classe de erro representa o problema; o middleware traduz esse problema para HTTP. Assim, a regra de negócio não precisa conhecer Response.

2. Erro operacional versus erro de programação

Nem toda falha significa que o código está quebrado. Algumas situações são resultados esperados do funcionamento de uma API.

Erro operacional

A aplicação está funcionando, mas não consegue concluir aquela operação: recurso inexistente, horário ocupado, credencial inválida, timeout de uma dependência ou entrada inválida.

Erro de programação

Existe um defeito ou uma suposição falsa no código: acessar propriedade de undefined, esquecer um await, quebrar uma invariável ou enviar duas respostas.

SituaçãoCategoria típicaResposta possível
Agendamento não existeOperacional404 APPOINTMENT_NOT_FOUND
Horário já ocupadoOperacional / domínio409 APPOINTMENT_CONFLICT

A categoria não é decidida apenas pela classe nativa. Um TypeError normalmente indica bug; um Error criado por você pode representar uma regra esperada. O significado vem do contexto e do contrato.

3. throw, retorno e rejeição

throw interrompe o fluxo atual. Em uma função síncrona, a exceção sobe pela pilha até um catch. Em uma função async, lançar um valor faz a Promise retornada ser rejeitada.

function ensurePositive(value: number): void {
  if (value <= 0) {
    throw new Error("O valor deve ser positivo.");
  }
}

async function loadAppointment(id: string) {
  const appointment = await repository.findById(id);

  if (!appointment) {
    // A Promise retornada por esta função será rejeitada.
    throw new AppointmentNotFoundError(id);
  }

  return appointment;
}

Você pode lançar qualquer valor, mas não deveria

throw "falhou";       // JavaScript permite
throw { code: 123 }; // JavaScript também permite
throw new Error("falhou"); // forma recomendada

Lançar instâncias de Error preserva name, message, stack e pode preservar uma causa. O handler ainda precisa ser defensivo porque bibliotecas ou código legado podem rejeitar Promises com valores desconhecidos.

4. Encaminhamento automático no Express 5

Handlers e middlewares que retornam uma Promise têm seus erros lançados e rejeições encaminhados ao fluxo de erro do router.

app.get("/appointments/:id", async (request, response) => {
  const appointment = await appointmentService.findById(
    request.params.id,
  );

  return response.status(200).json(appointment);
});

Se findById lançar, você não precisa escrever um try/catch apenas para chamar next(error). O Express 5 trata a rejeição como encaminhamento ao próximo middleware de erro.

Express 4 — padrão frequente

try/catch, wrappers como asyncHandler ou bibliotecas auxiliares eram usados para encaminhar rejeições.

Express 5

Quando o callback retorna a Promise, uma rejeição é encaminhada automaticamente. Isso reduz código repetitivo, mas não substitui decisões locais de recuperação.

Armadilha: iniciar uma Promise sem retorná-la nem aguardá-la quebra o vínculo com o handler. Use await ou return.
app.post("/jobs", async (request, response) => {
  // Ruim: tarefa "solta"; uma rejeição pode escapar do ciclo da requisição.
  queue.publish(request.body);

  return response.status(202).end();
});

5. Middleware de erro

O Express identifica um middleware de erro pela assinatura com quatro parâmetros: (error, request, response, next). Mesmo que next não seja usado no caminho principal, ele faz parte do contrato.

import type { ErrorRequestHandler } from "express";

export const errorHandler: ErrorRequestHandler = (
  error,
  request,
  response,
  next,
) => {
  console.error(error);

  return response.status(500).json({
    error: {
      code: "INTERNAL_SERVER_ERROR",
      message: "Ocorreu um erro inesperado.",
    },
  });
};
O parâmetro next não é decorativo: ele será necessário quando os headers já tiverem sido enviados, pois nesse cenário o handler deve delegar ao tratamento padrão do Express.

6. A ordem de registro

O pipeline é executado na ordem em que foi registrado. O handler de erros deve aparecer depois das rotas e dos middlewares que podem falhar.

app.use(express.json());
app.use(requestId);
app.use(routes);

// 404 é criado depois que nenhuma rota respondeu.
app.use(notFoundHandler);

// O error handler é a última fronteira HTTP.
app.use(errorHandler);

Um 404 por rota inexistente não é, por si só, uma exceção automática. Significa apenas que o pipeline terminou sem encontrar um handler que enviasse a resposta. Por isso, é comum haver um middleware final que cria um erro de 404.

7. Erro depois de headers enviados

Depois que a resposta começou a ser enviada, talvez já não seja possível trocar status, headers ou corpo por um JSON de erro. Tentar fazer isso costuma gerar o clássico problema “Cannot set headers after they are sent”.

export const errorHandler: ErrorRequestHandler = (
  error,
  request,
  response,
  next,
) => {
  if (response.headersSent) {
    return next(error);
  }

  // Ainda é seguro construir uma resposta HTTP.
  return response.status(500).json({
    error: {
      code: "INTERNAL_SERVER_ERROR",
      message: "Ocorreu um erro inesperado.",
    },
  });
};
Em streaming e downloads, uma falha pode acontecer depois de parte do corpo ter sido transmitida. Nesse caso não existe uma resposta JSON limpa a ser “substituída”; a conexão pode precisar ser encerrada.

8. Criando um erro de aplicação

Uma classe base centraliza os dados necessários para classificar uma falha, sem acoplar services ao Express.

export type ErrorDetails = ReadonlyArray<{
  field?: string;
  message: string;
}>;

type AppErrorOptions = {
  code: string;
  message: string;
  statusCode: number;
  details?: ErrorDetails;
  cause?: unknown;
};

export class AppError extends Error {
  readonly code: string;
  readonly statusCode: number;
  readonly details: ErrorDetails;
  readonly isOperational = true;

  constructor(options: AppErrorOptions) {
    super(options.message, { cause: options.cause });
    this.name = new.target.name;
    this.code = options.code;
    this.statusCode = options.statusCode;
    this.details = options.details ?? [];
  }
}

Erros específicos

export class AppointmentNotFoundError extends AppError {
  constructor(appointmentId: string) {
    super({
      code: "APPOINTMENT_NOT_FOUND",
      message: "Agendamento não encontrado.",
      statusCode: 404,
      details: [{ field: "appointmentId", message: appointmentId }],
    });
  }
}

export class AppointmentConflictError extends AppError {
  constructor() {
    super({
      code: "APPOINTMENT_CONFLICT",
      message: "Este horário não está disponível.",
      statusCode: 409,
    });
  }
}
Critério útil: crie classes específicas quando o nome carregar significado para regras, testes ou mapeamento. Não crie uma classe diferente apenas para trocar uma frase.

9. Formato previsível de resposta

{
  "error": {
    "code": "APPOINTMENT_CONFLICT",
    "message": "Este horário não está disponível.",
    "details": [],
    "requestId": "req_01J..."
  }
}

code

Identificador estável para clientes, testes, tradução e métricas. Não dependa da mensagem humana.

message

Explicação segura e legível. Pode mudar; o cliente não deveria tomar decisões por ela.

details

Informação estruturada adicional, especialmente útil em validação de campos.

requestId

Permite correlacionar a resposta do cliente com logs internos.

status HTTP

Comunica a categoria do resultado no protocolo; não substitui o código de negócio.

sem stack

Stack trace é diagnóstico interno e não faz parte do contrato público em produção.

11. Stack trace por ambiente e segurança

Em desenvolvimento, stack traces aceleram o diagnóstico. Em produção, eles podem revelar caminhos de arquivos, nomes internos, consultas, dependências e partes da implementação.

const isDevelopment = process.env.NODE_ENV === "development";

const body = {
  error: {
    code: publicError.code,
    message: publicError.message,
    details: publicError.details,
    requestId: response.locals.requestId,
    ...(isDevelopment && error instanceof Error
      ? { stack: error.stack }
      : {}),
  },
};

O log interno pode conter stack e causa; a resposta pública deve conter apenas dados permitidos. Também evite registrar tokens, senhas, cookies, dados pessoais e bodies completos sem uma política explícita de redaction.

12. Logs, contexto e request ID

Uma mensagem genérica protege o cliente, mas alguém precisa conseguir investigar a causa real. O log deve ligar erro, requisição e dependências sem expor segredos.

logger.error({
  event: "request_failed",
  requestId: response.locals.requestId,
  method: request.method,
  path: request.originalUrl,
  statusCode,
  errorName: normalizedError.name,
  errorMessage: normalizedError.message,
  stack: normalizedError.stack,
  cause: normalizedError.cause,
});

Erros esperados podem ser registrados como warn ou até não gerar log em algumas rotas. Bugs e indisponibilidades merecem error, métricas e possivelmente alertas. “Logar tudo como erro” cria ruído e esconde o que realmente importa.

13. Implementação completa do handler

import type { ErrorRequestHandler } from "express";
import { AppError } from "../errors/app-error.js";

function normalizeError(value: unknown): Error {
  if (value instanceof Error) {
    return value;
  }

  return new Error("Um valor que não é Error foi lançado.", {
    cause: value,
  });
}

export const errorHandler: ErrorRequestHandler = (
  thrownValue,
  request,
  response,
  next,
) => {
  if (response.headersSent) {
    return next(thrownValue);
  }

  const error = normalizeError(thrownValue);
  const isAppError = error instanceof AppError;

  const statusCode = isAppError ? error.statusCode : 500;
  const code = isAppError
    ? error.code
    : "INTERNAL_SERVER_ERROR";
  const message = isAppError
    ? error.message
    : "Ocorreu um erro inesperado.";
  const details = isAppError ? error.details : [];

  console.error({
    requestId: response.locals.requestId,
    method: request.method,
    path: request.originalUrl,
    statusCode,
    error,
  });

  return response.status(statusCode).json({
    error: {
      code,
      message,
      details,
      requestId: response.locals.requestId,
      ...(process.env.NODE_ENV === "development"
        ? { stack: error.stack }
        : {}),
    },
  });
};
Por que normalizar? O parâmetro chega como desconhecido do ponto de vista de segurança. Ao converter qualquer valor lançado em uma instância de Error, o restante do handler trabalha com uma forma mínima consistente.

14. Edge cases

Edge cases são situações nos limites do fluxo normal. Eles importam porque erros geralmente aparecem exatamente quando estado, tempo, rede ou ciclo de resposta saem do caminho feliz.

14.1 Erro depois de headers enviados

Verifique response.headersSent e delegue com next(error). Não tente montar outro JSON.

14.2 Erro do banco de dados

Não exponha mensagens, SQL, nome de tabela ou código específico do driver. Traduza erros conhecidos — como violação de unicidade — para um erro de domínio; mantenha indisponibilidade como falha de infraestrutura e preserve a causa no log.

14.3 Valor lançado que não é Error

Normalize o valor. O handler deve tolerar strings, objetos e rejeições estranhas sem quebrar enquanto tenta tratar o primeiro erro.

14.4 Promise rejeitada

No Express 5, uma Promise retornada pelo handler é encaminhada. Promises “soltas”, callbacks de APIs antigas e eventos assíncronos podem exigir tratamento explícito.

14.5 Timeout de serviço externo

Use um timeout real, cancele a operação quando possível e traduza o resultado. Sem timeout, uma requisição pode ficar presa até limites externos encerrarem a conexão.

14.6 O cliente cancelou a requisição

O trabalho interno pode continuar mesmo depois que o cliente desconecta. Considere AbortSignal para propagar cancelamento a fetch, consultas e tarefas que suportem aborto.

14.7 Falha dentro do próprio error handler

Mantenha o handler simples. Serialização circular, logger quebrado ou acesso inseguro a propriedades pode produzir um segundo erro e esconder o original.

14.8 Duas operações concorrentes passam na validação

Checar “horário livre” antes de inserir não basta contra corrida. Garanta a regra no banco com constraint/transação e traduza a violação para 409.

14.9 JSON inválido antes das rotas

express.json() pode gerar erro de parsing. O handler deve reconhecer a falha e devolver 400, em vez de um 500 genérico.

14.10 Processo em estado potencialmente corrompido

Capturar uma exceção não garante que o processo está saudável. Falhas fatais ou invariáveis quebradas podem justificar encerramento gracioso e reinício pelo orquestrador, não uma tentativa cega de continuar.

15. Como testar erros

Teste unitário

Confirme que o service lança o erro correto quando a regra falha, sem HTTP envolvido.

Teste de integração

Faça uma requisição real à aplicação e verifique status, código, detalhes e ausência de informações sensíveis.

it("returns 409 when the time is already booked", async () => {
  await repository.create({
    professionalId: "pro_1",
    startsAt: "2026-07-15T14:00:00.000Z",
  });

  const response = await request(app)
    .post("/appointments")
    .send({
      professionalId: "pro_1",
      startsAt: "2026-07-15T14:00:00.000Z",
    });

  expect(response.status).toBe(409);
  expect(response.body).toMatchObject({
    error: {
      code: "APPOINTMENT_CONFLICT",
      message: "Este horário não está disponível.",
      details: [],
    },
  });
  expect(response.body.error.stack).toBeUndefined();
});

16. Mini projeto: API resiliente de agendamentos

Expanda a API das fases anteriores com um catálogo explícito de erros.

Requisitos funcionais

  • 404 para agendamento ou profissional inexistente.
  • 409 para colisão de horário.
  • 422 para dados semanticamente inválidos.
  • 401/403 em operações protegidas.

Requisitos técnicos

  • Classe base de erro.
  • Handler global e middleware de 404.
  • Request ID em resposta e log.
  • Stack apenas em desenvolvimento.
  • Teste de cada contrato de erro.

17. Checklist de domínio

  • Consigo explicar por que um 404 esperado não é igual a um bug.
  • Sei como throw se transforma em rejeição dentro de uma função async.
  • Entendo quando o Express 5 encaminha automaticamente uma rejeição.
  • Sei por que o middleware de erro tem quatro parâmetros e fica no final.
  • Não exponho stack trace, SQL, tokens ou mensagens internas em produção.
  • Uso códigos estáveis de erro além da mensagem humana.

Fase 9

Banco de dados, SQL e Prisma

PostgreSQLPrisma ORMExpress 5TypeScriptAPI financeira

Persistência não é apenas trocar um array por uma chamada ao ORM. Um banco relacional aplica regras, coordena acessos concorrentes, mantém relações e precisa continuar correto quando duas requisições chegam quase ao mesmo tempo.

Objetivo: compreender o modelo relacional antes do Prisma; modelar dados e restrições; escrever e interpretar SQL; usar migrations e Prisma Client; evitar N+1; paginar; trabalhar com transações; isolar testes; e construir operações financeiras atômicas.
requisição validada ↓ service / regra de negócio ↓ repository ↓ Prisma Client ↓ SQL + transação + constraints ↓ PostgreSQL

1 — Modelo mental

1. A aplicação solicita uma operação; o banco é um sistema independente.
2. O schema define estruturas e invariantes que valem para qualquer cliente.
3. O ORM traduz chamadas de alto nível para consultas ao banco.
4. Uma transação agrupa mudanças que devem acontecer juntas.
5. Concorrência significa que uma verificação pode ficar desatualizada antes da escrita.
Regra central: validação no service melhora a mensagem; constraint no banco protege a verdade. Em dados importantes, você normalmente precisa das duas.

2 — O que é um banco relacional

Um banco relacional organiza dados em relações — normalmente apresentadas como tabelas — e permite conectar registros por chaves. O banco não é um arquivo JSON sofisticado: ele possui linguagem de consulta, catálogo de schema, planejador, índices, transações, controle de concorrência, permissões e mecanismos de durabilidade.

Aplicação

Express, services e repositories expressam casos de uso e políticas.

Banco

Armazena dados, executa consultas e garante restrições mesmo diante de concorrência.

4 — Tipos de dados

O tipo influencia validação, armazenamento, ordenação, operações permitidas e representação no cliente. Use tipos que representem o domínio, não apenas os que parecem convenientes.

  • text/varchar para texto.
  • integer/bigint para inteiros.
  • numeric para precisão decimal controlada.
  • boolean para verdadeiro/falso.
  • date para data civil e timestamptz para instante.
  • uuid para identificadores UUID.
  • jsonb para estruturas que realmente não pedem modelagem relacional.
JavaScript e BIGINT: valores acima de Number.MAX_SAFE_INTEGER não são seguros em number. Prisma pode devolver bigint, que também não é serializado por JSON.stringify sem conversão.

5 — Chave primária e chave estrangeira

5.1 — Primary key

Identifica cada linha de maneira única e não nula. Deve ser estável. E-mail pode mudar; um identificador técnico costuma ser melhor.

5.2 — Foreign key

Declara que um valor referencia uma linha existente em outra tabela, mantendo integridade referencial.

CREATE TABLE categories (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name VARCHAR(80) NOT NULL
);

ON DELETE precisa ser uma decisão de domínio: CASCADE, RESTRICT, SET NULL ou outra estratégia não são intercambiáveis.

6 — Relacionamentos

Um para um

Um usuário possui no máximo um perfil. A FK deve ser única.

Um para muitos

Um usuário possui muitas contas; cada conta pertence a um usuário.

Muitos para muitos

Muitos lançamentos podem ter muitas tags, usando tabela de junção.

Autorrelacionamento

Uma categoria pode ter categoria pai, ou um usuário pode indicar outro.

Pergunta de modelagem: “onde fica a foreign key?” Em um-para-muitos, ela normalmente fica no lado “muitos”.

ALTER TABLE categories

ALTER TABLE transactions
ADD CONSTRAINT transaction_amount_not_zero CHECK (amount_cents <> 0);

Uma constraint composta é essencial quando a unicidade depende do contexto. “Categoria Alimentação é única” costuma significar única para aquele usuário, não no sistema inteiro.

8 — Índices

Índice é uma estrutura auxiliar para localizar linhas sem percorrer a tabela inteira. Ele pode acelerar leitura, filtros, ordenação e joins, mas aumenta armazenamento e custo de escrita.

CREATE INDEX transactions_account_occurred_at_idx
ON transactions (account_id, occurred_at DESC);
  • A ordem das colunas em índice composto importa.
  • Índice deve refletir consultas reais.
  • Foreign key não implica automaticamente o índice ideal em todas as situações.
  • UNIQUE normalmente é implementado por índice único, mas seu significado principal é integridade.
  • Use EXPLAIN/EXPLAIN ANALYZE antes de “otimizar no escuro”.

9 — SQL essencial

INSERT INTO accounts (id, user_id, name)
VALUES ($1, $2, $3);

SELECT id, name, balance_cents
FROM accounts
WHERE user_id = $1
ORDER BY created_at DESC
LIMIT $2 OFFSET $3;

UPDATE accounts
SET name = $1, updated_at = NOW()
WHERE id = $2 AND user_id = $3;

DELETE FROM categories
WHERE id = $1 AND user_id = $2;
Nunca concatene entrada externa em SQL. Use parâmetros ou o query builder do ORM. Tipagem não impede SQL injection em uma raw query construída incorretamente.

10 — JOINs

JOIN combina linhas relacionadas. Entender JOIN ajuda a interpretar include, depurar consultas e reconhecer multiplicação de linhas.

SELECT
  t.id,
  t.amount_cents,
  a.name AS account_name,
  c.name AS category_name
FROM transactions t
JOIN accounts a ON a.id = t.account_id
LEFT JOIN categories c ON c.id = t.category_id
WHERE a.user_id = $1;

INNER JOIN exige correspondência. LEFT JOIN preserva a linha da esquerda mesmo sem relação. Uma consulta com vários relacionamentos “muitos” pode gerar um produto de linhas e exigir outra estratégia.

11 — Transações e ACID

Uma transação define uma unidade de trabalho. Ou todas as operações confirmam com COMMIT, ou as alterações são desfeitas com ROLLBACK.

Atomicidade

O conjunto acontece por inteiro ou não acontece.

Consistência

Regras válidas continuam válidas ao fim.

Isolamento

Operações concorrentes não devem produzir anomalias proibidas.

Durabilidade

Depois do commit, o resultado deve sobreviver a falhas previstas.

BEGIN;
UPDATE accounts SET balance_cents = balance_cents - 5000 WHERE id = $1;
UPDATE accounts SET balance_cents = balance_cents + 5000 WHERE id = $2;
INSERT INTO transfers (...) VALUES (...);
COMMIT;

12 — Concorrência e isolamento

Duas requisições podem ler o mesmo saldo e ambas concluir que há dinheiro suficiente. O bug não aparece em testes sequenciais simples.

// Abordagem vulnerável: read → decide → write
const account = await repository.findById(sourceId);
if (account.balanceCents < amountCents) throw new InsufficientFundsError();
await repository.updateBalance(sourceId, account.balanceCents - amountCents);

Estratégias possíveis incluem atualização condicional, lock de linha, nível de isolamento mais forte, constraint e retry de falhas transitórias. A escolha depende do banco e do caso de uso.

UPDATE accounts
SET balance_cents = balance_cents - $1
WHERE id = $2 AND user_id = $3 AND balance_cents >= $1;
-- 0 linhas alteradas: inexistente, sem acesso ou saldo insuficiente.
Transação não é sinônimo automático de ausência de race condition. O nível de isolamento e a forma da consulta continuam importantes.

13 — Paginação

13.1 — Offset

ORDER BY occurred_at DESC, id DESC
LIMIT 20 OFFSET 40;

É simples e permite pular páginas, mas offsets altos custam mais e inserções concorrentes podem deslocar resultados.

13.2 — Cursor

WHERE (occurred_at, id) < ($cursorDate, $cursorId)
ORDER BY occurred_at DESC, id DESC
LIMIT 20;

É mais estável para feeds e grandes volumes. A ordenação precisa ser determinística; um campo não único costuma exigir desempate pelo ID.

14 — Problema N+1

const accounts = await prisma.account.findMany();
for (const account of accounts) {
  account.transactions = await prisma.transaction.findMany({
    where: { accountId: account.id },
  });
}

Uma consulta carrega N contas e depois executa uma consulta para cada conta: 1 + N. Corrija carregando relações de forma adequada, agregando em lote ou redesenhando a resposta.

Nem todo carregamento em duas consultas é N+1. Uma estratégia em lote pode ser deliberada e eficiente.

15 — O que Prisma resolve — e o que não resolve

Ajuda

  • Client gerado e tipado.
  • Autocompletar de filtros e relações.
  • Mapeamento entre models e banco.
  • Migrations versionadas.
  • Composição de consultas.

Não substitui

  • Modelagem relacional.
  • Índices corretos.
  • Transações conscientes.
  • Autorização.
  • Análise de SQL e performance.

18 — Models e relations

model User {
  id         String     @id @default(uuid()) @db.Uuid
  email      String     @unique
  accounts   Account[]
  categories Category[]
  createdAt  DateTime   @default(now()) @map("created_at")

  @@map("users")
}

model Account {
  id           String        @id @default(uuid()) @db.Uuid
  userId       String        @db.Uuid @map("user_id")
  name         String        @db.VarChar(80)
  balanceCents BigInt        @default(0) @map("balance_cents")
  user         User          @relation(fields: [userId], references: [id], onDelete: Cascade)
  transactions Transaction[]
  createdAt    DateTime      @default(now()) @map("created_at")
  updatedAt    DateTime      @updatedAt @map("updated_at")

  @@unique([userId, name])
  @@index([userId, createdAt])
  @@map("accounts")
}

Campos de relação como user não são necessariamente colunas; userId é o campo escalar que armazena a FK. @map e @@map permitem nomes idiomáticos no TypeScript e no SQL.

19 — Migrations

Migration é uma mudança versionada do schema do banco. Ela deve ser revisada como código, pois pode bloquear tabela, perder dados ou falhar em produção.

npx prisma migrate dev --name add_categories   # desenvolvimento
npx prisma migrate deploy                      # produção/CI
npx prisma migrate status
npx prisma migrate diff ...
db push não substitui migrations em um projeto que precisa de histórico confiável. É útil em protótipos e alguns workflows, mas produção pede mudanças revisáveis e reproduzíveis.

19.1 — Mudança incompatível

Adicionar coluna obrigatória em tabela preenchida exige estratégia: adicionar nullable/default, preencher dados, validar e só depois tornar obrigatória.

20 — Prisma Client e ciclo de vida

import { PrismaClient } from "./generated/prisma/client";

export const prisma = new PrismaClient();

Em um servidor de longa duração, normalmente existe uma instância compartilhada por processo. Criar um client por requisição pode multiplicar pools e conexões. O encerramento gracioso deve parar o servidor e desconectar quando apropriado.

process.on("SIGTERM", async () => {
  await prisma.$disconnect();
});

22 — select e include

select

select: {
  id: true,
  name: true,
  balanceCents: true,
}

Escolhe campos e reduz vazamento acidental.

include

include: {
  transactions: {
    take: 5,
    orderBy: { occurredAt: "desc" },
  },
}

Carrega relações.

Não retorne modelos do banco diretamente por conveniência. Use DTOs/mappers para serializar bigint, ocultar campos e estabilizar o contrato.

23 — Filtros e ordenação

const items = await prisma.transaction.findMany({
  where: {
    account: { userId },
    kind: kind ?? undefined,
    categoryId: categoryId ?? undefined,
    occurredAt: {
      gte: from ?? undefined,
      lte: to ?? undefined,
    },
    description: search
      ? { contains: search, mode: "insensitive" }
      : undefined,
  },
  orderBy: [{ occurredAt: "desc" }, { id: "desc" }],
});

O filtro de ownership deve estar na própria consulta sempre que possível. Buscar por ID e só depois comparar usuário amplia o risco de esquecer a autorização.

24.1 — Offset

skip: (page - 1) * limit,

24.2 — Cursor

const rows = await prisma.transaction.findMany({

Um cursor simples por ID funciona quando a ordenação corresponde ao ID. Para ordenar por data com duplicatas, modele um cursor estável compatível com a ordem.

25 — Transações no Prisma

25.1 — Operações independentes

await prisma.$transaction([
  prisma.category.create({ data: categoryData }),
  prisma.account.update({ where: { id }, data: { updatedAt: new Date() } }),
]);

25.2 — Interactive transaction

await prisma.$transaction(async tx => {
  const source = await tx.account.findFirst({
    where: { id: sourceId, userId },
  });

  if (!source) throw new AccountNotFoundError();
  if (source.balanceCents < amountCents) {
    throw new InsufficientFundsError();
  }

  await tx.account.update({
    where: { id: sourceId },
    data: { balanceCents: { decrement: amountCents } },
  });
  await tx.account.update({
    where: { id: destinationId },
    data: { balanceCents: { increment: amountCents } },
  });
});

Mantenha transações curtas. Não faça HTTP, envio de e-mail ou espera desnecessária dentro delas. Para concorrência forte, complemente com estratégia de isolamento/lock/atualização condicional e retry.

26 — Erros do banco e tradução

O banco pode rejeitar operações por unicidade, foreign key, registro inexistente, conflito de serialização, timeout ou conexão. O repository/service traduz erros esperados para o vocabulário da aplicação, sem expor detalhes do driver.

try {
  await prisma.category.create({ data: { userId, name } });
} catch (error) {
  if (isUniqueConstraintError(error)) {
    throw new CategoryAlreadyExistsError();
  }
  throw error;
}

Evite transformar todo erro do Prisma em 400. Falha de conexão e bug de query continuam sendo erros internos.

27 — Seed

Seed cria dados iniciais ou demonstrativos. Deve ser idempotente quando possível e não pode ser confundido com migration.

// prisma/seed.ts
import { PrismaClient } from "../src/generated/prisma/client";

const prisma = new PrismaClient();

async function main() {
  await prisma.user.upsert({
    where: { email: "demo@example.com" },
    update: {},
    create: {
      email: "demo@example.com",
      accounts: { create: [{ name: "Carteira" }, { name: "Banco" }] },
    },
  });
}

main().finally(() => prisma.$disconnect());

28 — Banco de testes

Testes de integração precisam de banco isolado do desenvolvimento e produção. O schema deve ser migrado e o estado limpo de modo determinístico.

Opções

  • Database dedicado por suíte.
  • Schema PostgreSQL por worker.
  • Container temporário.
  • Transação revertida, quando compatível.

Cuidados

  • Execução paralela.
  • Sequências e IDs.
  • FKs na limpeza.
  • Migrations reais.
  • DATABASE_URL protegida.
Proteção obrigatória: o script de limpeza deve abortar se a URL não indicar explicitamente ambiente de teste.

29 — Repository com Prisma

export class PrismaAccountRepository {
  async findOwnedById(userId: string, accountId: string) {
    return prisma.account.findFirst({
      where: { id: accountId, userId },
    });
  }

  async listByUser(userId: string) {
    return prisma.account.findMany({
      where: { userId },
      select: { id: true, name: true, balanceCents: true },
      orderBy: { createdAt: "asc" },
    });
  }
}

O repository encapsula persistência, mas não precisa esconder toda capacidade do Prisma atrás de métodos genéricos como findAll. Métodos orientados ao caso de uso preservam intenção.

30 — Modelando dinheiro

Não use ponto flutuante para valores monetários que precisam de exatidão.

// R$ 12,34
const amountCents = 1234n;
  • Centavos inteiros são simples para uma moeda com duas casas.
  • Decimal/numeric é melhor quando a escala varia ou cálculos exigem decimais.
  • Valor zero deve ser rejeitado.
  • Defina convenção de sinal: receita positiva e despesa negativa, ou tipo + valor sempre positivo.
  • Moeda deve ser explícita quando o sistema é multimoeda.
function toMoneyResponse(value: bigint) {
  return { amountCents: value.toString() };
}

31 — Isolamento por usuário

const transaction = await prisma.transaction.findFirst({
  where: {
    id: transactionId,
    account: { userId: authenticatedUserId },
  },
});

Não confie em userId vindo do body. Use a identidade autenticada. Para evitar enumeração, muitos sistemas respondem 404 quando o recurso não existe ou pertence a outro usuário.

Relação válida não prova autorização. Uma categoria existente pode pertencer a outro usuário; valide o vínculo no mesmo contexto.

32 — Erros comuns

ORM como caixa-preta

Sem SQL, índices e transações, bugs ficam invisíveis.

Sem constraints

O service sozinho perde corridas concorrentes.

Client por requisição

Cria pools e pressão de conexão.

include indiscriminado

Respostas enormes e dados privados.

Offset sem ordem

Páginas não determinísticas.

Transação longa

Aumenta lock, contenção e timeout.

Saldo calculado no JavaScript

Read-modify-write vulnerável.

db push em produção

Perde revisão e histórico de mudança.

33 — Edge cases

33.1 — Valor zero

Uma transação financeira com zero costuma não representar movimento. Valide na aplicação e, quando possível, com CHECK.

33.2 — Origem igual ao destino

Transferência para a mesma conta não altera patrimônio e pode criar registros sem sentido.

33.3 — Saldo muda após a leitura

Duas transferências simultâneas podem consumir o mesmo saldo. Use operação atômica e estratégia de concorrência.

33.4 — Categoria de outro usuário

O ID existe, mas não pertence ao usuário autenticado.

33.5 — Conta removida durante a operação

A FK, ordem de locks e comportamento de exclusão precisam ser previstos.

33.6 — Retry e duplicidade

Repetir a requisição após timeout pode duplicar transferência; idempotency key será aprofundada em fase avançada.

33.7 — BigInt no JSON

JSON.stringify(1n) falha. Converta para string ou contrato monetário.

33.8 — Data sem desempate

Várias transações podem ter o mesmo timestamp; ordene também pelo ID.

33.9 — Cursor apagado

Um cursor baseado em registro pode desaparecer. Defina política ou use cursor codificado com valores de ordenação.

33.10 — Migration com dados existentes

Adicionar NOT NULL ou UNIQUE pode falhar por registros antigos.

33.11 — Nome com caixa e espaços

“Mercado”, “ mercado ” e “MERCADO” podem ser duplicatas de negócio. Normalize e defina collation/índice adequado.

33.12 — Falha após commit

A resposta HTTP pode falhar depois do commit. O cliente não sabe se a operação ocorreu; pense em idempotência e consulta posterior.

33.13 — Deadlock

Transferências opostas podem bloquear contas em ordem inversa. Adote ordem consistente de acesso e retry.

33.14 — Testes paralelos

Suites que limpam as mesmas tabelas interferem entre si. Isole bancos/schemas ou serialize conscientemente.

35 — Exemplo integrado: transferência

type TransferInput = {
  userId: string;
  sourceAccountId: string;
  destinationAccountId: string;
  amountCents: bigint;
};

export async function transfer(input: TransferInput) {
  if (input.amountCents <= 0n) throw new InvalidAmountError();
  if (input.sourceAccountId === input.destinationAccountId) {
    throw new SameAccountTransferError();
  }

  return prisma.$transaction(async tx => {
    const accounts = await tx.account.findMany({
      where: {
        userId: input.userId,
        id: { in: [input.sourceAccountId, input.destinationAccountId] },
      },
      select: { id: true },
    });

    if (accounts.length !== 2) throw new AccountNotFoundError();

    const debited = await tx.account.updateMany({
      where: {
        id: input.sourceAccountId,
        userId: input.userId,
        balanceCents: { gte: input.amountCents },
      },
      data: { balanceCents: { decrement: input.amountCents } },
    });

    if (debited.count !== 1) throw new InsufficientFundsError();

    await tx.account.update({
      where: { id: input.destinationAccountId },
      data: { balanceCents: { increment: input.amountCents } },
    });

    return tx.transfer.create({
      data: {
        userId: input.userId,
        sourceAccountId: input.sourceAccountId,
        destinationAccountId: input.destinationAccountId,
        amountCents: input.amountCents,
      },
      select: { id: true, amountCents: true, createdAt: true },
    });
  });
}

Este exemplo melhora o read-modify-write ao fazer débito condicional. Em um sistema financeiro real, ainda seriam avaliados ledger imutável, isolamento, locks, retries, idempotência, auditoria e reconciliação.

37.2 — Regras obrigatórias

  • Valor não pode ser zero.
  • Transferência precisa ser atômica.
  • Conta de origem e destino devem ser diferentes.
  • Saldo insuficiente deve impedir toda a operação.
  • Categoria precisa pertencer ao usuário.
  • Usuário não acessa nem altera dados de outro usuário.
  • Listagens têm ordenação determinística e paginação.
  • Dinheiro não usa ponto flutuante.

37.3 — Cenários de teste

  • Criação e listagem com sucesso.
  • Duplicidade de nome.
  • Recurso inexistente.
  • Recurso de outro usuário.
  • Transferência válida.
  • Saldo insuficiente.
  • Mesmo ID de origem e destino.
  • Falha no crédito com rollback do débito.
  • Duas transferências concorrentes.
  • Paginação sem duplicar ou perder ordem dentro da estratégia escolhida.

Aprendendo Express 5 com TypeScript

Esta página é o mapa central do curso. Ela reúne os mesmos princípios que usaremos nas aulas: explicação detalhada, comparação com frontend, exemplos tipados, exercícios sem resposta direta, mini projetos, clean code, regras de negócio, edge cases e práticas usadas no mercado.

Foco em APIsExpress como backend para React, mobile e integrações.
TypeScript sempreTodos os exemplos e exercícios serão tipados.
Do zero ao realDe HTTP básico até agenda com chat em tempo real.
Não vamos aprender Express apenas como uma coleção de métodos. O objetivo é entender o que acontece em cada camada, por que determinada decisão é tomada e quais problemas aparecem em aplicações reais.
Metodologia

Como o curso vai funcionar

Cada assunto será construído em camadas. Primeiro, veremos o conceito. Depois, um exemplo pequeno. Em seguida, você aplicará a ideia em um exercício e, por fim, consolidará o conhecimento em um mini projeto.

  1. Conceito: explicação começando pelo óbvio, sem assumir conhecimento prévio de backend.
  2. Ligação com o frontend: comparação com React, services, hooks, componentes e chamadas HTTP.
  3. Exemplo em TypeScript: código curto e comentado.
  4. Documentação oficial: leitura dos pontos importantes do Express 5.
  5. Exercício guiado: problema para você resolver sem receber a resposta pronta.
  6. Dicas progressivas: perguntas e pistas para orientar o raciocínio.
  7. Revisão: análise da sua solução, clareza, tipos, responsabilidades e casos extremos.
  8. Refatoração: melhoria do código somente depois de entendermos o problema.
  9. Mini projeto: aplicação pequena usando os conceitos em conjunto.

Como serão os exercícios

Em vez de entregar algo como:

app.get("/users/:userId", (request, response) => {
  // resposta completa pronta
});

O exercício será orientado por perguntas:

Onde o identificador do usuário está chegando?

Qual deve ser o tipo desse parâmetro?

O que deve acontecer quando ele não existe?

Qual status HTTP representa esse caso?

Essa validação pertence à camada HTTP ou à regra de negócio?

Clean code não significa criar muitas pastas e classes. Primeiro criaremos código simples e correto. As camadas surgirão quando houver responsabilidades reais para separar.

Fase 1

Primeiros passos com Express 5

Nesta fase, aprenderemos o funcionamento básico da aplicação Express e o ciclo completo de uma requisição dentro de uma rota.

Conteúdos

  • O que express() retorna.
  • O que é uma aplicação Express.
  • app.listen().
  • app.get(), app.post() e outros métodos.
  • Callback de rota.
  • Request e Response.
  • res.send(), res.json(), res.status() e res.end().
  • Rota inexistente.
  • Resposta enviada mais de uma vez.
  • headersSent.

Exemplo

import express from "express";

export const app = express();

app.get("/health", (request, response) => {
  return response.status(200).json({
    status: "ok",
  });
});

O que cada parte significa

  • app.get: registra uma rota que responde ao método HTTP GET.
  • "/health": caminho da rota.
  • request: informações que chegaram do cliente.
  • response: objeto usado para construir a resposta.
  • status(200): define o status HTTP.
  • json(): serializa um objeto JavaScript como JSON e encerra a resposta.

TypeScript aplicado ao Express

O objetivo não será apenas “colocar tipos”, mas entender o que está sendo garantido, o que continua inseguro e onde a validação em runtime se torna necessária.

Conteúdos

  • Inferência e anotação.
  • type e interface.
  • Tipos do Express.
  • Generics.
  • Tipagem de params, body, query e resposta.
  • unknown versus any.
  • Narrowing e type guards.
  • Tipos opcionais.
  • Union types.
  • Tipos literais.
  • Readonly.
  • DTOs.
  • Tipo estático versus dado real recebido pela rede.

Exemplo tipando o body

import type { Request, Response } from "express";

type CreateTaskBody = {
  title: string;
  description?: string;
};

type CreateTaskResponse = {
  id: string;
  title: string;
  description: string | null;
};

app.post(
  "/tasks",
  (
    request: Request<
      Record<string, never>,
      CreateTaskResponse,
      CreateTaskBody
    >,
    response: Response<CreateTaskResponse>,
  ) => {
    const { title, description } = request.body;

    return response.status(201).json({
      id: crypto.randomUUID(),
      title,
      description: description ?? null,
    });
  },
);
O tipo CreateTaskBody não obriga o cliente a enviar esse formato. Ele apenas informa ao TypeScript como queremos trabalhar com o valor. A validação real virá com Zod.

Fase 4

Middlewares e pipeline da requisição

Middleware é uma função que participa do caminho da requisição. Ela pode ler ou alterar dados, encerrar a resposta ou passar o controle para a próxima etapa. 

requisição
  ↓
logger
  ↓
autenticação
  ↓
autorização
  ↓
validação
  ↓
controller
  ↓
resposta

Conteúdos

  • Middleware global.
  • Middleware por router.
  • Middleware por rota.
  • next().
  • Ordem de registro.
  • Logging.
  • Request ID.
  • Tempo de resposta.
  • res.locals.
  • Autenticação e autorização.

Exemplo

import type { NextFunction, Request, Response } from "express";

export function requestLogger(
  request: Request,
  response: Response,
  next: NextFunction,
): void {
  const startedAt = Date.now();

  response.on("finish", () => {
    const duration = Date.now() - startedAt;

    console.log({
      method: request.method,
      path: request.path,
      statusCode: response.statusCode,
      duration,
    });
  });

  next();
}

Erro comum

export function brokenMiddleware(
  request: Request,
  response: Response,
  next: NextFunction,
): void {
  console.log("Recebi a requisição");
  // Não respondeu e não chamou next()
}

A requisição fica pendurada porque o pipeline não foi encerrado nem continuado.

Fase 5

Validação de dados com Zod

Todo dado vindo de fora da aplicação deve ser tratado como não confiável: body, params, query, headers, arquivos e respostas de serviços externos.

Conteúdos

  • Parsing versus validação.
  • Schema.
  • Coerção.
  • Sanitização.
  • Mensagens de erro.
  • Schemas reutilizáveis.
  • Validação parcial para PATCH.
  • Refinements.
  • Inferência com z.infer.

Exemplo

import { z } from "zod";

export const createUserSchema = z.object({
  name: z.string().trim().min(2).max(100),
  email: z.string().trim().toLowerCase().email(),
  birthDate: z.coerce.date(),
});

export type CreateUserInput = z.infer<typeof createUserSchema>;

Edge cases

  • String com apenas espaços.
  • E-mail com letras maiúsculas.
  • Data inválida.
  • Número enviado como string.
  • Campo ausente.
  • null versus undefined.
  • PATCH com body vazio.
  • Campos adicionais não esperados.

Tratamento de erros no Express 5

O objetivo será criar erros previsíveis para o cliente sem esconder problemas de programação nem vazar detalhes sensíveis.

Conteúdos

  • Erro operacional versus erro de programação.
  • throw.
  • Promise rejeitada.
  • Encaminhamento automático de erros assíncronos no Express 5.
  • Middleware de erro.
  • Erros personalizados.
  • 404, validação, conflito e erro interno.
  • Stack trace por ambiente.

Middleware de erro

import type {
  ErrorRequestHandler,
} from "express";

export const errorHandler: ErrorRequestHandler = (
  error,
  request,
  response,
  next,
) => {
  console.error(error);

  return response.status(500).json({
    error: {
      code: "INTERNAL_SERVER_ERROR",
      message: "Ocorreu um erro inesperado.",
    },
  });
};

Formato de erro

{
  "error": {
    "code": "APPOINTMENT_CONFLICT",
    "message": "Este horário não está disponível.",
    "details": [],
    "requestId": "..."
  }
}

Edge cases

  • Erro depois de headers enviados.
  • Erro do banco de dados.
  • Valor lançado que não é Error.
  • Promise rejeitada.
  • Timeout de serviço externo.
